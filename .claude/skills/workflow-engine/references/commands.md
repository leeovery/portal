# Engine Commands

*Reference for **[workflow-engine](../SKILL.md)***

---

The CLI door: `node .claude/skills/workflow-engine/scripts/engine.cjs <command> [args]`. Skill prose prescribes these calls at exact points; this catalogue exists for understanding a command's full contract.

**Response contract (every mutation/transaction):** one decision-ready JSON line on stdout Рђћ `{"ok": true, Рђд}` with derived state riding so no follow-up read is needed. Failures: `{"ok": false, "error": "Рђд"}` on stderr, exit 1. Knowledge-base side effects are warn-don't-block Рђћ failures land in `warnings`, never abort. Commands that commit report `committed` (short sha, or `null` plus a note on a clean tree); commands documented as commit-free rely on the calling flow's commit cadence. Every engine-made commit also stages `.workflows/.knowledge` when it exists.

## Grammar

Every command follows six rules Рђћ new commands must too:

1. **Identity is positional, in containment order**: `{work-unit}`, then `{topic}`, then `{item}`.
2. **Required values with closed vocabularies are positional enums** (`discussion-map set Рђд {state}`, `cache stamp {wu} research-analysis|gap-analysis`, `workunit create {wu} {work-type}`).
3. **Payloads are always named flags, even when required** (`-m {message}`, `--findings-file {path}`).
4. **Optional modifiers and alternate addressing are flags** (`--skipped`, `--parent`, `--external {id}`).
5. **Mappings are `key=value` pairs** (`discovery-map sequence {wu} {topic}={order} Рђд`).
6. **A reserved word sharing a slot with user-named values must be a flag** Рђћ why `commit --inbox`/`--workflows` are flags beside positional `commit {work-unit}`.

One carve-out: the **`render` surface catalogue** passes every non-address scalar as a flag (`--gate`, `--phase`, `--m`, `--variant`, Рђд), including required closed vocabularies Рђћ the address is the only positional. Uniformity across the thirteen surfaces beats rule 2 there.

## Commands

**`boot`** Рђћ the entry pipeline: runs `workflow-migrate/scripts/migrate.cjs` (resolved relative to the engine, cwd = project root), then `knowledge check`, then `knowledge compact` when ready Рђћ one call in place of the sequential Step 0 commands. Response: `{"ok": true, "migrations": {"changed": false, "output": "<trimmed report>"}, "knowledge": "ready"|"not-ready", "compacted": false, "kb_committed": null, "warnings": []}`. `migrations.changed` mirrors the orchestrator's own files-updated signal; `output` is the report with the prose stop-gate lines stripped (update counts kept). A failing migrate.cjs is a hard error (`{ok:false}` on stderr, exit 1) Рђћ migrations must never half-run silently. A failing `check` reports `not-ready`, and boot never initialises anything itself: a not-ready response additionally carries `"system_config": {"status": "valid"|"absent"|"invalid", "provider": <name|null>, "model": <name|null>}` Рђћ read from `~/.config/workflows/config.json` only, never the credentials file, so no secret can enter the response Рђћ which the calling skill's knowledge gate branches on to drive `knowledge setup` conversationally. A failing `compact` lands in `warnings`, never blocks. When the store is ready, knowledge-store dirt boot finds is committed scoped to `.workflows/.knowledge` (`chore(knowledge): initialise store` for untracked store files Рђћ the first boot after a `knowledge setup` run; `chore(knowledge): compact store` otherwise; sha in `kb_committed`, commit failure a warning). The conversational pieces (migration summary, review gate, the knowledge gate) stay in the calling skill.

```bash
engine boot
```

**`manifest`** Рђћ the field surface: read and write work-unit manifest fields by dot-path. Thin dispatch in `engine.cjs`, logic in `domain/fields.cjs`, schema validation and locking from the kernel modules.

**Dot-path addressing.** Every command takes `command <dotpath> [field] [value]`. The path joins work unit, phase, and topic with dots; segment count determines the access level:

| Segments | Level | Path | Field | Resolves to |
|----------|-------|------|-------|-------------|
| 1 | Work unit | `my-epic` | `work_type` | `work_type` |
| 2 | Phase | `my-epic.planning` | `format` | `phases.planning.format` |
| 3 | Topic | `my-epic.discussion.auth-flow` | `status` | `phases.discussion.items.auth-flow.status` |

Topic-level access routes through items (`phases.{phase}.items.{topic}.{field}`); the field argument may itself be a dot-path (`sources.auth-api.status`). The wildcard topic `*` collects from every item in a phase (`get`/`exists` only; `get` returns a JSON array of `{topic, value}` objects). The reserved prefix `project` routes to the project manifest (`.workflows/manifest.json`) with the field embedded in the dot-path Рђћ `get project.defaults.plan_format`, `set project.defaults.plan_format {format}` Рђћ which is why `project` is a reserved work-unit name (dots and phase names are also barred, enforced at `workunit create`).

**Output contract, deliberately split.** Reads print bare stdout Рђћ they are prose substitution surfaces: scalars raw (no quotes), subtrees as two-space-indented JSON, `exists` printing `true`/`false` (always exit 0), `key-of` the bare key, `list` a JSON array of manifests, missing `get` paths empty stdout exit 0. Read errors keep `Error: Рђд` on stderr with exit 1 (real error) or exit 2 (expected miss Рђћ `key-of`/`resolve` on a missing work unit, or a `key-of` value not found; `get` misses are the empty-stdout exit 0 above, never exit 2). Mutations answer with the engine's one-line JSON response and fail as `{"ok": false, "error": Рђд}` stderr exit 1 like every other verb. `apply <work-unit> --file <ops.json>` is the batch form of set/delete: ops `{"op": "set", "path": <dotpath>, "fields": {Рђд}}` / `{"op": "delete", "path": <dotpath>, "field": Рђд}`, values native JSON (`null` is `null`, never `'~'`), every op validated with the entry named before anything is written, all paths inside the one work unit (one lock, one atomic write Рђћ the project manifest is out of scope), a missing delete target failing the whole batch. No git commit Рђћ the calling flow's commit covers it.

```bash
engine manifest get    <dotpath> [<field.path>]          # scalar raw / subtree JSON; missing Рєњ empty, exit 0
engine manifest set    <dotpath> <field> <value>                 # single field (positional shorthand)
engine manifest set    <dotpath> <field>=<value> [<field>=<value> Рђд]  # uniform batch Рђћ never mixed with positional
engine manifest push   <dotpath> <field> <value>         # append, creating the array
engine manifest pull   <dotpath> <field> <value>         # remove first deep-equal match; no-op safe
engine manifest delete <dotpath> <field.path>            # errors when the path is absent
engine manifest apply  <work-unit> --file <ops.json>     # batch set/delete across one work unit Рђћ atomic
engine manifest exists <dotpath> [<field.path>]          # true/false, always exit 0
engine manifest list   [--status <s>] [--work-type <t>]  # JSON array of manifests
engine manifest key-of <dotpath> <field.path> <value>    # reverse lookup (e.g. task_map external Рєњ internal)
engine manifest resolve <work-unit>.<phase>[.<topic>]    # artifact paths for KB-indexed phases
```

`set` batches: additional `<field>=<value>` pairs after the positional pair land under one lock in one write, answered by one response listing every field written (`{"ok": true, "path": Рђд, "set": {Рђд}}`); pairs split on the **first** `=` only, so values may contain `=`. Values parse as JSON first (arrays, objects, numbers, booleans), falling back to string; a bare `~` is null, the same sentinel `task complete --next-task` takes Рђћ shell-quote anything zsh would eat (`'[]'`, `'{}'`, `'~'`). Validation is schema-driven (`kernel/manifest-schema.cjs`): `work_type`, work-unit `status`, per-phase item statuses (discovery's empty vocabulary refuses every status write Рђћ map items carry no status), and `*_gate_mode` fields; a refused value fails a whole batch with nothing written. Dot-paths with empty segments are refused outright (an unset shell variable would alias the project manifest), and work-unit-level mutations (`set`, `push`, `pull`, `apply` set-ops) refuse a field whose first segment names a phase Рђћ a typo'd dot-path would build a shadow tree beside `phases.*` that no read joins; reads and `delete` stay free so a stray tree can be inspected and repaired. Mutation responses: `set` echoes the fields written; `push` carries `pushed` and the new `length`; `pull` carries `removed` (`false` on any no-op) and `length`; `delete` carries `deleted: true`.


**`workunit`** Рђћ work-unit lifecycle. `create` is the work-type commit Рђћ discovery's durability boundary as one transaction. It creates the manifest (identity fields, `status: in-progress`, empty phases, project-manifest registration; an existing manifest is reused as-is, `created: false`), copies each `--import` into `imports/`, moves each `--seed` Рђћ a live inbox path, its folder deriving the `inbox:{idea|bug|quickfix}` source tag Рђћ into `seeds/`, installs the `--session-log-file` content verbatim as `discovery/sessions/session-001.md` (the content is model-authored Рђћ the engine never writes prose; a creation outside discovery passes `--no-session-log` instead Рђћ log-less must be explicit Рђћ landing no log and, for an epic, no `active_session` marker), sets `phases.discovery.active_session = "001"` for an epic, and commits scoped to the work unit plus `.workflows/.inbox` when seeds moved out of it (`discovery({wu}): create work unit ({work_type})` Рђћ engine-owned message). Landing filenames are normalised (lowercase; whitespace/punctuation runs Рєњ `-`; repeats collapsed; `.md` forced; imports normalising to dotfiles are skipped and reported in `skipped_imports`, seeds fall back to `seed.md`) and collisions are suffixed `-2`, `-3` against both the destination directory and the batch; every landing gets an `imports[]`/`seeds[]` manifest entry with a fresh ISO-8601 UTC timestamp and a knowledge-base index (warn-don't-block). Validation completes before any mutation Рђћ a missing import fails the whole call with `{"ok": false, "error": Рђд, "missing_imports": [Рђд]}` so the calling flow can re-prompt and re-run against untouched state. Response: `{"ok": true, "work_unit": Рђд, "work_type": Рђд, "created": bool, "imports": [{"path": Рђд}Рђд], "seeds": [{"path": Рђд, "source": Рђд}Рђд], "skipped_imports": [Рђд], "session_log": Рђд, "committed": "<short-sha>", "warnings": []}`.

`complete`, `cancel`, and `reactivate` are the lifecycle transactions, each one transaction: manifest write, knowledge-base sync (warn-don't-block), scoped commit (`.workflows/{wu}`). `complete` and `cancel` also purge the unit's scratch cache (`.workflows/.cache/{wu}/` Рђћ rebuildable per-phase scratch; reactivation regenerates on demand): disk-only, except pre-gitignore tracked residue whose deletions ride the transaction commit. `complete` sets `status: completed` and stamps `completed_at` (today, UTC Рђћ same form as `created`); its commit message rides on `-m` because it varies by caller (manual completion vs pipeline-terminal vs review-skipped Рђћ the engine can't derive how completion happened). No knowledge-base action Рђћ completed units retain their chunks Рђћ except when completing a cancelled unit with a finished pipeline directly (the one cancelled state complete accepts): cancellation removed the unit's chunks, so that path re-indexes them (warn-don't-block). `cancel` sets `status: cancelled` and removes the unit's chunks (`knowledge remove --work-unit`); commit message engine-owned: `workflow({wu}): mark as cancelled`. `reactivate` restores `status: in-progress` from completed **or** cancelled Рђћ a finished pipeline (derived next phase `done`) is no bar; the reactivated unit surfaces as finalising until a phase is reopened Рђћ clears a stale `completed_at`, and Рђћ only from cancelled, whose removal emptied the index Рђћ re-indexes what cancellation removed (every completed artifact in an indexed phase, shape-valid `imports/` and `seeds/` entries, on-disk analysis caches, and Рђћ epics Рђћ the on-disk `discovery/sessions/session-*.md` logs); commit message engine-owned: `workflow({wu}): reactivate work unit`. Illegal transitions refuse loudly: complete/cancel refuse an already-completed/cancelled unit respectively and route the opposite closed state through reactivate (complete alone also accepts the cancelled finished-pipeline state); reactivate refuses an in-progress unit. Statuses are validated against the shared schema's work-unit vocabulary Рђћ the engine is never the permissive path. Responses: `{"ok": true, "work_unit": Рђд, "status": Рђд, "committed": "<short-sha>", "warnings": []}` plus `completed_at` (complete) or `previous_status` (reactivate); `committed: null` plus a note when nothing was staged.

`pivot` converts an in-progress feature to an epic as one transaction: flips `work_type: epic` in the work-unit manifest **and** the project manifest's registration (upserted when a legacy unit predates registration), registers the feature's single topic (topic name = work unit name) on the discovery map with backfill semantics (`routing` = `research` when the research phase exists, else `discussion`; summary/description left for the next epic entry's summary-backfill), re-indexes the whole unit via the same walk reactivate uses Рђћ chunk metadata carries `work_type`, which is why pivot re-indexes Рђћ and commits both manifests (`workflow({wu}): pivot to epic`, engine-owned). Refuses a non-feature, a unit that isn't in-progress, and a topic already on (or dismissed from) the map Рђћ all before anything is touched. Response: `{"ok": true, "work_unit": Рђд, "work_type": "epic", "routing": Рђд, "committed": "<short-sha>", "warnings": []}`.

`absorb` merges a feature into an in-progress epic as a new topic, then deletes the feature Рђћ one transaction owning the mechanical tail; the judgment (choosing the feature, the epic, and the topic name with the user) stays in the calling prose, and the verb takes the decided inputs. Validation completes before any mutation Рђћ feature exists and is a feature with a discussion (item **and** file) and no specification-or-beyond phases; epic exists, is an epic, is in-progress; the topic name is dot/slash-free and free in the epic (map item, dismissed list, discussion item, discussion file) Рђћ so a refusal leaves both work units byte-identical: no crash window between the feature's deletion and the commit. It moves the discussion to `discussion/{topic}.md`, research files (topic collisions suffix `-{feature}`, then numbered), and imports/seeds (filename collisions suffix `-2`, `-3` Рђћ create's dedupe Рђћ with manifest entries carrying their **original** timestamps and seed provenance); mirrors each phase item's status onto the epic; registers the map item with backfill semantics (`routing` = `research` when the feature did research); removes the feature's knowledge-base chunks and indexes the moved artifacts at their epic identities (completed phase artifacts; imports and seeds always Рђћ warn-don't-block); deletes the feature directory, its project-manifest registration, and its scratch cache (`.workflows/.cache/{feature}/`); and lands ONE commit staging `.workflows/{feature}`, `.workflows/{epic}`, and `.workflows/manifest.json` (`workflow({feature}): absorb into {epic}`, engine-owned). The feature's `discovery/sessions/` logs are not moved Рђћ they are deleted with the feature directory; git history is provenance. Response: `{"ok": true, "feature": Рђд, "epic": Рђд, "topic": Рђд, "discussion": {"path": Рђд, "status": Рђд}, "research": [Рђд], "imports": [Рђд], "seeds": [Рђд], "routing": Рђд, "committed": "<short-sha>", "warnings": []}`.

`promote` moves a completed epic specification assessed as cross-cutting to its own cross-cutting work unit Рђћ one transaction owning the mechanical tail; the judgment (the assessment, the cc unit's name, the one-line description, user confirmation) stays in the calling prose, and the verb takes the decided inputs. Validation completes before any mutation Рђћ the source is an in-progress epic whose topic carries a `completed` specification item with `specification.md` on disk; the `--to` name passes the work-unit name rules (create's own) and is free (no `.workflows/{to}` directory, no project-manifest registration); spec source names that would break path addressing refuse as tampered Рђћ so a refusal leaves the epic byte-pristine and no cc unit exists. It moves the spec directory to `specification/{to}/` (topic = work unit name for a cross-cutting unit), moves each spec source whose discussion file exists on disk into the cc unit's `discussion/` (registered `completed` Рђћ sources of a completed spec were incorporated; the epic's own discussion items stay, git history is provenance), creates the cc manifest log-less and already completed (`status`/`completed_at` Рђћ the cc pipeline is terminal after spec Рђћ plus `source_work_unit`/`source_topic` origin provenance and the spec item's `date`, all stamped today UTC), registers it in the project manifest, sets the epic's spec item to `status: promoted` + `promoted_to: {to}`, syncs the knowledge base (moved artifacts indexed at their cc identities, the epic's old chunks removed Рђћ warn-don't-block), and lands ONE commit staging `.workflows/{wu}`, `.workflows/{to}`, and `.workflows/manifest.json` (`spec({wu}): promote {topic} to cross-cutting work unit`, engine-owned). Response: `{"ok": true, "work_unit": Рђд, "topic": Рђд, "cc_work_unit": Рђд, "cc_status": "completed", "discussions": [{"name": Рђд, "path": Рђд}Рђд], "specification": {"path": Рђд}, "status": "promoted", "promoted_to": Рђд, "committed": "<short-sha>", "warnings": []}`.

```bash
engine workunit create <work-unit> <work-type> --description <text> (--session-log-file <path> | --no-session-log) [--import <path> Рђд] [--seed <path> Рђд]
engine workunit complete <work-unit> -m "<message>" [--pipeline [--skipped-review]]   # --pipeline: the bridge context Рђћ the confirmation section is the "{Type} Completed" banner
engine workunit cancel <work-unit>
engine workunit reactivate <work-unit>
engine workunit pivot <work-unit>
engine workunit absorb <feature> --into <epic> --topic <name>
engine workunit promote <work-unit> <topic> --to <cc-work-unit> --description <text>
```

**`discussion-map`** Рђћ Discussion Map subtopic writes. `add` and `set` each load the work unit's manifest, apply the transition, save atomically, and print one decision-ready JSON line: `{"ok": true, "subtopic": "Рђд", "status": "Рђд", "all_decided": false, "unresolved_count": N}` Рђћ no follow-up read needed, and no git commit (the calling session's commit cadence picks the manifest change up).

```bash
engine discussion-map add <work-unit> <topic> <subtopic> [--parent <subtopic>]   # new subtopic, starts pending
engine discussion-map set <work-unit> <topic> <subtopic> <state>                 # pending|exploring|converging|decided|deferred
```

Subtopic names are kebab-case slugs; `--parent` nests under an existing top-level subtopic (two levels max).

**`discovery-map`** Рђћ the Discovery Map's writes. `sequence` records the suggested execution order as one transaction: validates every topic exists under `phases.discovery.items` and every order is a positive integer, sets each topic's `order`, and commits scoped to the work unit (`discovery({wu}): sequence topic map`); response `{"ok": true, "ordered": {"{topic}": N, Рђд}, "committed": "<short-sha>"}` (`committed: null` plus a note when nothing was staged). Choosing the order is the caller's judgment Рђћ the command only records it.

The per-item map operations write the manifest with no git commit (the calling session's commit cadence picks the change up) and enforce the map's lifecycle gates in the engine, deriving each item's lifecycle with the same join the epic detail builder uses. `add` creates a map item as `{routing, source, summary[, description]}` Рђћ never a `status` field (map-item lifecycle is computed at render time, not stored); the positional routing is validated against the same enum `reroute` takes, `--source` defaults to `discovery`, the name follows rename's dot/slash-free rule, an active duplicate is refused, and a dismissed name is refused unless `--force-dismissed` carries the user's confirmed re-add decision (the entry is then pulled off the dismissed list). `--backfill` (mutually exclusive with `--summary`/`--description`) lands the item without either field Рђћ keys absent, not `""` Рђћ so the next epic entry's summary-backfill drafts them from the topic's artifacts; for topics arriving with work already done (absorb, pivot). `edit` sets `summary` and/or `description` Рђћ at least one flag, any lifecycle. `remove` hard-deletes a **fresh** item and pushes its name onto `phases.discovery.dismissed`. `rename` moves a **fresh** item to a new key preserving every field and its map position, moving `discovery/briefs/{old}.md` to `{new}.md` when it exists and rewriting the item's `brief_path` when it points at the canonical `discovery/briefs/{old}.md` (a pointer aimed anywhere else is left alone); the new name must not collide with an active item or an existing brief file Рђћ checked whether or not the old brief exists Рђћ (a dismissed-list match is allowed Рђћ the entry is left alone) and must be dot/slash-free. `reroute` sets a **fresh** item's `routing`. The fresh gate on remove/rename/reroute also refuses whenever ANY research or discussion item exists for the topic, whatever lifecycle the join derives Рђћ the map item is those items' historical anchor. `handle` sets `handled: true` (any lifecycle except handled/cancelled); `unhandle` clears the marker (handled only). Gate refusals name the blocking lifecycle and the recovery path (e.g. `"X" can't be removed Рђћ research is in flight on it; cancel from the epic menu instead`). Responses: `{"ok": true, "work_unit": Рђд, "name": Рђд, "op": Рђд, "lifecycle": Рђд, "map_total": N}` Рђћ every op carries `map_total` (items on the map after the op) so the calling flow needs no follow-up read Рђћ plus op-specific fields: add and edit echo the values written (add also `undismissed` when forced past a dismissed match), remove carries `dismissed: true`, rename carries `renamed_from`/`preserved_fields`/`matches_dismissed` (plus `brief_moved: true` when the brief file moved), reroute `routing`, handle/unhandle `handled`.

```bash
engine discovery-map sequence <work-unit> <topic>=<order> [<topic>=<order> Рђд]   # suggested execution order, scoped commit
engine discovery-map add <work-unit> <name> <research|discussion> (--summary <text> [--description <text>] | --backfill) [--source <tag>] [--force-dismissed]
engine discovery-map add-batch <work-unit> --file <topics.json>   # whole topic set, one lock/save Рђћ atomic; entries {name, routing, summary, description?, brief_path?, force_dismissed?}; source always `discovery`
engine discovery-map edit <work-unit> <name> [--summary <text>] [--description <text>]
engine discovery-map remove <work-unit> <name>       # fresh only; name lands on the dismissed list
engine discovery-map rename <work-unit> <old> <new>  # fresh only; every field preserved, brief file + brief_path follow
engine discovery-map reroute <work-unit> <name> <research|discussion>   # fresh only
engine discovery-map handle <work-unit> <name>       # any lifecycle except handled/cancelled
engine discovery-map unhandle <work-unit> <name>     # handled only
```

**`discovery-session`** Рђћ the epic discovery-session lifecycle. `open` starts a session at the log's first write (lazy creation): it allocates the next session number from the on-disk logs (highest `session-NNN.md` plus one, first = `001` Рђћ disk is the source, no manifest field stores the counter), moves the drafted log into place as `discovery/sessions/session-NNN.md` (moved, not copied Рђћ the draft is consumed), and sets `phases.discovery.active_session` Рђћ one manifest write under the work-unit lock. **No commit** and no knowledge-base index: the session is live, and the calling flow's commit cadence picks up the log and marker at its next natural break. The draft's content is model-authored and installed verbatim Рђћ the engine never writes prose. Validation completes before any mutation Рђћ the work unit must exist and be an epic (the sole work type with a resumable discovery session loop; single-phase work gets its log at `workunit create`), no session may already be open, and the draft must exist with content Рђћ so a refusal leaves everything pristine, the draft included. Response: `{"ok": true, "work_unit": Рђд, "session": "NNN", "path": Рђд}`.

`close` ends the session the `phases.discovery.active_session` marker names, as one transaction: delete the marker (resume detection on the next entry sees a closed session), index the marker's session log into the knowledge base (warn-don't-block), and commit scoped to the work unit with the caller's message Рђћ one call covers whatever the session left dirty (map writes, the log's Topics Identified and Conclusion, harvest briefs). The message rides on `-m` because it varies by caller (topics synthesised vs edits-only finalisation). The log's content is model-authored **before** the call Рђћ the engine never writes prose. Refuses loudly, manifest untouched, when no marker is set (the marker is set at open and always pairs with an existing log, so a browse-only session has nothing to close) or when the marker names a session with no log on disk. Response: `{"ok": true, "work_unit": Рђд, "session": "NNN", "session_log": Рђд, "committed": "<short-sha>", "warnings": []}` (`committed: null` plus a note when nothing was staged).

```bash
engine discovery-session open <work-unit> --session-log-file <path>
engine discovery-session close <work-unit> -m "<message>"
```

**`topic`** Рђћ phase-item transitions. `start`, `complete`, `reopen`, and `supersede` are lifecycle bookkeeping with no git commit Рђћ the calling flow's commit cadence picks the manifest change up. `start` creates the phase item with `status: in-progress` when absent (init-phase semantics: `phases.{phase}.items.{topic}`, status only) or sets an existing item back to `in-progress`; a completed item must go through `reopen` Рђћ resuming is not starting Рђћ and a cancelled item through `reactivate`. Response: `{"ok": true, "phase": "Рђд", "topic": "Рђд", "status": "in-progress", "created": bool}`. `complete` sets `status: completed` on an existing item and, when the phase is research/discussion/investigation/specification, indexes the phase artifact into the knowledge base. Response: `{"ok": true, "phase": "Рђд", "topic": "Рђд", "status": "completed", "warnings": []}`. `reopen` sets a `completed` item back to `status: in-progress` Рђћ the resume path `start` refuses. Anything not completed is refused (a cancelled item must go through `reactivate`), and there is no knowledge-base sync Рђћ the item's chunks stay live until re-completion re-indexes over the same identity. Response: `{"ok": true, "phase": "Рђд", "topic": "Рђд", "status": "in-progress"}`. `supersede` sets `status: superseded` and `superseded_by: {topic}` (always the absorbing **topic** name) and removes the item's chunks from the knowledge base Рђћ legal only in phases whose shared-schema status vocabulary contains `superseded` (schema-driven, no hardcoded list). The `--by` topic must already exist in the same phase; a proposed item is refused (no artifact Рђћ reconcile deletes it instead), an already-superseded item is refused, and a cancelled item must go through `reactivate`. No commit Рђћ supersession is batch-oriented; the calling flow supersedes the set, then commits once. Response: `{"ok": true, "phase": "Рђд", "topic": "Рђд", "status": "superseded", "superseded_by": "Рђд", "warnings": []}`.

`cancel` and `reactivate` are the epic transactions, each one transaction: manifest write (stash/restore `previous_status`, cancel also drops the topic's discovery-map `order`), knowledge-base sync (remove on cancel; re-index on reactivate when the restored status is `completed` in an indexed phase), and a commit scoped to `.workflows/{wu}` (`workflow({wu}): cancel {topic} ({phase})` / `Рђд reactivate Рђд`). Response: `{"ok": true, "topic": "Рђд", "phase": "Рђд", "status": "Рђд", "committed": "<short-sha>", "warnings": []}` (`committed: null` plus a note when nothing was staged).

Across the family, the knowledge base is a derived index Рђћ its failures land in `warnings`, never block.

```bash

engine topic complete <work-unit> <phase> <topic>
engine topic reopen <work-unit> <phase> <topic>
engine topic supersede <work-unit> <phase> <topic> --by <topic>
engine topic cancel <work-unit> <phase> <topic>
engine topic reactivate <work-unit> <phase> <topic>
```

**`task`** Рђћ implementation-task bookkeeping: format-blind, manifest-side only. The engine never reads or writes a task backend and knows no plan-format names Рђћ the session does the plan surgery, these commands record it against `phases.implementation.items.{topic}`. Each command is load Рєњ apply Рєњ save plus one decision-ready JSON line followed by rendered gate sections; no git commit (the session's per-task commit cadence picks the manifest change up).

```bash
engine task init <work-unit> <topic>                       # create-or-resume the implementation item
engine task start <work-unit> <topic> <internal-id>        # record current_task; fresh task: reset fix_attempts, drop the fix-tracking cache file; restarting the in-flight task preserves both
engine task fix-attempt <work-unit> <topic> <internal-id> --findings-file <path>
engine task complete <work-unit> <topic> (<internal-id> | --external <id>) [--skipped] [--next-task <id|~>] [--phase <N>] [--phase-complete]
engine task analysis-cycle <work-unit> <topic>             # increment both analysis counters
```

`init` creates the item with `status: in-progress`, gated gate modes, zeroed counters, empty `linters`/`project_skills`, `current_phase: 1`, `current_task: null` Рђћ or, when it already exists, performs a session-only reset (three gate modes Рєњ `gated`, `analysis_cycle_session` Рєњ 0; `analysis_cycle_total`, arrays, and progress fields untouched; `fix_attempts` Рєњ 0 unless `current_task` has a live fix-tracking file Рђћ a crash-resume mid-task keeps the in-flight pair). Response: `{"ok": true, "mode": "created"|"resumed", "gates": {Рђд}, "counters": {Рђд}}`. `start` records the internal id as `current_task`, resets `fix_attempts`, and drops the task's fix-tracking cache file (clean slate per task) Рђћ unless the id is already `current_task` AND its tracking file exists (a true resume, or a post-compaction re-run to re-fetch the gate sections), where both survive as that task's convergence history; re-running `start` for the in-flight task is idempotent. `fix-attempt` increments `fix_attempts` and appends the findings file's content verbatim under a `## Attempt {N}` section in `.workflows/.cache/{wu}/implementation/{topic}/fix-tracking-{internal-id}.md`; response carries `attempts`, `threshold_reached` (РЅЦ 3), and `fix_gate_mode`. `complete` pushes the internal id to `completed_tasks` (skips included Рђћ the plan carries the skip distinction; an id already recorded is a no-op push, same response), zeroes `fix_attempts` (the finished task's counter never leaks into the next; its tracking file stays as history), resolves `--external` by mirroring manifest `key-of` over `{wu}.planning.{topic}.task_map`, sets `current_phase`/`current_task` (`~` = null) when flagged, and pushes to `completed_phases` on `--phase-complete` (`--phase`, or the phase embedded in the internal id). `analysis-cycle` responds with `cycle_total`, `cycle_session`, `over_session_limit` (> 3), and `analysis_gate_mode`.

**Gate sections.** After the JSON line, each verb appends the implementation loop's state-derived gate sections Рђћ rendered `=== DISPLAY: Рђд ===` / `=== MENU: Рђд ===` blocks (`domain/projections/tasks.cjs`), always printed, no flag; the JSON line stays the machine-readable contract. Each marker names its gate and carries its own emission instruction: the calling flow emits the content verbatim at the gate the marker names (DISPLAY as a code block, MENU as markdown), never at the call itself, and never parses it for decisions. Per verb: `init` and `complete` always carry `MENU: blocked tasks` Рђћ either can be the session's latest response when the loop stops on blocked tasks (the blocked-task list itself is plan-format state the session renders). `start` carries `MENU: task gate` when `task_gate_mode` is `gated` Рђћ the approval menu named for the internal id Рђћ and nothing in auto mode. `fix-attempt` carries `DISPLAY: fix threshold` (the attempt-count callout) when the threshold is reached, plus `MENU: fix gate` when the gate is `gated` or the threshold forces it; the `a`/`auto` option renders only while the gate is `gated`. `analysis-cycle` carries `DISPLAY: cycle limit` and `MENU: cycle gate` when `cycle_session` is over the session limit. Conversational content (reviewer findings, executor summaries, blocked-task lists) never renders here Рђћ it stays with the session.

**`inbox`** Рђћ archive / restore / delete one or more inbox items as a single transaction. Paths are validated strictly against the inbox layout (`.workflows/.inbox/{ideas|bugs|quickfixes}/Рђд`, archived items under `.inbox/.archived/Рђд`) before anything moves; one commit covers the whole set (`workflow(inbox): archive {slug}` for one item, `Рђд archive {N} items` for several Рђћ same forms for restore/delete). Response: `{"ok": true, "archived": [pathsРђд], "committed": "<short-sha>"}` (key matches the verb: `archived` / `restored` / `deleted`).

```bash

engine inbox restore <path> [<path> Рђд]   # .archived/{folder}/ Рєњ live
```

**`cache`** Рђћ analysis-cache stamping: record that an analysis ran over the current completed inputs. Collects and checksums the input files exactly as the read side (`computeAnalysisCacheStatus` in domain/derivations) does Рђћ a fresh stamp is `valid` by construction. Writes `checksum`, `generated` (current ISO timestamp), and the input file names to the kind's manifest home: `phases.research.analysis_cache` (`files`) for `research-analysis`, `phases.discovery.gap_analysis_cache` (`input_files`) for `gap-analysis` Рђћ then indexes the kind's `.state/` cache file (`research-analysis.md` / `discovery-gap-analysis.md`) into the knowledge base (warn-don't-block). Errors when no qualifying inputs exist Рђћ the analyses' preconditions skip the stamp in that case. Response: `{"ok": true, "kind": "Рђд", "checksum": "Рђд", "files": N, "warnings": []}`. No git commit Рђћ the calling flow's commit cadence picks the manifest change up.

**`commit`** Рђћ the scoped commit helper: `git add -- .workflows/{wu}` (`.workflows/.inbox` with `--inbox`, or the whole `.workflows` tree with `--workflows` Рђћ migrations touch many work units plus `.workflows/.state`) plus commit. Every engine-made commit Рђћ this helper and every transaction Рђћ also stages `.workflows/.knowledge` when it exists (exists-guarded, `domain/commit.cjs`): transactions dirty the store as a side effect of their knowledge sync, and that dirt belongs with the write that produced it. A clean tree is fine: `{"ok": true, "committed": null, "note": "nothing to commit"}`, exit 0. `--plan <topic>` widens the scope with the plan's declared storage: it stages `.workflows/manifest.json` and every `storage_paths` entry recorded on the planning item at plan init (the format's authoring doc declares them; entries are validated relative-only and skipped when neither on disk nor tracked, while deleted-but-tracked paths still stage their deletions). A planning item without `storage_paths` (pre-upgrade plan) fails loudly with the one-line repair. Code never rides a `--plan` commit Рђћ implementation's code commits stay with the session and raw git.

```bash

engine commit --inbox -m "<message>"
engine commit --workflows -m "<message>"
```

**Transaction confirmation sections.** The lifecycle verbs append labelled sections after their JSON line, the same pattern as the `task` gate sections: `workunit complete`/`cancel`/`reactivate` carry `DISPLAY: confirmation` (cancel/reactivate preceded by `DISPLAY: kb warning` when `warnings` is non-empty); `topic cancel`/`reactivate` likewise; `topic complete` and `discovery-session close` carry `DISPLAY: kb warning` alone when `warnings` is non-empty Рђћ no confirmation line, the calling flow owns its own conclusion display; `workunit absorb` carries the post-absorption summary, `promote` the promotion summary, and `pivot` a `DISPLAY: kb warning` (when warranted) plus `MENU: pivot continuation`. The JSON line stays the machine contract; the calling flow emits each section verbatim at its marker's named moment and never parses it for decisions.

**`render`** Рђћ the surface catalogue (`domain/render.cjs`): named runtime surfaces returning demarcated `=== DISPLAY: Рђд ===` / `=== MENU: Рђд ===` sections the calling flow emits verbatim at each marker's named moment. Address-backed values come from the manifest (JSON state only Рђћ the engine never parses markdown artifacts); judgment content arrives as a JSON payload file written to the phase cache with the Write tool and validated loudly per-field. Gate-mode branching happens inside the surface: the response carries the menu when the mode is `gated` and the auto-proceed line when it is `auto` Рђћ the calling flow branches on which section arrived, never on the mode itself. No git commit; nothing is written.


The bridge continuation surfaces take a bare `<work_unit>` address (work-unit-level, type read from the manifest). The continue-* selection step is not a `render` surface: each navigation gateway's index dump appends `DISPLAY: selection` / `MENU: selection` sections from the shared selection projection Рђћ emitted only at the select step, per their markers.

The `finding` surface serves both review-findings loops (planning and specification) with per-consumer labels in the payload; `proposed-task` serves both synthesis loops (implementation analysis, review actions) the same way Рђћ its gate mode rides as a flag because one consumer carries the mode in the cycle response and the other in staging-file frontmatter. `task-list` takes `--variant existing` for plan-construction's review-of-existing-tables gate (no auto-opt-in option; "confirmed" phrasing). The `finding` surface renders a diff as a single ` ```diff `-fenced section Рђћ space-prefixed context lines placing the change, `-`/`+` change lines Рђћ so the host's diff colouring applies; prose artefact content (`content`) renders as a code block. The `render` group also keeps its dev/debug primitives (`signpost`, `box`, `wrap`, `tree`) Рђћ authoring aids only, never called from skill flows.
