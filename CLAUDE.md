# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Test

```bash
go build -o portal .                        # build binary
go test ./...                               # unit lane: fast, hermetic — no real tmux servers*, no daemons, no built binaries
go test -tags integration -p 1 ./...        # integration lane: real tmux + real daemons + built binaries. -p 1 is REQUIRED
go test ./cmd -run TestOpenCommand_PathArgument_SkipsTUI   # run single unit test
go test -tags integration ./cmd -run TestCommitNowSymptom   # run single integration test
```

*The unit lane still includes a handful of fast real-tmux **client** tests (`internal/tmux/*_realtmux_test.go` — per-test `-L` sockets, no daemons, no subprocess binaries). **Every test that spawns a `portal state daemon` or execs a built `portal` binary lives behind `-tags integration`**, where the daemon-pgrep sandbox is compiled in — do not add such a test to the unit lane.

`-p 1` on the integration lane is load-bearing, not a preference: with no CI, all packages otherwise run in parallel on one machine and the daemon-timing tests (composite e2e suites, hysteresis measurement) flake under mutual contention.

Lint via `golangci-lint run` (config in `.golangci.yml`: the standard set plus the `modernize` linter). Not wired into CI — it's run locally / from the Mint release step. No code generation.

Tests **must not** use `t.Parallel()` — the cmd package injects mocks via package-level mutable state (`bootstrapDeps`, `openDeps`, `doctorDeps`, etc.) and cleans up with `t.Cleanup()`.

### Visual capture harness

`cmd/capturetool` is a **separate offline program** (NOT a `portal` subcommand) that renders a named deterministic fixture of the real TUI through the shared `tui.Build` constructor with every tmux seam faked (`internal/capture`) — it opens no tmux server and reads no real config (an import guard keeps `internal/capture` out of the production binary). `testdata/vhs/` holds one **`vhs`** `.tape` per canonical screen plus committed reference PNGs; `vhs testdata/vhs/<tape>` re-captures a screen. To view a screen live: `go run ./cmd/capturetool --fixture <name>` (`--appearance dark|light`; `capture.FixtureNames()` lists fixtures). This is the visual-verification harness for the Modern Vivid reskin.

## Architecture

Portal is a CLI for interactive tmux session management, built with **Cobra** (CLI) + **Bubble Tea v2** (TUI) + **Lipgloss v2** (styling). The TUI is reskinned to the **Modern Vivid** design — an owned mode-matched canvas, a closed colour-token layer, restyled chrome/modals/loading, and a concurrent cold-boot flip (see "Modern Vivid TUI" and "Cold-path startup flip" below).

### Command flow

`main.go` → `cmd/root.go` → subcommands. Root's `PersistentPreRunE` handles tmux server bootstrap (auto-starts server if not running) and injects a shared `tmux.Client` + `serverStarted` flag into `cmd.Context()`.

**Key command:** `cmd/open.go` — the main entry point (`portal open` / the `x` shell function from `portal init`). A bare positional runs the resolution grammar (exact session → glob → path → alias → zoxide) and creates/attaches directly; domain-pinning flags force a single domain (`-s/--session` attaches an existing session/glob and never mints, `-p/--path` mints at a literal dir, `-a/--alias`, `-z/--zoxide` — a pinned miss hard-fails rather than falling through to the picker); `-f/--filter <text>` skips resolution and opens the picker pre-filtered; with no args it launches the TUI picker. Two-or-more targets (or a single glob that may expand to K≥2) route through the multi-target burst below. The former `portal attach` command is retired — a spawned burst window now runs `portal open --session <name>`.

**Multi-target burst:** `cmd/open_burst.go` / `cmd/open_burst_run.go` — when `portal open` receives 2+ targets (or one glob that may expand to K≥2), `orderedOpenTargets` recovers the true left-to-right target order from the raw argv, `dispatchOpenBurst` runs an atomic read-only pre-flight (any unresolvable target aborts the whole set before anything opens), and `runOpenBurst` opens the surfaces via `spawn.SplitTriggerFirst`: the FIRST surface (the trigger) absorbs the invoking terminal and self-connects LAST, while the N−1 non-trigger surfaces are spawned FIRST into host-terminal windows over `internal/spawn` (see "Multi-window spawn" below). Each spawned window runs the composed argv `portal open --session <name> --ack <batch>:<token>` (or `--path <dir>` for a mint surface); `--ack` is a hidden internal flag on `open` (`cmd/open.go`) — the spawned window writes its `@portal-spawn-<batch>-<token>` server-option marker as its last act before the attach/mint handoff so the burst can confirm it came up. There is no `portal spawn` CLI.

**Diagnostics:** `cmd/doctor.go` — `portal doctor [--fix]` runs an ordered catalog of read-only health checks across the resurrection machinery, renders one line per check, and drives a scriptable exit code (0 iff all pass). It is bootstrap-exempt (starts no server, heals nothing on the read-only path), subsumes the retired `state status`, replaces `clean` (via its `--fix` repairs), and folds in the retired `spawn --detect` as a host-terminal check line. `--fix` applies low-stakes, reversible-by-reconstruction repairs — prune stale `hooks.json` entries, prune stale `projects.json` records, sweep old logs — then re-diagnoses, with the exit code driven solely by the post-repair pass.

**Teardown:** `cmd/uninstall.go` — `portal uninstall` removes ONLY Portal's tmux-runtime footprint: it kills `_portal-saver` (delivering SIGHUP so the daemon flushes a final atomic state) then unregisters the global hooks, and touches NO files — saved sessions and config survive, and `portal open` re-bootstraps the runtime on the next run. Bootstrap-exempt.

**Resume-hook command:** `cmd/hooks.go` — `portal hook set/rm/list` manages per-pane resume hooks (see "Resume hooks" below). `hook` is the canonical verb; `hooks` is retained as a permanent, silent cobra alias (back-compat for machine-written `portal hooks set …`). Bootstrap-exempt (config-file only).

### Resolution chain (cmd/open.go → internal/resolver/)

Path arguments go through: direct path detection → alias lookup → zoxide query → TUI fallback with filter text.

### Session connection (two modes)

- **Outside tmux (bare shell):** `AttachConnector` uses `syscall.Exec` to hand off the process to `tmux attach-session -A` (atomic, replaces the portal process)
- **Inside tmux:** `SwitchConnector` uses `tmux switch-client` (two-step: create detached session, then switch)

### Config path resolution (cmd/config.go)

All config files (`projects.json`, `aliases`, `hooks.json`, `prefs.json`, `terminals.json`) resolve via `configFilePath`: per-file env var → `XDG_CONFIG_HOME/portal/` → `~/.config/portal/`. `terminals.json` (the host-terminal spawn escape hatch, `PORTAL_TERMINALS_FILE`) is user-authored and read-only at spawn time, loaded once via `buildResolver`; like `prefs.json` it maps to the empty log component (outside the closed log-component vocabulary). On first access, `migrateConfigFile` performs a one-shot move from the old macOS path (`~/Library/Application Support/portal/`) — never overwrites existing files at the new path. `configFileComponents` maps each file to its log component for the migrate breadcrumb; `prefs.json` maps to the empty component (`""`), which suppresses the migrate log emission while still performing the move (prefs is outside the closed log-component vocabulary). `prefs.json` resolves via `prefsFilePath` / `loadPrefsStore` (`cmd/config.go`, alongside `loadProjectStore`), wired into the TUI at construction through the `WithInitialMode` / `WithModePersister` / `WithAppearance` options in `cmd/open.go` (the appearance pref is read once at construction and feeds the owned-canvas mode gate).

### Logging & observability

All logging flows through `internal/log` — bind a component logger once per package via `var logger = log.For("<component>")`, then call `logger.{Debug,Info,Warn,Error}(msg, attrs...)` with slog attrs. `main.go` calls `log.Init` first thing (before Cobra parses argv, so `process_role` is resolved from `os.Args`) and owns the single `os.Exit` via a panic-recovering exit shape — **bare `os.Exit` outside `main` is prohibited** (the daemon self-eject is the one sanctioned exception, and it pairs itself via `log.Close`). `process:` lifecycle markers (`start` / `exit` / `exec` / `panic` / `log-level resolved`) bracket every run and **bypass the level filter** so they are always present as forensic tripwires.

- **Format & rotation:** human-readable text with a `component:` prefix, so `grep "<component>:" portal.log` reconstructs a subsystem's behaviour at the production-default level (JSON also available from the handler). `portal.log` is a symlink to a calendar-daily `portal.log.<date>`; a size-cap valve overflows to `portal.log.<date>.N`; past-day files are sealed `chmod 0400`. Retention defaults to 30 days with one INFO breadcrumb per deleted file.
- **Closed taxonomy:** 17 component names (the `spawn` component was added by the multi-window-spawn feature, and the `resolve` component by the cli-verb-surface-redesign feature — both spec-governed amendments, not call-site inventions) + a closed attr-key vocabulary + four levels (production default **INFO**). New components/attrs require amending the spec — never invent at call-site. The `spawn` component's attr keys (`batch`/`terminal`/`bundle_id`/`resolution`/`session`/`ack`/`opened`/`total`/`detail`) are likewise spec-governed. The `resolve` component (bound + emitted only in `cmd/open.go`; `internal/resolver` stays a pure, log-free library) carries the `open` resolution-decision line — message `resolved`, attr keys `target` (raw input) / `domain` (session/path/alias/zoxide, or `miss`) / `resolved_path` (resolved directory, or session name for a session hit; empty on a miss) — one INFO line per bare positional resolved through the guessing chain; glob (and pinned) targets are deterministic and emit no line. Baseline attrs (`pid` / `version` / `process_role`) are injected per-record by the handler.
- **Env:** `PORTAL_LOG_LEVEL` (`debug`/`info`/`warn`/`error`, invalid → `info` + WARN), `PORTAL_LOG_ROTATE_SIZE` (e.g. `500M`/`1G`, default 500 MiB), `PORTAL_LOG_RETENTION_DAYS` (default 30). `portal doctor --fix` forces a full log-retention sweep as a maintenance side-action.
- **What's instrumented:** every `hooks.json` / `aliases` / `projects.json` mutation emits a breadcrumb from the **store method** (the chokepoint, not the caller); cycle summaries (`capture: tick complete`, `bootstrap: orchestration complete`, `restore: skeleton/geometry complete`, `spawn: opened N/N`, the `clean:` sweeps) emit one INFO per cycle with per-item detail at DEBUG; saver/daemon lifecycle and the hydrate exec chain have closed event catalogs.

### DI / testing pattern

All external dependencies use small interfaces (1-3 methods). Commands expose package-level `*Deps` structs (e.g., `bootstrapDeps`, `openDeps`, `hooksDeps`) — tests set these to mock implementations and restore via `t.Cleanup()`. A test that Executes a real command body MUST inject every tmux-touching `*Deps` seam (see the tmux boundary bullet under "Test isolation" — cmd's `TestMain` poisons `TMUX` and the `PORTAL_*` paths package-wide so a missed injection fails loudly). Integration-tagged tests that exec the built binary (e.g. `cmd/root_integration_test.go`, the commit-now suite) build it via `portalbintest` (never a hand-rolled `go build` — the `-tags integration` sandbox must not be dropped) and drive it as a subprocess.

#### Test isolation for daemon-spawning tests

**Lane rule first:** every test that spawns a `portal state daemon` or execs a built `portal` binary MUST carry `//go:build integration`. The unit lane (`go test ./...`) contains no such tests; the integration lane runs serially (`go test -tags integration -p 1 ./...`).

Any test that runs `portal state daemon` directly OR via `portal open` / bootstrap MUST call `portaltest.IsolateStateForTest(t *testing.T) (env []string, stateDir string)` (in `internal/portaltest/isolated_env.go`) and apply the returned env to every spawned subprocess (`cmd.Env = env`). Without this, the spawned daemon inherits the developer's `$XDG_CONFIG_HOME` and writes to the real `~/.config/portal/state/` — the slow-open / empty-previews / zombie-session incident is the canonical example of how a leaked test daemon corrupts the live install. The helper also registers a `t.Cleanup` fingerprint-diff backstop that walks the developer's state dir post-test and fails on any delta; the backstop is **defence-in-depth, not a substitute** for the env override. No lint or CI enforcement exists — the rule is contributor-discipline plus the structurally-mandatory `*testing.T` parameter (which prevents importing the helper from non-`*_test.go` code).

The returned env slice also carries a **poisoned `TMUX`** (`portaltest.PoisonedTmuxSocket`): the test process usually runs inside the developer's real tmux, and a subprocess inheriting the ambient `TMUX` (or falling back to the default socket) silently attaches to the REAL server — a real orphan daemon was observed capture-paning the developer's live sessions this way. Every subprocess that needs a server must append its own `TMUX=<test socket>,0,0` AFTER the slice (`exec.Cmd` env dedupe is last-wins); one that forgets fails loudly at connect time. `SpawnIsolatedDaemon` does this via its required `tmuxSocketPath` parameter.

For daemon-spawning tests, prefer `portaltest.SpawnIsolatedDaemon` (which pins both `PORTAL_STATE_DIR` and `TMUX`) + `portaltest.RegisterSubprocessCleanup` over hand-rolling `exec.Cmd` / `cmd.Wait`. These helpers wrap the canonical SIGKILL+Wait+reap pattern that prevents zombie subprocesses and ensures the test's `*testing.T` sees deterministic teardown order — both are load-bearing on macOS where unreaped subprocesses can hold open the state dir's file descriptors past the test's `t.Cleanup` window. Fixtures whose tmux server can host writers at teardown (a saver daemon's SIGHUP flush, session-closed hook subprocesses) should also call `portaltest.RegisterStateDirTeardownGuard(t, stateDir)` — registered after `IsolateStateForTest`, before `tmuxtest.New` — so the bounded quiescence wait runs between kill-server and the TempDir RemoveAll (fixes the "TempDir RemoveAll: directory not empty" teardown flake).

**ABSOLUTE INVARIANT — a test must NEVER mutate or affect the real system: not the filesystem outside its temp dirs, not the default tmux server, and NOT any OS process the test did not spawn.** All tests run on the developer's machine (there is no CI), so the developer's live `portal state daemon` is always present during a test run. Isolation therefore spans three boundaries, all mandatory:

1. **Filesystem / state** — `IsolateStateForTest` (above); the fingerprint backstop enforces it.
2. **tmux** — only a per-test `-L <unique>` socket via `tmuxtest`; **never** the default socket — including **indirectly**: a test that Executes a real Cobra command body (no `*Deps` injection) inherits that body's production wiring, and several bodies build `tmux.DefaultClient()`, which honours the ambient `TMUX` — the developer's REAL server when tests run inside tmux. *Incident of record #2 (pre-redesign history — `portal state cleanup` was since folded into `portal uninstall` and one of the two tests, `TestStateUserFacingSubcommandsExitZero`, removed; the surviving `TestVersionGuard_NotInvokedForExemptCommands` now Executes the `uninstall` body with its `*Deps` injected): two cmd tests ran the real `portal state cleanup` body uninjected — on every `go test ./cmd` it kill-sessioned the developer's live `_portal-saver` (SIGHUP'ing the real daemon) and unregistered the real global hook table. This was the saver-churn mystery in the developer's portal.log.* Any test Executing a real command body must inject the command's `*Deps` seam for every tmux-touching dependency. **Enforcement is structural:** the cmd package's `TestMain` poisons `TMUX` package-wide (the tmux-boundary counterpart of its `PORTAL_*_FILE` path poison), so a missed injection dials a dead socket and fails loudly instead of silently reaching the real server — this poison immediately caught two further offenders when introduced (an uninjected full-production-bootstrap `portal list` Execute — the creator of the phantom 0.8.3 saver daemons — and a partially-injected commit-now deps struct whose nil `IsRestoring` defaulted to a live `DefaultClient` query). Tests that assert inside/outside-tmux *behaviour* must `t.Setenv` `TMUX` themselves; tests needing a real server use `tmuxtest`'s explicit `-S` sockets, which override `TMUX`.
3. **Processes** — the daemon is identified **machine-wide by argv** (`pgrep -fx '^portal state daemon( |$)'`), so state/file isolation does **NOT** make a test daemon distinguishable from the developer's real daemon at the process level. An unscoped sweep in a test will SIGKILL the real daemon. **Enforcement is structural, not disciplinary:** `state.PgrepPortalDaemons` is filtered by a **default-deny daemon-pgrep sandbox** (`internal/state/pgrep_sandbox.go`, `//go:build integration`, compile-time absent from production) that surfaces only daemons the test registers. `IsolateStateForTest` enables it; `SpawnIsolatedDaemon` (orphans, by pid + state dir) and the saver-read helpers (by live `_portal-saver` pane_pid — respawn- and daemon.pid-manipulation-immune) register ownership. Because the sweep only SIGKILLs PIDs its `Pgrep` seam returns, an unregistered PID (the real daemon — including after its own churn respawns) can never be a kill candidate. `TestPgrepSandbox_ExcludesUnregisteredPID` proves the exclusion. **Do NOT** run the production orphan sweep, `state.PgrepPortalDaemons`, or any `pgrep`/`pkill`/broad-signal path in a test without `IsolateStateForTest` active, and never reintroduce a machine-global destructive process op in a test. *Incident of record: a global-`pgrep` sweep integration test repeatedly SIGKILL'd the developer's live daemon (unpaired `process: start` with no `exit` in portal.log) — file isolation held; process isolation did not exist until this sandbox.*

   **The sandbox crosses the process boundary via a registry file.** A test-spawned `portal` binary runs its bootstrap orphan sweep in its own process, where in-process registrations don't exist — and the sweep runs at step 4, *before* `EnsureSaver`, so its legitimate set is empty and an unfiltered enumeration would SIGKILL the real daemon (verified with a pgrep-shim canary harness during the fix). Two pieces close it: `portalbintest` builds every test-staged binary with **`-tags integration`** (compiling the sandbox in; the shipped release binary never carries it), and `IsolateStateForTest` sets **`state.SandboxRegistryEnv`** (`PORTAL_TEST_SANDBOX_REGISTRY`) — the path of a file listing test-owned state dirs, re-read on every enumeration (`SpawnIsolatedDaemon` appends each orphan's dir). Env set + file unreadable ⇒ enabled with zero owned dirs (subprocess sweeps kill nothing). `TestPgrepSandbox_RegistryEnvActivatesCrossProcess` pins the contract. Never hand-roll `go build` for the portal binary in a test — route through `portalbintest` so the tag is never dropped.

### Server bootstrap

`PersistentPreRunE` runs a ten-step `bootstrap.Orchestrator` (in `cmd/bootstrap/`) for commands needing tmux (all except the bootstrap-exempt `skipTmuxCheck` set: `version`, `init`, `help`, `alias`, `hook`, `doctor`, `uninstall`, `state`, and the internal `__complete` completion verb). Step ordering is load-bearing; "Return" is the post-step boundary, not a numbered step:

1. **EnsureServer** — start the tmux server if not running. `StartServer` creates the server with an initial detached session named `_portal-bootstrap` (`tmux.PortalBootstrapName`) — an **intentional, permanently-present anchor**, not a leak. A tmux server with zero sessions exits immediately, so this session keeps the freshly-started server alive long enough for the rest of bootstrap (notably `EnsureSaver` and `Restore`) to run. It is created on every cold server start and is **never killed by production code**; like `_portal-saver`, its leading-underscore name is filtered out of `ListSessions` and excluded from `sessions.json` capture, so it is invisible to the picker and never saved/restored. Do **not** mistake its lingering presence in raw `tmux list-sessions` output for an orphan to clean up — it is load-bearing.
2. **RegisterPortalHooks** — install global tmux hooks (e.g. `client-attached` running `portal state signal-hydrate`) idempotently. Registration is declarative per-event "ensure exactly one": each managed event is read via `ShowGlobalHooksForEvent` (`show-hooks -g <event>`) and converged to a single Portal entry carrying the current desired body. Per-event reads are mandatory because the no-arg global `show-hooks -g` is blind to `pane-*`/geometry `window-*` events on tmux 3.6b — the blind spot that previously let duplicate `portal state notify` hooks stack unbounded on `pane-focus-out`/`window-layout-changed`; any pre-existing stack now self-collapses to one on the next bootstrap (no dedicated cleanup migration). `UnregisterPortalHooks` (invoked by `portal uninstall`) reaps via the same per-event seam, and `managedEvents` is the single source of truth from which both the teardown event set and the teardown fingerprint set are derived. The former `migrateHydrationHooks` / `migrateSessionClosedHook` one-shot helpers are folded into this unified convergence path.
3. **Set `@portal-restoring`** — must precede saver/restore so the daemon and hydrate helpers can detect they are inside a restoration window.
4. **SweepOrphanDaemons** — best-effort `pgrep -fx '^portal state daemon( |$)'` enumeration; identity-checked SIGKILL of any candidate that is not the `_portal-saver` pane's PID. Runs before `EnsureSaver` so the new saver-pane daemon's first tick is uncontested by leftover daemons from prior server lifetimes. Composes with the kill-barrier in `EnsureSaver` (which handles the single recorded `priorPID`) to converge multi-orphan installs to a singleton in one bootstrap. Errors log at WARN under `ComponentBootstrap` and are swallowed; never escalates to a fatal abort.
5. **EnsureSaver** — bootstrap (or version-upgrade) the `_portal-saver` detached session that hosts `portal state daemon`. Best-effort; failure surfaces as a `SaverDownWarning`.
6. **Restore** — invoke `internal/restore` to reconstruct skeleton + geometry + scrollback FIFOs from the saved state. Never escalates to a fatal abort; corrupt state surfaces as a warning.
7. **EagerSignalHydrate** — best-effort write of the hydrate signal byte to every freshly-armed `@portal-skeleton-*` pane's FIFO. Production wiring uses `state.DefaultFIFOSignaler` / `state.SendHydrateSignal` — the no-seam entry point that pins `OpenFIFOForSignal` + `time.Sleep`. The lower-level `state.WriteFIFOSignal` is the seam-bearing variant used only for retry-ladder unit tests. Runs while `@portal-restoring` is still set so daemon `captureAndCommit` suppression remains in force during helper-driven scrollback replay. Eliminates the per-session signaling gap that would otherwise leave N-1 saved sessions' helpers waiting on `client-attached` events that never fire for non-attached sessions. A non-nil return logs at WARN under `ComponentBootstrap` and is swallowed; never escalates to a fatal abort.
8. **Clear `@portal-restoring`** — fatal on failure (the marker must not leak past bootstrap).
9. **CleanStaleMarkers** — best-effort cleanup of `@portal-skeleton-*` server-option markers whose paneKey is no longer represented by a live pane. Runs after Clear so it observes post-restore tmux state, and before Sweep so any stale markers protecting orphan FIFOs are unset first, allowing those FIFOs to be reclaimed in the same bootstrap.
10. **SweepOrphanFIFOs** — best-effort cleanup of orphan `hydrate-*.fifo` files whose paneKey is no longer represented by a live `@portal-skeleton-*` marker.

(The former step 11 **CleanStale** — hooks stale-entry cleanup — was removed from the orchestrator and re-homed on the `_portal-saver` daemon, which runs it (alongside a throttled stale-`projects.json` prune) on a throttled idle-tick cadence; `portal doctor --fix` is the manual backstop.)

After step 10 the orchestrator returns the accumulated warnings. How they (and the per-step progress) reach the user depends on the path — see **Cold-path startup flip** below: the warm / CLI path runs the bootstrap synchronously and drains warnings post-bootstrap (or after the `LoadingMinDuration` = 1.2s loading-page pad), while the cold + TUI path runs it concurrently and warnings ride the progress channel to a post-load notice band.

### Cold-path startup flip (concurrent bootstrap)

On the **cold + TUI path** — a fresh tmux server *and* the TUI picker, decided by a cheap `tmux has-server` probe (`shouldRunConcurrentBootstrap`) — the 11-step bootstrap runs **concurrently** with the TUI instead of synchronously in `PersistentPreRunE`. Bubble Tea launches immediately on the honest loading page; the orchestrator runs in a goroutine streaming a `tea.Msg` per real step (and per restored session) over a **progress channel** (`cmd/bootstrap_progress.go`, `cmd/bootstrap/progress_emitter.go`) carrying `serverStarted` + progress, and the model transitions to Sessions on `BootstrapCompleteMsg`. The 11 real steps map to **5 friendly labels** (`internal/tui/loading_progress.go` is the single mapping authority); only `Restoring sessions` carries the `N/M` counter, fed by a per-session callback injected at `internal/restore/restore.go`. A **fatal** step becomes a `tea.Quit` with an in-TUI `state.red` error frame + non-zero exit (not a `PersistentPreRunE` error return); **soft warnings** ride the channel and surface as a post-load notice band after the picker appears (replacing the old stderr/alt-screen flush). The first real paint gates on the detect-or-timeout appearance gate so the correct canvas shows from frame one.

The **warm path** (server already up — `serverStarted=false`) and the **CLI / direct-path** (`!isTUIPath`) keep today's untouched synchronous bootstrap with no loading page and zero new risk. The TUI is inert during loading (animation only), which contains the restore/daemon race surface. Startup-ordering and warm-path parity are covered by `cmd/concurrent_*_test.go` (daemon-spawning tests under `IsolateStateForTest`).

### Daemon self-supervision

The `portal state daemon` runs a per-tick **saver-membership self-check** before each `captureAndCommit`. On every ticker fire (1s `TickerPeriod`), the daemon asks "am I still the pane process of the live `_portal-saver` session?" via `tmux.SaverPanePIDOrAbsent` plus a `pane_pid == os.Getpid()` comparison. A failing probe (saver absent, pane pid mismatch, or transient tmux error) increments an in-process consecutive-absence counter. A passing probe resets it to 0.

When the counter reaches `selfSupervisionHysteresisTicks = 3` consecutive ticks (declared in `cmd/state_daemon.go`; clamp-floor + clamp-ceiling pinned by `TestSelfSupervisionHysteresisTicks_ClampInvariant`), the daemon ejects in a fixed, load-bearing order:

1. Emits `daemon: self-eject ticks=N threshold=3` (INFO under the `daemon` component).
2. Calls `log.Close(0)`, which emits the paired `process: exit code=0` lifecycle marker (and does **not** itself exit) — so the self-eject stays a *marked* termination and the four-way `process:` classification holds (no unpaired `process: start`).
3. Calls `os.Exit(0)` immediately (via the `osExit` seam) — bypassing `daemonShutdownFunc` so the divergent-view daemon does **not** execute one more `captureAndCommit` / `gcOrphanScrollback` cycle on its way out (same reasoning as Component A's straight-to-SIGKILL choice). No `daemon: shutdown` line fires on this path.
4. Leaves `daemon.pid` stale on disk. The next acquire's Component C pre-check handles this correctly (recorded PID is dead → pre-check skips → new daemon acquires cleanly). Cleanup logic to delete `daemon.pid` before the eject would be racy against a concurrent pre-check and would invert the layered-enforcement contract — **do not add it**.

The hysteresis value justification is documented in-source as a comment block above the constant (per-scenario tick counts measured against real tmux 3.6b, 2× safety factor, clamp result, measurement date, binary version). The integration-tagged harness `cmd/state_daemon_hysteresis_measurement_test.go` re-runs the experiment to verify the safety-factor invariant whenever it executes.

### Resume hooks

Per-pane hooks are registered via `portal hook set --on-resume "cmd"` and persisted in `hooks.json` (the CLI verb is `hook`; `hooks` remains a permanent, silent cobra alias — the `hooks.json` filename and the `hooks` log component keep the plural, unchanged). `portal hook rm --on-resume` removes the entry for the current pane (resolved from `$TMUX_PANE`); pass `--pane-key <key>` as a literal pass-through to remove an entry for any hook key, bypassing the `$TMUX_PANE` requirement (enables pruning of entries whose pane no longer exists).

**Hooks are keyed by a rename-immune hook key, not the mutable session name.** The key is `<@portal-id or session_name>:window.pane` — prefer the immutable `@portal-id` session user-option (stamped at creation; see the `session` row), else fall back to the session name for legacy/un-stamped sessions (the name fallback coincides with the on-disk key, so **no migration** and no `hooks.json` re-keying). The fix's central invariant is that **every site that produces or consumes a key derives it by the identical rule**: registration reads it live via `tmux.ResolveHookKey` (`HookKeyFormat`), stale-cleanup enumerates live keys via `tmux.ListAllPaneHookKeys` (same format), and restore bakes it from **saved** state via `tmux.HookKey(sess.PortalID, sess.Name, w, p)`. Keying off `@portal-id` (frozen at creation, untouched by `rename-session`) is what fixes the rename-orphans-hook bug: renaming a session — externally or via the in-TUI `r` modal — no longer desynchronizes the key, so the registered hook still fires after the next reboot. `@portal-id` rides `sessions.json` (`Session.PortalID`) and is re-stamped on the recreated live session at restore, so it survives repeated reboots and post-restore stale-cleanup (which would otherwise re-orphan the just-fired hook).

They fire **only inside the hydrate helper's exec chain** (`portal state hydrate`), launched as the initial process of each restored pane via `respawn-pane -k` during bootstrap step 6. After scrollback replay the helper looks up the baked `--hook-key` in `hooks.json` and exec's either `sh -c '<HOOK>; exec $SHELL'` or a bare `$SHELL` — meaning hooks fire on reboot recovery, not on every detach/reattach inside the same tmux server lifetime. The firing path **must never read the live `@portal-id`** (an ordering trap): the baked key is a pure function of saved state, so firing is correct independent of the restore re-stamp's timing. The key uses **saved** window/pane indices so lookups stay addressable across base-index drift. Stale hook entries are cleaned lazily by the daemon and explicitly via `portal doctor --fix`.

### Modern Vivid TUI

The picker is reskinned to **Modern Vivid (MV)** — a token-based design on an owned canvas. This is a *reskin, not a rebuild*: the list model, grouping, filtering, and persistence are behaviourally unchanged; the structure below is the styling layer.

- **Owned mode-matched canvas.** Portal paints its own opaque backdrop on every cell — near-black `#0b0c14` (dark) / near-white `#e1e2e7` (light) — in two layers: leaf styles carry `.Background(canvas)`, and an **outer full-terminal fill** (the last layer in `model.go`'s `View`) pads to `termW × termH` so no edge bleeds and empty rows are painted. The fill never participates in the list's height budget, so a vertical change (e.g. a notice band appearing) re-pads underneath it without perturbing the one-row-per-delegate pagination invariant. `restore.go` set-and-restores the terminal *default* background via OSC 11 (see the `tui` row).
- **Light/dark detection (`appearance_gate.go`).** The canvas mode is chosen by a **detect-or-timeout first-paint gate**: an OSC 11 query (`tea.RequestBackgroundColor`) races a ~50ms timeout; the first to resolve wins (single-resolution — never paint-then-flip), with a **dark** no-answer fallback. The `appearance` pref (`auto|light|dark`) overrides — `light`/`dark` pin the canvas and skip both detection and the wait. `NO_COLOR` is the one carve-out: no canvas, no detection, colourless render on the terminal's native fg/bg (state stays glyph-backed — `●`/`▌`/`⚠`/`✓` — never colour-only).
- **Render structure (files in `internal/tui`).** Shared chrome: `header.go` (block `PORTAL` wordmark + violet `▌` caret + subtitle + 2px rule), `section_header.go` (page label + count + `/ to filter` hint), `footer.go` / `filter_footer.go` (condensed keymap footer + right-aligned `? help`; `right_anchored_row.go` shares their geometry). The **keymap descriptor** (`keymap.go`, `keymapEntry` with `Core` / `RightAligned` / `Destructive` flags) is the single source for **display** — it drives both the footer and the `?` help body; `keymap_dispatch_guard_test.go` guards descriptor↔dispatch drift (dispatch is separate). Modals share a **blank-screen layer** (`modal.go` clears the page to the canvas + `placeModalOnClearedCanvas`) and a joined-panel frame (`panel.go`'s `renderJoinedPanel`): `help_modal.go` (descriptor-driven `?` help), `kill_modal.go` / `delete_modal.go` over the shared `destructive_confirm.go` builder, `rename_modal.go`, and the two-mode `edit_modal.go` (see grouping section). Notice bands route through a **single-slot arbiter** (`notice_band.go`) — the `▌` left-bar no-tags signpost / command-pending banner / inline flash (`sessions_flash.go`) — placed under the title separator, recomputing the list height when they appear/clear. Edge states: `empty_states.go`, `filtering_no_matches.go`. The honest loading screen is `loading_view.go` (+ `loading_progress.go`).
- **Keymap revision (§12.2).** Navigation is **arrows only** — all vim aliases (`h/j/k/l`, `g/G`), `PgUp/PgDn/Home/End`, and uppercase bindings are dropped (move `↑`/`↓`, page `Ctrl+↑`/`Ctrl+↓`); `pinArrowOnlyNav` strips them from the live `bubbles/list` keymap on **both** Sessions and Projects. `k` = kill (Sessions); `d` = delete (Projects); `x` toggles Sessions⟷Projects in both directions; `s` is Sessions-only (cycle grouping); `?` is newly bound (it was previously swallowed so `bubbles/list` couldn't self-toggle help) and opens the per-page help modal.

### Session grouping & tags

The sessions list renders in three modes cycled by the `s` key on the sessions page: **Flat** → **By Project** → **By Tag** → Flat. The cycle is unconditional (always includes By Tag, even with zero tags or zero sessions) and the last-used mode persists in `prefs.json` on every press (best-effort; failure non-fatal). The mode shows in the title via `SessionListTitle()` (`Sessions` / `Sessions — by project` / `Sessions — by tag`); the footer adds only an `s switch view` hint. While the `/` filter is focused `s` is a literal filter character.

- **Tags are directory-anchored (v1).** Tags live on the `Project` record (`projects.json`), not per session. A session's effective tags = its directory's tags, looked up live at grouped-render time. Tags are managed only in the projects edit modal (a Tags chip-field beside Name/Aliases), never per session and with no CLI. The edit modal (`edit_modal.go`, spec §8.2) is a **two-mode (navigate/edit) immediate-persist** model uniform across all three fields — a deliberate behaviour change from the old "tags persist live, Name/Aliases batch" asymmetry, so **Name and Aliases now persist immediately too**. Navigate mode: `Tab`/`Shift+Tab` (or `↑`/`↓`) move between fields, `←`/`→` move across chips + the trailing `+ add` slot (entering a chip field lands on `+ add`); `x` deletes a focused chip; `Esc` closes. Edit mode (one element live, entered by `Enter`/`e`, or `Enter`/`+` on `+ add` which spawns an empty chip already in edit): `Enter` **commits & persists that single item** (no dirty state, no save key, no batch), `Esc` discards just that edit (a brand-new empty chip vanishes). Falling-out rules: empty-on-commit deletes, empty Name reverts to prior, duplicate-on-commit silently dedupes (tags case-sensitive). `Esc` never discards saved work. A tag change reloads projects so the grouping index refreshes. Deferred: per-session `@portal-tags` + `--tag=`, live-grouped filtering, tag exclusion.
- **Session → directory resolution.** Each live session is mapped to its directory via the `@portal-dir` session user-option (stamped at creation from the git-root — by both `create.go` and `QuickStart`, so a session stays anchored to its origin dir even after the pane `cd`s away). When absent (legacy sessions created before the stamp shipped), the grouped render derives the directory from the active pane's `current_path` → git-root and uses it for *this* render, caching the guess **in-memory** (into `m.sessions`, so later rebuilds in the same picker session skip the pane read) — gated to grouped modes only; Flat and the zero-tags signpost perform zero pane reads. The guess is never stamped back to tmux (drift would freeze a wrong dir); it re-derives on the next picker launch, so a session that has `cd`'d back self-corrects. The lookup key and stored `Project.Path` are both reduced via `project.CanonicalDirKey`.
- **Render-layer grouping (the key invariant).** Group headings are **real, non-selectable `HeaderItem` rows** interleaved before each group's session rows by `injectGroupHeaders` — every list row (header or session) is exactly one delegate line, so `bubbles/list` pagination is exact and a page can never render more lines than the viewport (the earlier in-`SessionItem` heading injection drew uncounted extra lines, overflowing the frame so the terminal scrolled the title and cursor off the top — the original cursor-invisible / missing-title / left-shift bug). `HeaderItem.FilterValue()==""` so headers vanish the moment a filter query is typed (flatten-on-filter for free); `ensureSessionRowSelected` + `skipHeaderRow` keep the cursor on session rows for initial selection and arrow/paging nav across group boundaries (the §12.2 keymap revision dropped `g`/`G` and the other vim/page-jump aliases — see "Modern Vivid TUI"). Indentation (delegate): group headers are indented to align with the title box's left edge (`groupHeaderIndent`, col 2); grouped session rows nest one level further in (`groupRowIndent` — cursor at col 2, name at col 4) when `GroupKey != ""`; Flat rows render flush at col 2 as before. By Project is Pattern A (one item per session under its project name; key is canonical path). By Tag is Pattern B (one item per `(session, tag)` pair — a multi-tag session repeats under each heading). Unresolved/untagged sessions collect in pinned **Unknown** (By Project) / **Untagged** (By Tag) catch-alls (counted, empty-suppressed, each with their own `HeaderItem`). Zero-tags-anywhere By Tag shows a "No tags yet" signpost over the plain list (degrade-with-message, not silent flatten). `internal/tui/model.go`'s `rebuildSessionList` is the single chokepoint all entry points (the `s` toggle, `SessionsMsg` refresh, `ProjectsLoadedMsg`, and the projects-edit → sessions transition) route through. **Build constraint:** the picker must not be routed through `lipgloss/tree` — grouping is pure `lipgloss` styling layered into the existing `SessionDelegate`.

### Multi-window spawn (`portal open` multi-target + multi-select)

`internal/spawn` is **one service reached two ways**: the **`portal open` multi-target burst** (`cmd/open_burst.go` / `cmd/open_burst_run.go`, driven when `open` gets 2+ targets or a K≥2-expanding glob) and the **in-process picker burst** (Sessions-page multi-select). Both run the identical flow: detect host terminal → pre-flight → sequential spawn of the N−1 external windows → self-connect the calling window to the Nth. **Net N windows, never N+1** (hard invariant): the trigger window is reused as one session (via the existing `AttachConnector` outside tmux / `SwitchConnector` inside, or a local mint when the trigger surface is a mint), so only the N−1 others are externally spawned. The spawned windows run `portal open --session <name> --ack …` (or `--path <dir>` for a mint), **not** a separate spawn CLI.

- **Detection** is a standalone op (not an adapter method — the unsupported banner needs identity without spawning): outside tmux → env fast-path (`GHOSTTY_*`/`__CFBundleIdentifier`) else self-walk the process tree; inside tmux → enumerate the session's clients via `list-clients`, then **select-winner-then-locality-check** (`detectInsideTmux`, `detect_inside.go`): pick the **triggering client** — the most-active by `client_activity` across **all** clients, local and remote alike, first-listed winning an exact tie — and walk **only that winner**. This gates locality on *the client the user is acting through*, not on "is any host-local client attached to the session" (the inverted filter-then-tiebreak order was the root cause of the remote-trigger-spawns-on-local-terminal bug: a remote trigger with a host-local bystander on the same session used to drive windows onto the wrong machine). Resolves the winner's `client_pid → process-tree → .app` to a macOS **bundle id** (via `defaults read` of Info.plist), **family-matched** (`dev.warp.Warp-*`). A remote/mosh winner (**even with a host-local client also on the session**), an empty client list, or a transient `ps`/`defaults` error on the winner walk → NULL → unsupported → honest no-op (the transient case **fails safe to NULL** — never spawn on uncertainty — and additionally logs a `spawn` WARN). Walking only the winner deliberately drops the old "one flaky `ps` cannot mask a resolvable local client" resilience for the winner — an owned trade of resilience for correctness. Runs **once per picker session**, cached, off the ~50ms first-paint gate; `rebuildSessionList` must not re-walk.
- **Adapters** implement `OpenWindow(command []string) Result` only. Resolution precedence: **config (`terminals.json`) → native Ghostty → unsupported** (within config, most-specific wins: exact bundle id → `.app`/alias → longer glob → bare `*`). The spawn layer composes the command (`<os.Executable()> open --session <session> --ack <batch>:<token>`, or `--path <dir>` for a mint surface) as an **env-self-sufficient argv** (`/usr/bin/env PATH=<picker's PATH> …` with `TMUX`/`TMUX_PANE` stripped — the spawned N−1 must run *out* of tmux) and hands `{command}` to the adapter verbatim, so no adapter/recipe needs an env slot. Using `os.Executable()` (not a PATH lookup) keeps the spawned binary version-identical so the warm-command latch (`state.BootstrappedLatchSatisfied`) stays satisfied → each spawned `open` is an abridged fast-path, not a full bootstrap. All OS/terminal specifics (osascript, `-1712`/`-1743` AppleEvent codes, TCC) are quarantined behind the typed `Result` taxonomy (`OutcomeSuccess`/`SpawnFailed`/`PermissionRequired` + an `OutcomeUnknown` zero-sentinel; opaque `detail`/`guidance`). The **native Ghostty adapter** (`internal/spawn/ghostty.go`) additionally wraps the argv it is handed in an explicit shell fallback — `ghosttyEmbed` renders `wrapWithShellFallback(command)` = `["bash", "-lc", "<rendered composed argv>; exec \"$SHELL\" -il"]` through the shared `renderCommandString` (which owns the nested `'\''` re-quoting; naive concatenation would corrupt the command) — so a burst-spawned window lands at the user's interactive login shell after its session's exec chain finishes instead of dead-ending on Ghostty's "Process exited. Press any key to close the terminal." keypress. `wait after command` is correspondingly **dropped** from `ghosttyScriptTemplate` (the record is now single-field `{command:"%s"}`); the exec'd `$SHELL`, not the flag, keeps the window alive-and-usable. The wrap is argv-agnostic (attach and mint surfaces alike) and Ghostty-scoped by design — the shared `composeOpenArgv`/`renderCommandString` are unchanged, so config `terminals.json` recipes still receive a plain `{command}` with no injected shell metacharacters (a custom terminal's own post-command behaviour is the user's business — an accepted residual).
- **Confirmation is an explicit token ack**, not osascript success: each spawned `portal open … --ack` writes a `@portal-spawn-<batch>-<token>` **tmux server option** immediately before its exec handoff (best-effort — it still execs if the write fails); the burster watches for it with a per-window `spawnAckTimeout` (~8s, timer starts at *each* window's own spawn, so cumulative sequential delay never eats a later window's budget). Missing marker at timeout = failed window. `<batch>`/`<token>` are option-name-safe nanoids (never the renameable session name). The `@portal-spawn-` prefix is namespaced away from `@portal-skeleton-`, so the only all-options enumerator (`ListSkeletonMarkers`) is blind to it; markers self-clean on every terminal path (success / pre-flight abort / reported failure / cancel).
- **Pre-flight + partial-failure contract:** pre-flight `has-session` gates all-or-nothing — any marked session gone ⇒ atomic abort (nothing opens), prune the gone one(s), keep survivors marked (a second Enter retries). Past pre-flight, a per-window failure is **leave-what-opened**: opened windows stay (Portal doesn't own/tear-down host windows), the trigger's self-attach is skipped, and the failed/un-acked sessions stay marked so a retry re-opens exactly those. A `permission-required` result **stops the burst** (every later window would hit the same per-`(source,target)` wall) and surfaces guidance once per batch. Result classification, message renderers, log emission, the exec boundary, the nanoid alphabet, and the net-N split are all single-sourced in `internal/spawn` so the open burst and picker burst cannot drift (byte-identical parity tests).
- **Multi-select mode** (`internal/tui`, `m` key) is a general selection mode on the Sessions list: `m` enters the mode marking the currently-highlighted session as the first selection, and a subsequent `m` toggles the highlighted row (zero-selected still reachable — double-`m` toggles the auto-marked row off, `Esc` clears, and entering while the cursor is on a group-header row enters with zero); `Enter` opens the marked set; `Esc` clears+exits. **Selection is keyed on session identity**, so a multi-tag By-Tag session (Pattern B, one row per tag) marks once and shows `●` on all its rows, and the banner counts distinct sessions. Sticky across filter/paging/regroup/`Space`-preview (pruning only an externally-killed session). N=0 Enter exits; N=1 degenerates to a plain attach; N≥2 dispatches the burst as an async `tea.Cmd` (input-locked while pending — only `Ctrl-C`/`Esc` live, which cancels). On a **resolved-unsupported** host terminal `m` is proactively blocked at entry — it sets a plain transient flash (`multi-select isn't available over a remote connection` for NULL/remote, `… on this terminal` for a named-undriven terminal) and never opens the walkable dead-end mode; the retained reactive `decideBurst` no-op still catches the async in-flight window (mode entered before detection resolved), and the `?` help drops the `m` row while it is blocked (call-site filter in `sessionsHelpKeymap()` — the static `sessionsKeymap()` descriptor is unchanged, so `keymap_dispatch_guard_test` stays green). The persistent **unsupported banner is named-only** — its gate `unsupportedBannerActive()` carries an `!m.detectIdentity.IsNull()` discriminator, so a NULL/remote client keeps its standard `Sessions ··· N` header and By-Tag signpost (there is nothing actionable to show), while a named-undriven terminal still gets the `⚠ unsupported terminal — <name> · <bundleID>` + `see docs` banner. The shared reactive-no-op / CLI-burst copy (`spawn.UnsupportedNoopMessage`) is plain language: `can't open new windows over a remote connection — nothing opened` (NULL) / `can't open new windows in <name> · <bundleID> — nothing opened` (named). Single-slot notice-band precedence: filter line → `Opening n/N…` → transient flash → multi-select banner → unsupported banner → no-tags signpost. Delivered Paper frames: `sessions-multi-select-active`, `sessions-multi-select-preflight-abort`, `sessions-unsupported-terminal` (dark-only; NO_COLOR glyph-backed).
- **State/observability footprint** is near-zero: reads `terminals.json`, writes only transient `@portal-spawn-*` server options; **does not touch** `sessions.json`, the daemon capture loop, `prefs.json`, or restore. The `spawn` component emits one INFO batch summary (`spawn: opened N/N`, `total` = N incl. the trigger's self-attach target; `opened` = sessions surfaced) + DEBUG per-window, from the chokepoint on both callers.
