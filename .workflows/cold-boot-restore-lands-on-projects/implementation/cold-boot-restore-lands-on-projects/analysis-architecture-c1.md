SUMMARY: The cold-route landing-decision deferral is architecturally sound — a single-discriminator early return in transitionFromLoading (progressReceiver != nil, identical to the refetchSessionsAfterRestore guard) that leaks no state and always resolves the decision. sessionsLoaded is provably written in only two places (the warm-only line in transitionFromLoading, after the cold early-return, and the SessionsMsg arm reached only when activePage != PageLoading), so the cold route cannot latch defaultPageEvaluated against the stale interim list; the decision resolves at whichever of the post-restore SessionsMsg / ProjectsLoadedMsg lands second, and the failing-refetch path quits as before — no path strands the interim PageSessions. The warm/cold split stays contained to the 3-line method; conditional complexity does not spread. The pre-existing transition+refetch coupling is duplicated across the two gate-closing arms (LoadingMinElapsedMsg, BootstrapCompleteMsg), but that duplication predates this change and the deferral does not worsen it. Integration coverage is thorough — the seven new tests assert active page (not just list contents), cover both interleavings of ProjectsLoadedMsg, the interim-window invariant, the zero-session non-over-correction, filter routing, and the error-quit path. No actionable architectural concern introduced by this implementation.
