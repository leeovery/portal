- FINDING: previewLogger opened in openTUI is never closed in the inside-tmux return path
  SEVERITY: low
  FILES: cmd/open.go:431-435, cmd/open.go:485, internal/state/logger.go:213-219
  DESCRIPTION: openTUI calls state.OpenLogger to back the preview-attach pipeline, but the resulting *state.Logger (which owns an *os.File) is never closed. In the outside-tmux happy path syscall.Exec replaces the process and the fd is reclaimed by the kernel - invisible. In the inside-tmux preview-Enter path the connector is SwitchConnector.SwitchClient (a normal call, no exec), processTUIResult returns nil, openTUI returns, and the process terminates normally. The fd is still reclaimed by process exit, so the leak has no observable victim today - but the lifecycle is asymmetric: the logger is constructed at TUI build time and survives past the program's stated owner (the running Bubble Tea program). If portal ever gains a long-running mode where openTUI is invoked repeatedly in the same process (REPL, daemon, test harness running the full openTUI path through subprocess-less injection), the fd starts to leak in earnest. Cycle 1's connector refactor moved the connector handoff post-TUI; the logger lifecycle should travel with it. The minor seam asymmetry - the logger is the only resource in this file with no defer close - is the actual issue, not the kernel-cleaned fd.
  RECOMMENDATION: Add a `defer previewLogger.Close()` immediately after the OpenLogger call (Logger.Close already handles nil safely). One line, no behaviour change in production paths, removes the asymmetry that any future maintainer would have to re-derive.
