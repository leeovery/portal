# Analysis Report: Session Rename Orphans Resume Hook (Cycle 3)

## Summary
Cycle 3 confirms the fix is converged. Standards found nothing; the production code is soundly consolidated (single `tmux.HookKey`/`HookKeyFormat` primitives, the extracted `runHookStaleCleanup`, the additive `portal_id` schema field, and the three lockstep-pinned `@portal-id` literal guards including the cycle-2 source-of-truth cross-check). The three surviving findings are all test-layer or API-surface concerns that, after code-verified filtering, are net-negative to act on: two are the same already-considered-and-declined class from cycle 2 (integration-harness / per-test-file scaffolding extraction that would over-couple deliberately self-contained tests), and one proposes removing call-less-but-in-spec public tmux-client API surface that is out of scope for this bugfix and already de-risked by retired doc-comments. No tasks are proposed.

## Discarded Findings
- Rename+reboot integration harness copy-pasted across 5 functions in 3 same-package files (duplication, medium) — This is the SAME cross-file integration-test extraction class that cycle 2 already rigorously evaluated and discarded. Code verification confirms the framing is not genuinely stronger: "Block A" (the isolated-env preamble) is only ~5-6 lines of near-boilerplate setup (`BuildPortalBinaryDir` / `PrependPATH` / `IsolateStateForTest` / `stateDir` + `hooksPath` setup) that each file already annotates with an explicit back-reference ("see the task-3-5 harness rationale") — the rationale already has a single source. After the preamble each of the three files diverges meaningfully: distinct tmux socket prefixes (`ptl-3-5-` / `ptl-3-6-dur-` / `ptl-3-7-`), distinct fixtures (single-pane vs two per-pane cross-fire-detection side-effect files vs legacy-unstamped-no-hook), and distinct non-vacuous guard assertions. The 3-6 helpers (`rebootAndHydrate`, `seedScrollback`) are file-local and tailored to the durability test's two-cycle shape; promoting and widening them to also serve the 3-5 and three-way 3-7 legs is exactly the parameterisation-reintroduces-coupling trap cycle 2 named. The codebase deliberately keeps these integration files self-contained across build-tag boundaries. Higher coupling risk than maintenance benefit for a converged cycle-3; re-proposing an already-declined class is churn, not convergence. Discarded.
- Third redundant AllPaneLister stub in package cmd (duplication, low) — Verified: `mockCleanPaneLister` (clean_test.go) and `stubAllPaneLister` (bootstrap_production_test.go) are structurally identical (`struct{ panes []string; err error }` with one canned-return method). But they are idiomatic per-test-file scaffolding serving genuinely different clusters: `stubAllPaneLister` is documented as scaffolding "consumed by run_hook_stale_cleanup_test.go" (the bootstrap/stale-cleanup test cluster), and `mockCleanPaneLister` is the clean command's own purpose-named local ("implements AllPaneLister for clean command tests", ~11 call sites). Collapsing them would cross-couple the clean-command test file to a stub owned by and documented as belonging to the bootstrap cluster, and would forfeit self-documenting purpose-naming — to save a 4-line struct+method on a low-severity finding. Idiomatic per-test-file scaffolding; discarded per the same reasoning cycle 2 applied to the cmd stale-cleanup helpers.
- Two now-dead name-based wrapper methods `ListAllPanes` / `ResolveStructuralKey` (architecture, low) — The "dead" claim is technically accurate (repo-wide grep confirms zero PRODUCTION callers; only `internal/tmux/tmux_test.go` unit tests and doc-comment mentions), but removal is discarded on three grounds. (1) Out of scope: this is a bugfix; retiring public tmux-client API surface is not part of it. (2) Chesterton's Fence: the spec explicitly asserted the name-based `StructuralKeyFormat` / `ListAllPanes` "remain available for any non-hook structural use" (daemon self-supervision, skeleton-marker cleanup, sessions.json delta); the surviving format constant IS live via `ListAllPanesWithFormat(StructuralKeyFormat)` at cmd/bootstrap/stale_marker_cleanup.go, so the wrapper methods sit one repoint away from a legitimate re-use. (3) Already de-risked: both methods' doc-comments were correctly retired to steer future callers away from name-based hook keying ("used for structural enumeration ... not for hook-key lookup"), which is the actual drift hazard the spec named. The architecture agent itself recommends "Leave as-is unless a lint gate flags unused exported methods." No correctness change required; discarded.
