- FINDING: findOrAppendSession silently drops PortalID when copying a Session — a latent hook-orphan trap in this fix's own domain
  SEVERITY: medium
  FILES: internal/state/capture.go:254-266, internal/state/capture.go:237-248, internal/state/capture.go:183-208
  DESCRIPTION: The whole fix hinges on PortalID riding sessions.json faithfully (spec §Cross-Reboot Persistence: losing the id → bare shell on the next reboot + stale-cleanup deletes the just-fired hook). Yet findOrAppendSession, the one place that constructs a Session by hand in the capture path, copies only Name/Environment/Windows and omits PortalID — despite its own doc-comment claiming "a shallow copy of ps". Today this append branch is UNREACHABLE: mergeSkippedPanes only calls mergePane→findOrAppendSession for sessions where sessionLive is true (live[ps.Name] present), and live is built from fresh.Sessions by buildLiveStructure, so any session that reaches mergePane is already in fresh.Sessions and the loop at :255-259 returns the existing index. So there is no active bug. But this is precisely the seam a future change would tug on. The surrounding capture code already reasons about "a session present in keep but contributing no pane rows (killed between enumeration and pane read — natural churn)"; if anyone later relaxes the sessionLive gate to reintroduce such a session from prev (the natural next step), the reintroduced Session would carry PortalID == "", erasing the id from the snapshot and re-orphaning the hook on the next reboot — the exact failure mode this fix exists to prevent, reintroduced silently because the field-copy is incomplete rather than total. No test covers this (the durability integration test calls CaptureStructure(client, nil, nil, nil) — skipSet nil, prev nil — so the merge path is never exercised with the new field).
  RECOMMENDATION: Make findOrAppendSession copy the whole Session (add PortalID: ps.PortalID to the appended struct literal, or copy ps and zero only Windows) so the "shallow copy of ps" the comment promises is actually total. This closes the trap structurally rather than relying on the current unreachability of the branch, which is an invariant no test pins. If the branch is genuinely believed permanently dead, prefer deleting the append arm and returning an error/panic on the impossible case so the dead code cannot silently mis-copy a future live path.

- FINDING: Byte-identity of the three @portal-id literals is guarded only behaviourally, not statically — the spec's primary risk ("missed key-producing site") has no cheap static tripwire
  SEVERITY: low
  FILES: internal/session/create.go:29, internal/tmux/tmux.go:849, internal/state/capture.go:42
  DESCRIPTION: The fix's correctness rests on three independent embeddings of the literal "@portal-id" — session.PortalIDOption (the const), HookKeyFormat's #{?@portal-id,...} conditional, and captureFormat's trailing #{@portal-id} column — staying byte-identical (spec §Constant, §Risks → Missed key-producing site). The code cannot share a single const into these two raw tmux format strings, which the spec accepts, and consistency IS in fact exercised end-to-end: the real-tmux cross-site test pins HookKeyFormat, and the durability integration test round-trips captureFormat against a live stamped server asserting PortalID re-persists. So drift would be caught — but only by integration-tagged / real-tmux tests, each of which re-declares its OWN copy of the "@portal-id" literal to dodge an import cycle. A one-character typo in captureFormat (e.g. @portal_id) is caught only if the integration suite actually runs; there is no unit-level guard asserting strings.Contains(HookKeyFormat, PortalIDOption) && strings.Contains(captureFormat, PortalIDOption). For the single most load-bearing invariant of this change, a fast static tripwire is cheap insurance.
  RECOMMENDATION: Add a small unit test (no tmux) in internal/tmux or a cross-package guard asserting HookKeyFormat and captureFormat each contain the exact session.PortalIDOption literal (and that PortalIDOption == "@portal-id"). This makes the "all three literals agree" invariant fail in milliseconds on a typo rather than depending on the integration suite being run.
