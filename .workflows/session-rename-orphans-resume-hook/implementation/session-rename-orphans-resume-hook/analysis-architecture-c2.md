  DESCRIPTION: The fix's central invariant requires three independent embeddings of "@portal-id" to stay byte-identical: the source-of-truth constant session.PortalIDOption (create.go:29), the literal inside tmux.HookKeyFormat (tmux.go:849), and the literal inside state.captureFormat (capture.go:42). tmux and state cannot import session (import cycle), so the literals are necessarily duplicated. The two static guards that run in tmux-less CI (tmux/hookkey_test.go, state/portal_id_literal_guard_test.go) each assert only that their own format string contains a *locally-hardcoded* copy of "@portal-id" — they never import or compare against session.PortalIDOption. Consequently, if the source-of-truth constant were ever changed (e.g. to "@portal-uid"), BOTH static guards still pass (they check against their own copy), stamping writes @portal-uid while both format strings still read @portal-id, and every stamped session silently orphans its hook — the exact bug this fix removes, reintroduced at scale. The only tests that would catch this are the //go:build integration + SkipIfNoTmux-gated end-to-end tests (e.g. rename_restore_cleanup_survival_integration_test.go, rename_reboot_*_integration_test.go) which stamp via the real constant and read back through the pipeline. Those do not run in the default `go test ./...` tmux-less path, so the drift class the spec explicitly calls out as the primary risk ("Missed key-producing site") has no fast tripwire against a change to the constant itself. This is a defence-in-depth gap, not a live bug: the current values all agree and are documented as load-bearing at each site.
  RECOMMENDATION: Add a fast, non-tmux-gated binding test in a package that can import all three (cmd imports session + tmux; a cmd-level or session-level *_test.go can import session and tmux, and can white-box-reach captureFormat only via state — so split as needed). Assert session.PortalIDOption == "@portal-id" AND that strings.Contains(tmux.HookKeyFormat, session.PortalIDOption) AND (via an exported test hook or a state-package test importing the constant is impossible due to cycle, so) that the state guard's local literal equals session.PortalIDOption by having the cmd-level test compare both live format strings against the real constant. Minimum viable improvement: in cmd (which already imports both session and tmux), assert strings.Contains(tmux.HookKeyFormat, session.PortalIDOption) and session.PortalIDOption == "@portal-id" so a change to the constant fails fast without tmux. This closes the loop between the source-of-truth constant and the two format-string embeddings that cannot import it.
SUMMARY: The architecture is sound — clean seam decoupling (hook key vs tmux target), single-source cleanup helper, correct ordering-trap handling in restore (baked-from-saved-state key, re-stamp best-effort and non-load-bearing for firing), and comprehensive integration coverage of both rename triggers. The one architectural gap is that the three mandatory @portal-id literal embeddings are only cross-validated by integration-gated tests; the fast tmux-less guards check against locally-hardcoded copies rather than the source-of-truth constant, leaving a change to session.PortalIDOption without a fast tripwire.
