- FINDING: The `<key/glyph> <label>` footer-hint helper is independently re-authored in six render sites
  SEVERITY: medium
  FILES: internal/tui/footer.go:262 (renderFooterEntry), internal/tui/edit_modal.go:561 (editFooterGroup), internal/tui/rename_modal.go:183 (renameModalKeyHint), internal/tui/kill_modal.go:138 (killModalKeyHint), internal/tui/delete_modal.go:144 (deleteModalKeyHint), internal/tui/pagepreview.go:318 (previewFooterHint)
  DESCRIPTION: Every footer/modal-footer renders the same primitive — a key glyph in accent.blue, a one-cell canvas-painted gap, and a label in text.detail, joined horizontally. Six functions implement this byte-identical body (`keySeg := headerStyle(theme.MV.AccentBlue, mode, colourless).Render(key); gap := headerCanvasBg(...).Render(" "); label := headerStyle(theme.MV.TextDetail, ...).Render(label)`). The implementers were aware they were copying: `deleteModalKeyHint`, `previewFooterHint`, and `editFooterGroup` carry comments saying "mirrors killModalKeyHint" / "the shared footer key-hint shape". The shared leaf primitives (`headerStyle`, `headerCanvasBg`) already exist, so the composition is trivial — this is duplication well past the Rule of Three, not premature abstraction. Compounding it, the same `{glyph,label}` concept is modelled by two parallel struct types (`footerGroup` in edit_modal.go:518 and `previewFooterGroup` in pagepreview.go:240). The cost is drift risk: the accent.blue key / text.detail label / single-gap convention is the §3.4/§8.x footer contract, and any future change to it (e.g. the key-glyph colour, the gap width) must be hand-applied in six places or the modals diverge from the footer. `renderFooterEntry` already takes the key token as a parameter, so it (or a thin `renderKeyHint(key, label, keyTok, mode, colourless)` wrapper over it) is the natural single home.
  RECOMMENDATION: Extract one shared `renderKeyHint(key, label string, keyTok theme.Token, mode theme.Mode, colourless bool) string` (or promote `renderFooterEntry` to take key+label directly) and route killModalKeyHint, deleteModalKeyHint, renameModalKeyHint, editFooterGroup, and previewFooterHint through it. Collapse `footerGroup` and `previewFooterGroup` into one `{Key/Glyph, Label}` type. The empty-key case (editFooterGroup's `empty on save = delete` label-only group) is the one variant — handle it with an empty-key early-return inside the shared helper.

  SEVERITY: low
  FILES: internal/tui/kill_modal.go (renderKillModalContent, killModalHeaderRow, killModalBodyRows, killModalConsequenceRows, killModalFooterRow), internal/tui/delete_modal.go (renderDeleteModalContent, deleteModalHeaderRow, deleteModalBodyRows, deleteModalConsequenceRows, deleteModalFooterRow)
  DESCRIPTION: The two modals are the same domain element — a "destructive confirm" panel: a state.red `▲ <Title>` header (identical render), a body of state.red+bold target name followed by a single blank separator row and a word-wrapped text.detail consequence line at body-width 52, and a `y <verb> esc cancel` footer. delete_modal.go's comments confirm the relationship at every method ("Mirrors killModalHeaderRow", "Mirrors killModalConsequenceRows", "mirrors killModalKeyHint", "Mirrors killModalFooterRow"). Both already compose through the shared renderJoinedPanel, so the structural sharing is partial — but the per-compartment render logic is duplicated wholesale. This is the "two layers that mirror each other when the domain type could own the transformation" pattern: a single `renderDestructiveConfirm(title, verb, targetName string, extraBodyRows []string, consequence string, ...)` would own the destructive treatment once. Severity is low because the copy genuinely differs and the modal logic (updateKillConfirmModal / updateDeleteProjectModal) is correctly kept separate; this is a render-layer consolidation, not a correctness risk.
  RECOMMENDATION: Introduce a shared `renderDestructiveConfirm` (or a `destructiveConfirmSpec` struct + one renderer) that takes the title/glyph, the red-emphasised target name, optional extra body rows (the path), the word-wrapped consequence, and the footer verb.
