- FINDING: BootstrapProgressMsg carries Label/Name fields that the consumer ignores and re-derives
  SEVERITY: medium
  FILES: internal/tui/model.go:178-184, cmd/bootstrap_progress.go:247-254, internal/tui/loading_progress.go:84-114
  DESCRIPTION: The cold-boot progress channel ships a friendly Label (and a raw StepName Name) on every BootstrapProgressMsg — the producer computes ev.Label in cmd/bootstrap and copies it onto the message (bootstrap_progress.go:250) — but the only consumer, the model's BootstrapProgressMsg Update arm, folds the event through LoadingProgress.Apply, which reads ONLY Index/RestoreN/RestoreM and re-derives the friendly label from Index via stepLabelTable/LabelForStep. Nothing in internal/tui or its tests reads msg.Label or msg.Name. This is two parallel computations of the same value (the friendly label) kept in sync by nobody: the producer's §10.4 mapping (cmd/bootstrap) and the consumer's §10.4 mapping (loading_progress.go) are independent encodings of the identical 11-step→5-label table, so they can silently drift, and the wire format carries a field that is structurally dead weight. The model.go doc still describes Label/Name as "task 5-4 / 5-3 placeholder" fields, confirming they were meant to be wired but the design landed on consumer-side derivation without removing the producer-side path. The single-source-of-truth intent (loading_progress.go's "SINGLE SOURCE OF TRUTH for that contract") is undermined by a second authoritative copy being produced and transmitted.
  RECOMMENDATION: Pick one authority. Since loading_progress.go is the documented single source and is transport-free/independently testable, drop Label (and Name, if no consumer needs the raw StepName) from BootstrapProgressMsg and from the producer in bootstrap_progress.go:247-254, leaving Index as the stable key the consumer maps. If a consumer genuinely needs Name later, keep it but delete the redundant Label so the friendly mapping exists in exactly one place.
- FINDING: SessionItem/ProjectItem Description()/Title() are production-dead and the attached-marker literal has drifted from the token render path
  SEVERITY: low
  FILES: internal/tui/session_item.go:124-138, internal/tui/project_item.go:33-41
  DESCRIPTION: The custom SessionDelegate/ProjectDelegate render rows entirely through renderSessionRow / renderRowLine; only FilterValue() is consumed off the items by bubbles/list. SessionItem.Title()/Description() and ProjectItem.Title()/Description() are referenced only by tests now (no production caller). That alone is harmless leftover, but SessionItem.Description() additionally hard-codes the string `label + "  ● attached"` — duplicating both the attached-marker glyph/text and its two-space spacing — independently of the attachedMarker const (session_item.go:57) and the token-styled render in renderSessionRow. The two have already diverged in form (the live render builds a fixed-width column-aligned slot; Description() builds a bare two-space concatenation), and the literal "● attached" sits outside the centralised marker constant. The colour-literal guard cannot catch it because it is a plain string, so a future change to the marker text would silently leave Description() stale. This is a latent inconsistency seam: a dead method that re-authors a piece of the row vocabulary instead of deriving from the const.
  RECOMMENDATION: Either delete the now-dead Title()/Description() methods on both items (update the tests that exercise them to assert the live render path instead), or, if they are retained as the item's logical text projection, have Description() reuse the attachedMarker const rather than re-spelling "● attached" so the marker text lives in one place.
- FINDING: noMatchesFooterEntries filters the input-active footer by a magic label string
  SEVERITY: low
  FILES: internal/tui/filtering_no_matches.go:76-86, internal/tui/filter_footer.go:56-66
  DESCRIPTION: noMatchesFooterEntries derives the §7.3 reduced footer by copying filteringFooterEntries() and dropping the entry whose Label == "browse results" (filtering_no_matches.go:80). The correspondence is held by an exact-string match against a label defined in a different file (filter_footer.go:62). If the input-active footer's "browse results" copy is ever reworded, the filter silently stops removing the entry and the no-matches footer regains a browse-results hint that points at zero results — a correctness regression with no compile-time or obvious-test signal. This is correctness-by-string-coincidence across a file boundary, the same class of fragility the codebase otherwise avoids (the keymap descriptors model footer membership as a Core flag rather than label matching).
  RECOMMENDATION: Model the membership structurally rather than by label text — e.g. have filteringFooterEntries build from a small set where the browse-results entry is identifiable without its display string (a flag, or compose the no-matches set directly from the shared `type`/`esc` entries) so a copy change cannot break the filter.
