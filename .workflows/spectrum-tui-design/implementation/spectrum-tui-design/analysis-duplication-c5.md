- FINDING: Window-count pluralisation re-implemented across two modal/row renderers
  SEVERITY: low
  FILES: internal/tui/session_item.go:82-87 (windowLabel), internal/tui/kill_modal.go:54-60 (killWindowCount)
  DESCRIPTION: Two independently-authored helpers implement the identical "singular for exactly 1, plural otherwise" window-count rule. windowLabel returns `1 window` / `N windows` for the session row's count slot; killWindowCount returns `· 1 window` / `· N windows` for the kill modal's name-row trailer. The pluralisation decision (the only logic in both) is duplicated, and a swatch fixture hardcodes a third instance of the same phrasing (internal/capture/swatch.go:237 — `"3 windows"`, acceptable as test-fixture copy). This is only two production sites (below the strict Rule-of-Three), so the bar for extracting is borderline; flagged because both encode the same domain rule (English window pluralisation) that should not be able to drift — e.g. if a future "0 windows" edge ever needs special wording, it must be changed in lockstep.
  RECOMMENDATION: Extract one `windowCountWord(count int) string` (or `pluralWindows`) returning the bare `N window`/`N windows` phrase, and have killWindowCount prepend its `· ` lead rather than re-deriving the singular/plural branch. Keep the two call-site wrappers (one for the row slot, one for the modal trailer) but route both through the single pluralisation core. Low priority — small surface, no current drift.

- FINDING: pad-to-width-with-background helpers near-duplicated (canvas vs tint variants)
  SEVERITY: low
  FILES: internal/tui/header.go:224-230 (headerPadRight), internal/tui/notice_band.go:329-335 (noticeBandPadRight)
  DESCRIPTION: headerPadRight and noticeBandPadRight are structurally identical: each takes (seg, segWidth, w), returns seg unchanged when segWidth >= w, else JoinHorizontal(seg, style.Render(strings.Repeat(" ", w-segWidth))). The ONLY difference is which background style paints the trailing pad — headerCanvasBg(...) (the owned canvas) vs noticeBandTintStyle(tint, ...) (the band's tint). The notice_band.go doc comment even calls this out: "It mirrors headerPadRight but pads with the band's tint instead of the canvas." Two byte-for-byte copies of the same right-pad geometry now exist, so a change to the pad rule (e.g. clamping, or a different fill glyph) must be made twice. (padTo in session_item.go:472 and padLineToCanvasWidth in model.go:3761 are related but legitimately distinct shapes — padTo returns an unstyled string the caller backgrounds, padLineToCanvasWidth strips trailing bare spaces first — so they are not part of this pair.)
  RECOMMENDATION: Extract a single `padRightWithStyle(seg string, segWidth, w int, fill lipgloss.Style) string` carrying the shared "return-if-full, else join the styled pad" geometry. Have headerPadRight pass headerCanvasBg(mode, colourless) and noticeBandPadRight pass noticeBandTintStyle(tint, mode, colourless) as the fill style. Both keep their terse mode/colourless-bound signatures as thin wrappers; the geometry lives once.

  DESCRIPTION: The rule "AppearanceLight → Light canvas; AppearanceDark and AppearanceAuto → Dark canvas (the §2.6 no-answer fallback)" is encoded twice. newAppearanceGate is the production resolver (it additionally tracks pinned/pending); modeFromAppearance is a stripped copy in the capture swatch that maps the same three prefs.Appearance values to the same theme.Mode.   RECOMMENDATION: Promote the appearance→mode decision to a single exported helper both packages can call — e.g. `theme.ModeForAppearance(prefs.Appearance) theme.Mode` in (or beside) the theme package, returning Light only for AppearanceLight and Dark for Dark/Auto. If the prefs→theme import direction is undesirable, place the helper where both already depend (theme is imported by both).
