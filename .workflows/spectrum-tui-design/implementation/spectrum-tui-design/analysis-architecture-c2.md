- FINDING: Keymap descriptor is the documented "single source of truth" but governs only display, never dispatch
  SEVERITY: medium
  FILES: internal/tui/keymap.go:14-49, internal/tui/help_modal.go:10-15, internal/tui/footer.go:52-66, internal/tui/model.go:3056-3083, internal/tui/model.go:2322-2358
  DESCRIPTION: The per-page keymapEntry descriptors (sessionsKeymap / projectsKeymap / previewKeymap) are presented across many doc comments as the single source of truth for each page's keybindings, "so a binding change updates the footer and the help together." That claim is true only for the two *display* surfaces (the condensed footer and the ? help modal). The actual key dispatch in Update is an entirely separate hand-coded switch over literal strings — isRuneKey(msg, "k"), isRuneKey(msg, "x"), case "s", keyIsCode(msg, tea.KeyEnter) — with no structural link to the descriptor (the comments even concede "Key ... is a display token, not a tea key code — the live dispatch ... owns the actual key matching"). The architectural consequence: a keybinding is defined in two unconnected places, and the compiler cannot catch divergence. Rebind kill from `k` to some other key, or add a new bound key, and the footer/help can silently advertise a key that dispatch no longer honours (or omit one it does), with nothing failing to compile or test. The descriptor's framing oversells a single-source guarantee it only partially delivers — the dispatch half is the half a user actually feels. This is partly inherent to the reskin constraint (the dispatch switch pre-existed and had to stay behaviourally identical), so the risk is bounded, but the "single source of truth" doc framing should be scoped honestly to "footer + help display," and ideally the descriptor's Key field should be the value the dispatch switch consumes (or a test should assert every descriptor Key is matched by a dispatch arm) so the two cannot drift.
  RECOMMENDATION: Either (a) drive the live-dispatch key matching from the same descriptor Key values (so one edit moves display and dispatch together), or (b) keep them separate but add a guard test that asserts every non-help descriptor Key has a corresponding dispatch arm and vice-versa, and narrow the "single source of truth" comments to say "single source of truth for the footer + help *display*," removing the implication that dispatch is covered.

- FINDING: Three parallel footer-binding source models feed the footer family
  SEVERITY: low
  FILES: internal/tui/keymap.go:14-49, internal/tui/filter_footer.go:35-49, internal/tui/modal_footer.go:19-26, internal/tui/model.go:1069-1075
  DESCRIPTION: The footer/help surfaces are fed by three distinct value types plus a fourth ad-hoc source: keymapEntry (the descriptor — Core/RightAligned/HelpKey/HelpAction), filterFooterEntry (per-glyph multi-colour key clusters), footerHintGroup (a bare {key,label} for the edit + preview footers), and commandPendingHelpKeys() which returns a raw []key.Binding that footer.go then re-maps into filterFooterEntry. All four ultimately funnel into the single leaf renderKeyHint, so the leaf primitive is shared — but the model layer above it is fragmented across four representations of the same {glyph,label} concept. Each has a genuine distinguishing need (filterFooterEntry's per-glyph colours, keymapEntry's footer/help split), so this is not a clear over-engineering defect; but the command-pending path is the weak seam: it alone bypasses the descriptor world entirely (a []key.Binding from the model), re-deriving its glyphs/labels in commandPendingFooterEntries with bespoke `if glyph == "enter" { glyph = "⏎" }` string-rewriting that the descriptor path already encodes declaratively via HelpKey. That command-pending footer would sit naturally as a keymapEntry slice like every other footer.
  RECOMMENDATION: Leave keymapEntry / filterFooterEntry / footerHintGroup as-is (each earns its shape), but fold commandPendingHelpKeys()'s raw []key.Binding into a keymapEntry descriptor (or at least a filterFooterEntry slice authored directly), eliminating the one-off enter→⏎ rewrite and the only footer source that lives outside the descriptor/entry vocabulary.

- FINDING: Dead per-modal key-hint wrappers retained with no production callers
  SEVERITY: low
  FILES: internal/tui/kill_modal.go:71-75, internal/tui/delete_modal.go:74-78, internal/tui/rename_modal.go:175-179
  DESCRIPTION: killModalKeyHint, deleteModalKeyHint, and renameModalKeyHint are each a one-line passthrough to the shared renderKeyHint, and none has a single non-test caller — the modal footer rows all route through renderConfirmCancelFooter instead. They survive only because their own unit tests still call them (modal_footer_test.go). They are leftovers from the consolidation that introduced modal_footer.go (whose own comment names them as the things it *replaced*). Keeping three identical dead wrappers (plus the tests pinning them) is low-value surface area that invites a future reader to think there are three distinct key-hint shapes when there is one.
  RECOMMENDATION: Delete the three wrappers and their dedicated tests; the shared renderKeyHint already has direct test coverage. If a per-modal seam is wanted for symmetry, keep exactly one and document it as the canonical example rather than three byte-identical copies.

SUMMARY: The MV reskin composes cleanly at the leaf layer — the shared renderJoinedPanel, renderKeyHint, newBandBase, destructiveConfirmSpec, the appearanceGate, the LoadingProgress accumulator, and the single Build/Deps construction chokepoint are all well-scoped, single-source, and DRY across the modal/notice/preview families. The main architectural soft spot is that the keymap descriptor's "single source of truth" framing covers only footer+help display while live key dispatch is an unlinked hand-coded switch (medium); the footer-source layer carries three parallel entry models plus one off-descriptor command-pending path (low); and three dead per-modal key-hint wrappers linger (low).
