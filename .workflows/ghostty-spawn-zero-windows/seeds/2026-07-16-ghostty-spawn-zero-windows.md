# Multi-window spawn opens zero windows on Ghostty

First live test of the picker's multi-select spawn on the real Mac (portal 0.9.1, Ghostty 1.3.1) failed completely: 3 sessions selected, Enter pressed, zero windows opened, and the notice band showed `'portal-EfVRkk', 'portal-agent-first-3' failed to open — others left open`. A retry a few seconds later failed identically. portal.log shows `spawn: opened 0/3` for both batches (17:37:28 and 17:37:33 on 2026-07-16), no `portal attach --spawn-ack` process ever starting, and no permission-required line.

A same-day analysis session diagnosed this (evidence in the session, to be validated by investigation): the AppleScript template in `internal/spawn/ghostty.go` uses `make new surface configuration with properties {…}` and `make new window with properties {configuration:…}`, but Ghostty 1.3.1's scripting dictionary (`Ghostty.sdef`) has no `make` command at all — `surface configuration` is a record-type, and windows are created via the custom `new window` command taking a `with configuration` parameter. The script fails to even compile (osascript error -2741, reproduced locally), so osascript exits non-zero instantly, the adapter maps it to SpawnFailed, and every external window fails in milliseconds. The sdef-correct form — `new window with configuration {command:"…", wait after command:true}` — was confirmed to compile against the installed Ghostty. The in-code claim that the template was "validated (Ghostty 1.3.1)" appears never to have been exercised: the `-tags manual` test `TestManual_OpenWindow_OpensRealGhosttyWindow` would have failed with exactly this error had it been run.

Two secondary defects surfaced during the same diagnosis and are in scope for this report:

1. **Failure detail is invisible at the default log level.** The per-window spawn failure detail (the actual osascript error text in `Result.Detail`) is emitted at DEBUG only (`internal/spawn/logemit.go`), so at the production-default INFO the log records that windows failed but never why — this diagnosis required reproducing osascript outside portal. The `spawn` log catalog is spec-governed, so surfacing failures at WARN is a spec amendment, not just a code tweak.

2. **Misleading banner copy on total failure.** The partial-failure message suffix "— others left open" (`internal/spawn/message.go`) is static and renders even when opened=0 and nothing was left open. The copy is golden-spec-governed and parity-tested across CLI and picker.

Impact: the multi-window spawn feature is entirely non-functional on the primary supported terminal (native Ghostty adapter) — every burst fails, nothing self-attaches. Verification of any fix must include actually running the manual-tag test plus a live end-to-end multi-select burst confirming `opened N/N`, since the absence of any live validation is what let this ship.
