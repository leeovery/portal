- FINDING: Compile-guard's pass/fail boundary is drawn around one historical error string (-2741), not around "the template compiled"
  SEVERITY: low
  FILES: internal/spawn/ghostty_compile_ghosttycompile_test.go:20, internal/spawn/ghostty_compile_ghosttycompile_test.go:113-134
  DESCRIPTION: Fix 4's stated purpose is a regression tripwire that "fails a drifted template WITHOUT a human in the loop" — i.e. it should catch the general class of "the emitted AppleScript no longer compiles against Ghostty's dictionary." As built, the guard only hard-fails when the compiler output contains the exact string "-2741" (the specific code the historical `make new surface configuration with properties` form yields); every other non-zero osacompile exit is routed to t.Skip as "environmental." The skip predicate is therefore drawn around "not exactly -2741" rather than around the intended "terminology could not be resolved." The consequence is a latent protection gap: a future template drift to a different-but-still-invalid form (a mistyped `new window`, a wrong property name, a different terminology error) would compile-fail with a different AppleEvent code and be silently skipped, giving false confidence that the tripwire is green. The guard as written protects only against re-introducing the one exact past bug, not the class of bug it is described as preventing. This is a genuine tension inside the test's own reasoning: reason #1 in the file's comment argues terminology resolves from the installed static sdef regardless of whether Ghostty is running — if that holds, an installed-Ghostty compile failure is essentially always a template problem and asserting a clean exit would be safe; reason #2 then hedges against reason #1 being wrong by skipping everything that is not -2741, which is what narrows the guard. The spec did sanction a skip-when-terminology-cannot-be-resolved fallback, so this is a low-severity soundness observation on where the fail/skip line sits, not a correctness bug — the guard passes today and the real functional proof remains the mandatory manual live validation.
  RECOMMENDATION: Draw the skip boundary around "terminology unresolvable" instead of "not -2741": pass on a clean exit, skip only on the signatures that mean osacompile could not reach the dictionary at all (osacompile missing from PATH / non-exit run error, or an app-not-found / "Can't get application" style resolution error), and fail on any other compile error against a resolved Ghostty dictionary. That keeps the false-failure protection the installed-but-not-running task added while restoring the general "did the template compile" coverage the guard advertises, so a future drift to a non--2741 broken form is still caught. Low priority; the current guard is correct for the exact historical regression and is a documented best-effort secondary net.
