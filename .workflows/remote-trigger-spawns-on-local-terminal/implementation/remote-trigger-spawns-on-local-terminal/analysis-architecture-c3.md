  DESCRIPTION: This change inverts how client_activity is used — from a tiebreak applied only among surviving host-local clients to the primary cross-client winner-selection signal across ALL clients (local and remote alike). The spawn-side docstring was correctly and fully rewritten to reflect that. But the producer of the value, tmux.ClientInfo (the type ListClients returns and the seam this change consumes through the ClientActivity mirror), still carries the old contract on its Activity field: "Activity is the local-only tiebreak used to choose among 2+ host-local clients." That statement is now false for the field's actual role across the codebase — activity now selects the trigger across remote clients too, which is the entire point of the fix. This is a seam-contract accuracy issue: the shared type's documented meaning contradicts its live usage. It matters because it is exactly the class of stale-contract drift the spec took explicit pains to eliminate on the spawn side (the "Owned Behaviour Change" section mandated rewriting the detect_inside.go contract so no reader is misled), yet the parallel doc on the value's origin was left describing the just-removed behaviour. A future maintainer reading ClientInfo.Activity would be told the field is a local-only tiebreak — the precise mental model that produced the original wrong-machine bug.
  RECOMMENDATION: Update the ClientInfo.Activity doc comment (internal/tmux/clients.go:12) to describe activity as the last-activity timestamp used to select the triggering (most-active) client across all attached clients, mirroring the corrected detect_inside.go contract, so the seam's producer and consumer describe the field identically.

The only architectural blemish is a stale contract comment on the shared tmux.ClientInfo.Activity field whose documented meaning now contradicts the inverted usage this change introduced.
