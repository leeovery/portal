# Intentional redesign of Portal's CLI verb surface

Portal's CLI grew by accretion — commands were added as they were needed, without a holistic design pass. The symptom that surfaced this: even the author is now fuzzy on the difference between `open` and `attach`. When the person who built it can't cleanly recall which verb does what, the surface has drifted past coherent.

The current shape: `portal open` (no args launches the TUI picker; one arg resolves a path/query through path → alias → zoxide → session and attaches in place), `portal attach <session>` (attaches in place to a named session via the connector), plus `x` (an alias), and the utility commands `hooks`, `clean`, `init`, `state`, `alias`, `version`. The `restore-host-terminal-windows` feature now adds a new operation — "open host terminal windows for N selected sessions" — provisionally named `portal spawn <sessions…>`.

The problem is overlapping, blurry verbs and input domains. `open` takes a path or query; `attach` takes a session name; the new command takes several session names and spawns windows. It's unclear to users — and to the author — which verb to reach for. Bolting `spawn` on in isolation just lengthens an organically-grown list without fixing the underlying incoherence.

The goal is to design the verb surface intentionally, in one coherent pass rather than command-by-command. That means reconciling `open`, `attach`, and the new window-spawn command; making the input domains legible (path/query vs single session-name vs multi-session); and deciding, deliberately, where to use distinct verbs versus overloading. A concrete live question: should the new operation be a distinct `spawn`, or a variadic `attach foo bar baz` where the argument count decides between attach-in-place (exec) and spawn-new-windows? The author likes the variadic-`attach` idea — it matches the session-name input domain — but it carries a count-dependent behaviour split. It should also settle where the picker sits in the mental model, and carry a compatibility/deprecation story (back-compat aliases), since these are existing user-facing commands people already have in muscle memory and scripts.

Origin: this surfaced while naming the new window-spawn command during the `restore-host-terminal-windows` discussion. That feature ships `portal spawn <sessions…>` provisionally so it isn't blocked — its picker path calls the spawn package in-process, so the CLI verb is a secondary surface and cheap to rename later. This redesign may revise or rename it. Best treated as its own feature (buildable: rename/restructure commands with back-compat aliases).
