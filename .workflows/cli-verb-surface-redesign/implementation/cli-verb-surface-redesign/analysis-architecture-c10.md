- FINDING: Degenerate single-surface burst round-trips Surface→QueryResult with a fabricated Domain that would be wrong if read
  SEVERITY: low
  FILES: cmd/open_burst.go:148, cmd/open_burst.go:154-163, cmd/open.go:360-378
  DESCRIPTION: The attach/mint dichotomy is represented three times across the open pipeline — cmd.Target+resolver.Domain (raw parsed argv, pre-resolution), resolver.QueryResult (SessionResult/PathResult carrying the true resolution provenance in .Domain, used for the resolve-decision log line), and spawn.Surface (SurfaceAttach/SurfaceMint, provenance stripped). resolveOpenSurfaces converts QueryResult→Surface (dropping the real domain — an alias/zoxide mint becomes a bare SurfaceMint), and then, for the single-surviving-surface degenerate case, dispatchOpenBurst converts back Surface→QueryResult via surfaceToResult so it can reuse openResolved's shared command-guard + ack-write + connector dispatch. surfaceToResult *fabricates* the domain: every mint surface is reconstructed as PathResult{Domain: DomainPath} regardless of whether the original hit was an alias or zoxide match, and every attach as SessionResult{Domain: DomainSession} (never DomainGlob). This is correct today only because openResolved reads the result's concrete TYPE (attach vs mint) but never its .Domain field, and the resolve-decision line was already emitted upstream in resolveOpenSurfaces — so the fabricated Domain is dead data on this path. It is a latent trap rather than a live bug: any future change that made openResolved consult r.Domain (e.g. to emit/re-emit a resolve line, drive a metric, or branch behaviour) would silently log/act on DomainPath for what was actually an alias or zoxide mint. The correctness of the fabrication depends on caller discipline (openResolved must keep ignoring Domain), which is exactly the kind of non-self-contained assumption that composes badly across future edits. The backward conversion is also mild unnecessary indirection: the value already flowed QueryResult→Surface moments earlier.
  RECOMMENDATION: Make the degenerate single-surface path self-contained rather than dependent on openResolved-never-reads-Domain. Cheapest option: have resolveOpenSurfaces (or dispatchOpenBurst) retain the original resolver.QueryResult for the single-surviving-surface case and pass that real result to openResolved, so no domain is fabricated. Alternatively, carry the true resolver.Domain on spawn.Surface so surfaceToResult reconstructs the accurate provenance; or, if the round-trip is kept, add a focused comment/assert at surfaceToResult stating the fabricated Domain is intentionally never consumed and must not be read (documenting the invariant the correctness rests on). No behavioural change is needed today — the goal is to remove the dormant wrong-value trap.
