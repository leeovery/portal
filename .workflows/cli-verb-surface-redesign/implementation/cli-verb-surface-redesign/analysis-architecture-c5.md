- FINDING: doctor's staleness diagnosis re-authors the classification the repair layer owns
  SEVERITY: medium
  FILES: cmd/doctor.go:451-477, cmd/doctor.go:432-444, cmd/doctor.go:399-427, internal/project/store.go:202-213, internal/hooks/store.go:265-271, cmd/run_hook_stale_cleanup.go:119-128
  DESCRIPTION: doctor's whole contract is "diagnose exactly what --fix repairs, then re-diagnose to confirm." But the read-only checks independently re-author the same staleness predicate the mutating repair uses, in parallel, for BOTH subsystems.
    - Projects: `checkStaleProjects` re-implements the exact `os.Stat` → {nil=live, ErrNotExist=stale, else=retained} switch that `project.Store.CleanStale` uses to select records for deletion. Two hand-maintained copies of the same predicate.
    - Hooks: `countStaleHookKeys` re-implements the `persisted ∉ live` membership test that `hooks.Store.CleanStale` uses, and `checkStaleHooks` re-implements the empty-live-set mass-deletion hazard guard that `runHookStaleCleanup` owns. The guard now lives in two places that must agree for doctor's "not-evaluable" to match --fix's "defer."
    This is the compose-don't-duplicate case the doctor design most needs to avoid: the count query is the read-only half of the prune, but it is defined independently rather than derived from the same predicate. If either store's definition of "stale" evolves (a broken-symlink project, a secondary hook-key form, a refined hazard guard), the diagnose layer silently diverges — doctor can report green while --fix prunes, or report N stale while --fix removes M≠N — breaking the "re-diagnosis confirms the repair" invariant. The predicates are currently consistent, so this is latent, not live.
  RECOMMENDATION: Have each store own a read-only staleness query (e.g. `project.Store.StaleProjects() ([]Project, error)` and a `hooks.Store` equivalent, hazard guard included), then define both the doctor check (count/report) and the prune (`CleanStale`) in terms of it. The count becomes `len(stale)` and the prune becomes "compute stale, then Save the complement" — one predicate, no drift surface.

- FINDING: Surface and QueryResult are parallel encodings of the attach/mint dichotomy with hand-maintained inverse converters
  SEVERITY: low
  FILES: cmd/open_surfaces.go:43-54, cmd/open_burst.go:154-163, cmd/open_burst.go:147-149, internal/spawn/surface.go:28-42, internal/resolver/query.go:41-71
  DESCRIPTION: The same Axiom-2 attach-vs-mint dichotomy is modelled twice: `resolver.QueryResult` (SessionResult|PathResult|MissResult) in the resolver layer and `spawn.Surface` (SurfaceAttach|SurfaceMint) in the spawn layer. resolveOpenSurfaces maps QueryResult→Surface (the `collect` closure); the single-surface degenerate burst path maps Surface→QueryResult (`surfaceToResult`, fabricating a synthetic Domain) purely to re-enter `openResolved`, which then switches back on the QueryResult type. So a lone-surface burst round-trips resolve→Surface→QueryResult→dispatch, and the two converters are hand-maintained inverses that must stay in lockstep. This is unnecessary indirection where the layers mirror each other; a mint/attach outcome that grew a third case would need updating in both type systems and both converters.
  RECOMMENDATION: Either let the degenerate single-surface case dispatch directly on `Surface.Kind` (a small attach-or-mint switch reusing openSessionFunc/openPathFunc + the command guard) instead of reconstructing a QueryResult, or converge the two vocabularies so one owns the transformation. Low urgency — it is contained to the N=1 burst tail — but it is the clearest mirrored-layer seam in the open path.

- FINDING: Target.Domain is stringly-typed and resolveOpenSurfaces' domain switch silently drops an unmodeled domain
  SEVERITY: low
  FILES: cmd/open_targets.go:9-35, cmd/open_surfaces.go:56-98, cmd/open_burst.go:76-83
  DESCRIPTION: `Target.Domain` is a bare string ("bare"/"session"/"path"/"alias"/"zoxide") while the parallel classification vocabularies (SurfaceKind, QueryResult) are typed. The domain set is a closed, design-time-known space, yet it is threaded as an untyped value and consumed by a switch in `resolveOpenSurfaces` that has NO default case (confirmed). A domain value outside the five modelled cases therefore produces no surface AND no miss — the target silently vanishes from the burst rather than opening or hard-failing. The guard test only covers pin↔flag coverage in `openTargetPins`; it does not enforce that every emittable domain has a resolveOpenSurfaces arm, so adding a future value-taking pin requires manual edits in openTargetPins, the pinDispatch table, a ResolveXPinAll variant, globExpandableDomain, and this switch, with the last unguarded and failing silently if missed.
  RECOMMENDATION: Make Domain a typed enum (mirroring SurfaceKind) so the switches are exhaustiveness-checkable, and add a `default:` arm to resolveOpenSurfaces that treats an unmodeled domain as a collected miss (or a hard error) rather than a silent drop. This closes the silent-vanish failure mode and ties the target-parsing seam to the type system instead of contributor discipline.
SUMMARY: The open/burst/resolver seams and the shared spawn log/classify layer compose cleanly, but doctor's diagnose layer independently re-authors the staleness predicates its repair layer owns (a real diagnose-vs-repair drift surface across hooks and projects), and the open path carries two minor structural smells — a mirrored Surface↔QueryResult round-trip and a stringly-typed, non-exhaustive domain switch that can silently drop a target.
