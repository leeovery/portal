- FINDING: `doctor` diagnose re-derives the "which entries are stale" classification the stores' `CleanStale` already owns
  SEVERITY: low
  FILES: cmd/doctor.go:465-491 (checkStaleProjects), cmd/doctor.go:443-458 (countStaleHookKeys), internal/project/store.go:191-235 (CleanStale, pre-existing), internal/hooks/store.go:249-296 (CleanStale, pre-existing)
  DESCRIPTION: `portal doctor` diagnoses stale entries and `portal doctor --fix` prunes them, and the two are meant to agree on the same subject — but each computes "what is stale" independently. `checkStaleProjects` re-implements, verbatim, the exact `os.Stat` three-way switch `project.Store.CleanStale` uses to select removals (`statErr == nil` → live; `errors.Is(statErr, os.ErrNotExist)` → stale; `default` → retained). `countStaleHookKeys` re-implements the `key ∉ liveSet` classification that `hooks.Store.CleanStale` uses. This is the parallel-computation-that-must-stay-in-sync pattern code-quality.md's "Compose, Don't Duplicate" warns against: because `--fix`'s prune runs through `CleanStale` (via `runHookStaleCleanup` / `deps.ProjectStore.CleanStale`) while the report runs through the doctor mirrors, any future change to the stale rule on one side (e.g. treating a permission-denied stat as stale, or normalising keys) would silently desync the diagnosis count from the set `--fix` actually prunes — inside a single command whose contract is that `--fix` repairs exactly what it diagnoses. The doctor mirrors are heavily documented as deliberate read-only copies, and each block is small (~10-12 lines) with simple stable logic, which is why this is low severity rather than higher.
  RECOMMENDATION: Have the diagnosis DERIVE the stale set from the same source the prune consumes rather than recomputing it. Cleanest for projects: add a read-only `project.Store.StaleRecords() ([]Project, error)` (or unexported classifier) that returns the would-remove set; `CleanStale` selects removals via it and `checkStaleProjects` counts `len()` of it — "which is stale" then lives in one place. Same shape for hooks: a read-only `hooks.Store.StaleKeys(liveKeys []string) []string` that both `CleanStale` and `countStaleHookKeys` consume (note the hooks path must keep doctor's empty-live-set hazard guard at the caller, not inside the shared classifier). If the store-method extraction is judged too much churn on pre-existing files for the payoff, leave as-is — the divergence risk is real but small; do not otherwise duplicate further.
SUMMARY: The multi-target open burst and picker burst are impressively single-sourced through internal/spawn (Burster, PartitionResults/FirstPermission, message renderers, log emitters, SplitTriggerFirst/SplitNetN), completion via completeSessionNames, and stale-hook cleanup via runHookStaleCleanup — no drift there. The one remaining duplication is doctor's read-only stale-classification for projects and hooks paralleling each store's CleanStale prune logic, a low-severity compose-don't-duplicate coupling between doctor's diagnose and its --fix repair.
