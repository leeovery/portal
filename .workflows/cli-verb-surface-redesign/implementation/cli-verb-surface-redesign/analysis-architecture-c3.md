AGENT: architecture
FINDINGS:
- FINDING: uninstall re-implements tmux "session gone" classification with a cmd-layer substring match, bypassing the typed-sentinel boundary the codebase mandates
  SEVERITY: medium
  FILES: cmd/uninstall.go:151-157, cmd/uninstall.go:135-149, internal/tmux/errors.go:22-35, internal/tmux/tmux.go:386-392
  DESCRIPTION: killSaver detects the idempotent "already gone" case via isSessionAbsentError, a raw strings.Contains(strings.ToLower(err.Error()), "can't find session") match at the cmd layer. internal/tmux/errors.go documents an explicit architectural invariant for exactly this classification: "Daemon-layer and other downstream callers MUST NOT perform substring matching against tmux stderr ... All such classification belongs here, behind a single sentinel, so a future tmux rephrasing requires a one-line change to the boundary discriminator and nothing else." The new uninstall command is precisely such a downstream caller and does exactly the prohibited thing. The reason the sentinel is not usable here exposes a second gap: KillSession returns its *CommandError un-wrapped (it never routes through wrapNoSuchSession the way ActivePaneCurrentPath / ShowEnvironment do), and the boundary discriminator matches only "no such session" while kill-session/has-session actually emit "can't find session" (confirmed against the tmux_test.go fixtures) — so tmux.ErrNoSuchSession does not even cover this command family's phrasing. Net effect: correctness now depends on a stderr string the cmd layer owns; a future tmux rephrase silently flips the idempotent no-op into a spuriously accumulated error. Blast radius is small (killSaver is HasSession-gated, the race window is tiny, and the completion message still prints), but it is the wrong pattern planted in new code, in direct contradiction of the codebase's own documented rule.
  RECOMMENDATION: Move the classification back to the internal/tmux boundary: route KillSession's error through wrapNoSuchSession and broaden the discriminator to also recognise the kill-session/has-session "can't find session" phrasing behind ErrNoSuchSession (or a sibling sentinel). Then have killSaver consume errors.Is(err, tmux.ErrNoSuchSession) instead of isSessionAbsentError, restoring the single-home invariant and deleting the fragile cmd-layer substring.

- FINDING: Multi-target routing is driven by a second raw-os.Args parser layered on cobra, with an unguarded "no global value-flag before the subcommand" correctness dependency
  SEVERITY: low
  FILES: cmd/open_burst.go:27-51, cmd/open_targets.go:27-92, cmd/open.go:207-210
  DESCRIPTION: The burst-vs-single routing gate does not read cobra's validated parse; it re-reads raw os.Args (openOwnArgs) and re-classifies each token against openTargetPins (orderedOpenTargets), a hand-maintained mirror of openCmd's live flag set. This dual-parser design is spec-mandated — cobra collapses interleaved order and repeated pins that the burst needs — and the flag-map↔cobra drift is guard-tested (TestOpenTargetPinsCoverValueTakingFlags). But two correctness dependencies are not guarded: (a) openOwnArgs assumes no value-taking global/persistent flag precedes the literal "open" token — a fragility the source itself flags "for review"; introducing such a flag later would let the scan misattribute the flag's value as a bare positional and misroute a single-target invocation into a burst; (b) routing hinges on os.Args agreeing with cobra's arity model for every token. This is the largest seam-drift surface the feature introduces; today it is correct only because no such global value-flag exists.
  RECOMMENDATION: Keep the raw scan (it is the right tool for order recovery), but harden the unguarded assumption: add a guard test asserting rootCmd carries no value-taking persistent/global flag (so introducing one fails loudly), or anchor the scan's start offset to cobra's known command path rather than the first literal "open" token, so a value that happens to equal "open" or a future pre-subcommand flag cannot shift the parse.

- FINDING: The attach/mint dichotomy is modelled in three parallel representations with a lossy backward conversion and two mirrored dispatch switches
  SEVERITY: low
  FILES: cmd/open_burst.go:147-156, cmd/open_surfaces.go:43-54, cmd/open.go:322-340, cmd/open_burst_run.go:253-258
  DESCRIPTION: The same attach-vs-mint decision is carried by three types — Target.Domain (string), resolver.QueryResult (SessionResult|PathResult), and spawn.Surface (SurfaceAttach|SurfaceMint) — with a forward projection (collect: QueryResult→Surface) and a backward re-inflation (surfaceToResult: Surface→QueryResult), plus two dispatch switches on the identical dichotomy (openResolved switches on QueryResult concrete type; connectTrigger switches on SurfaceKind). surfaceToResult hardcodes Domain:"path"/Domain:"session" irrespective of the surface's true resolving domain (alias/zoxide/glob), so the reconstructed QueryResult carries a Domain that is a placeholder, not the real provenance. It is harmless today — the single-surviving-surface path routes through openResolved, which ignores Domain and emits no further resolve line — but it is a latent trap: a type annotation that is correct only inside its current narrow use, which would produce a wrong domain the moment any future consumer of that degenerate path reads it (e.g. to log or re-emit a decision line).
  RECOMMENDATION: Low priority. If this area is touched, prefer connecting the single-surviving-surface case directly off spawn.Surface (reusing connectTrigger's SurfaceKind dispatch together with the command-on-attach guard and ack write) rather than round-tripping back into a synthetic QueryResult, so the attach/mint decision has one dispatch home and Surface never needs a lossy re-inflation.
SUMMARY: The resolve→surfaces→burst→connect flow composes cleanly overall — shared productionSpawnSeams and the spawn classify/message/split primitives keep the picker and open-burst paths from drifting, and the doctor diagnose/repair down-server safety is well-factored. The notable structural weakness is uninstall's cmd-layer substring classification of tmux stderr, which violates the codebase's own documented tmux-error-boundary invariant; two lower-severity seams (the raw-argv routing parser's unguarded assumption, and the three-way attach/mint modelling with a lossy back-conversion) are latent fragilities rather than active bugs.
