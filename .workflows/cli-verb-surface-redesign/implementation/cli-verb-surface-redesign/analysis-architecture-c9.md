- FINDING: Doctor diagnosis reimplements the stores' stale-entry predicate instead of deriving from it
  SEVERITY: medium
  FILES: cmd/doctor.go:415-460, cmd/doctor.go:467-493, internal/hooks/store.go:249-271, internal/project/store.go:191-213
  DESCRIPTION: The stale-entry classification exists in two independent copies for each store. For hooks, `hooks.Store.CleanStale` computes "removed = persisted key ∉ liveSet" (store.go:262-271) while `cmd/doctor.go`'s `checkStaleHooks`/`countStaleHookKeys` build their own liveSet and re-derive the same ∉ count (doctor.go:448-460). For projects, `project.Store.CleanStale` runs a three-way `os.Stat` switch (nil→keep, ErrNotExist→remove, else→keep) at store.go:202-213, and `checkStaleProjects` re-implements the identical switch read-only at doctor.go:477-488 (its own comment says "It mirrors project.Store.CleanStale's os.Stat classification"). The mutation path (daemon automation + `--fix`) routes through the stores' CleanStale, but the read-only diagnosis routes through the reimplemented copies. This is a diagnose-vs-repair split where the "is it stale?" query is authored independently from the "remove the stale ones" repair. If the staleness rule ever changes in a store (a new key-matching semantics, different handling of a symlinked/broken dir), doctor's report silently drifts from what `--fix` actually prunes — a latent correctness inconsistency in the one tool whose job is to accurately report health. Nothing forces the two to agree today.
  RECOMMENDATION: Extract the staleness predicate as a pure method on each store — e.g. `hooks.Store.StaleKeys(live []string) []string` and `project.Store.StaleEntries() ([]Project, error)` — and define `CleanStale` in terms of it ("remove StaleKeys/StaleEntries"). Have doctor's `checkStaleHooks`/`checkStaleProjects` count via that same classifier read-only, deleting `countStaleHookKeys` and the duplicated `os.Stat` switch. Diagnosis and repair then share one predicate and cannot drift. The mass-deletion hazard guard (empty-live-set deferral) stays where it is in doctor/runHookStaleCleanup — only the ∉ / os.Stat classification is unified.
- FINDING: Single-use tuiConfig intermediate mirrors tui.Deps 1:1, with an inaccurate shared-harness doc claim and no parity guard
  SEVERITY: low
  FILES: cmd/open.go:639-741
  DESCRIPTION: `tuiConfig` (cmd/open.go:640) duplicates the ~35-field `tui.Deps` seam struct, and `buildTUIModel` (cmd/open.go:708) copies it field-for-field into `tui.Deps` with zero transformation. Its doc comment states buildTUIModel is "the single model-construction chokepoint shared with the offline capture harness (cmd/capturetool)", but the capture harness does not use buildTUIModel or tuiConfig at all — `internal/capture/fixtures.go` builds its own `tui.Deps` and calls `tui.Build` directly (the true shared chokepoint is `tui.Build`, not buildTUIModel). So tuiConfig's only consumers are openTUI and cmd-package tests. That leaves a second 35-field struct that must be kept in lockstep with tui.Deps by hand, with no compiler help and no parity guard (unlike `openTargetPins`, which has TestOpenTargetPinsCoverValueTakingFlags): a field added to tui.Deps that openTUI must populate silently maps to the zero value unless a maintainer also threads it through tuiConfig and the buildTUIModel copy. The misleading doc compounds the risk by implying a harness-parity contract that isn't there.
  RECOMMENDATION: Either drop tuiConfig and have openTUI (and the tests that use it) populate `tui.Deps` directly, or, if the cmd-local shape is kept for test ergonomics, correct the doc comment to state the actual chokepoint is `tui.Build` (shared with capture via `capture.Fixture.Deps()`), and add a field-parity guard test so tuiConfig↔tui.Deps drift fails loudly.
- FINDING: Open-burst detector construction bypasses the "single source" spawn-seams bundle
  SEVERITY: low
  FILES: cmd/spawn_seams.go:27-68, cmd/open_burst_run.go:88-89
  DESCRIPTION: `buildProductionSpawnSeams` documents itself (spawn_seams.go:27-33) as the single construction site the open burst and picker both read "so their production wiring cannot drift", and it carries a `Detector` field. The picker (openTUI) and doctor both take their detector from that bundle. The open burst's `buildOpenBurstDeps`, however, defaults `Detector` via a separate `spawnDetector(cmd)` path (open_burst_run.go:88-89) rather than `sharedSeams().Detector`, and on the default path also builds the bundle for the other seams — so two `spawn.NewDetector(client)` instances are constructed, one of which (the bundle's) is discarded. Both paths wrap the same client, so behaviour is identical today, but the detector specifically is NOT single-sourced despite the bundle's explicit anti-drift claim, leaving a construction site the compiler can't reconcile the way the bundle intends.
  RECOMMENDATION: Default `OpenBurstDeps.Detector` from `sharedSeams().Detector` (it satisfies the `TerminalDetector` interface) so all three consumers read the one bundle field, and drop the separate `spawnDetector` construction for the burst — or, if a standalone detector default is deliberately kept to avoid building the whole bundle, narrow the bundle doc so it no longer claims the detector is single-sourced across the burst.
SUMMARY: The redesigned command surface composes cleanly — pins, globs, burst mechanics, retired-verb removal, and the shared spawn seams are well-factored, well-guarded, and match the spec. The one structural concern worth fixing is that doctor's stale-entry diagnosis reimplements the stores' own staleness predicate instead of deriving from it, so diagnose and repair can silently drift; two low-severity seam/indirection nits round out the pass.
