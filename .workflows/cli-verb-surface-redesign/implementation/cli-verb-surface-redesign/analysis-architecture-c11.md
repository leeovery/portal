- FINDING: The identity→adapter resolution seam is an unnamed function type repeated across 8 boundary sites
  SEVERITY: low
  FILES: cmd/open.go:662, cmd/open_burst_run.go:33, cmd/doctor.go:129, cmd/doctor.go:404, cmd/spawn_seams.go:36, internal/tui/build.go:54, internal/tui/spawn_detect.go:42, internal/tui/model.go:465
  DESCRIPTION: The host-terminal resolution seam — `func(spawn.Identity) (spawn.Adapter, spawn.Resolution)` — is the central contract that lets the picker, the multi-target open burst, and doctor all share one resolver. It is spelled out inline at eight distinct declaration sites spanning three packages (cmd, tui, spawn-adjacent), yet is never given a name. This contradicts the spawn package's own established convention: it already names the far-less-used single-return seam `ExecutableResolver func() (string, error)` (internal/spawn/command.go:6). The result is that the feature's most widely-consumed boundary contract is implicit and un-discoverable in the spawn API, and any future change to the signature (e.g. adding an error return, or wrapping the pair in a struct) must be hand-edited at all eight sites in lockstep with no compiler anchor. This is an API-surface consistency and maintainability seam, not a behavioural defect.
  RECOMMENDATION: Declare a named type in internal/spawn (e.g. `type AdapterResolver func(Identity) (Adapter, Resolution)`) alongside ExecutableResolver, and reference it at all eight sites (struct fields, the WithResolve option, and checkHostTerminal's parameter). This localises the boundary contract to one definition, makes the resolution seam a first-class part of the spawn API, and matches the package's existing named-func-type precedent.

- FINDING: The open burst constructs its host-terminal detector via a second construction path, bypassing the anti-drift seam bundle
  SEVERITY: low
  FILES: cmd/open_burst_run.go:88-90, cmd/spawn_seams.go:34-68
  DESCRIPTION: productionSpawnSeams exists explicitly to be "the single construction site the open burst and picker both read, so their production wiring cannot drift" (cmd/spawn_seams.go:44-50). Two of its three consumers honour this: openTUI (cmd/open.go:877) and resolveDoctorDeps (cmd/doctor.go:156) both read `buildProductionSpawnSeams(client).Detector`. The third — buildOpenBurstDeps — defaults every other shared seam (Resolve/Ack/Exe/Getenv/Logger) from the bundle via `sharedSeams()`, but defaults Detector from a separate one-off helper `spawnDetector(cmd)` (cmd/open_burst_run.go:89) instead of `sharedSeams().Detector`. Both paths currently call `spawn.NewDetector(client)` with the same client, so there is no behavioural drift today — but the Detector seam is the one seam of the seven for which the bundle's stated "cannot drift" invariant is not actually enforced. A future change to how the detector is constructed inside the bundle would silently not reach the open burst, which is exactly the failure mode the bundle was created to prevent.
  RECOMMENDATION: Default the open burst's Detector from `sharedSeams().Detector` like every other shared seam, and delete the now-redundant `spawnDetector` helper. If avoiding an eager terminals.json load when only the detector needs defaulting is the reason for the split, document that rationale on both the field and the helper; otherwise the single-source bundle should own all seven seams uniformly.

- FINDING: Picker-launch prediction is re-derived in bootstrap routing separately from the actual dispatch decision
  SEVERITY: low
  FILES: cmd/root.go:302-304 (isTUIPath), cmd/open.go:178-284 (openCmd.RunE)
  DESCRIPTION: Whether `portal open` launches the TUI is decided authoritatively inside openCmd.RunE (the -f branch, the no-target early return, and the -e-with-no-target case all reach openTUIFunc). PersistentPreRunE must predict that same outcome up front — via isTUIPath — to choose synchronous vs concurrent/deferred bootstrap, and on the latch-not-satisfied path the prediction is load-bearing: if isTUIPath said "TUI" but RunE went direct, the deferred bootstrap (owned by openTUI's goroutine) would never run. isTUIPath re-derives the picker-launch condition as `open && len(args)==0 && !anyOpenDomainPin`, which today agrees with RunE for every case. The pin vector is well-guarded (anyOpenDomainPin consults the shared openDomainPinFlags list, so a new pin is covered automatically), but the -f/-e picker-launch cases are matched only implicitly through the args==0 clause — so a future picker-launching flag that coexists with a positional would desynchronise the two sites. The coupling is documented and covered by concurrent_*_test.go, so this is a latent self-containedness seam rather than an active bug.
  RECOMMENDATION: Keep the single shared openDomainPinFlags anchor, and consider expressing the "does this invocation launch the picker" predicate once (a small helper both PersistentPreRunE and RunE consult) so the -f/-e picker-launch conditions are named rather than inferred from args==0. At minimum, keep the cross-reference comments in lockstep so any future picker-launching flag is added to both sites deliberately.
