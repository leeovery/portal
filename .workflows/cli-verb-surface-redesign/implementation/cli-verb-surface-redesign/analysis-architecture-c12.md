- FINDING: Resolver single/multi method families compose inconsistently; the "collected-miss" contract is split across two layers
  SEVERITY: medium
  FILES: internal/resolver/query.go:219, internal/resolver/query.go:239, internal/resolver/query.go:265, internal/resolver/query.go:341, internal/resolver/query.go:386, cmd/open_surfaces.go:65
  DESCRIPTION: The QueryResolver has grown two parallel method families — single-result resolvers (Resolve / ResolveSessionPin / ResolveAliasPin / ResolvePathPin / ResolveZoxidePin, which hard-fail on a miss) and "…All" resolvers (ResolveBareAll / ResolveSessionPinAll / ResolveAliasPinAll, which collect the miss as a *MissResult for the atomic burst pre-flight). Their composition pattern is not uniform, and the boundary that owns the error→collected-miss translation is fuzzy:
    (1) ResolveBareAll composes cleanly over Resolve (call it, convert any error to a MissResult) — the right shape.
    (2) ResolveSessionPinAll and ResolveAliasPinAll do NOT compose over their single-pin siblings; they re-author the non-glob domain logic. ResolveAliasPinAll re-does Get + validatedPath and hand-maps both the unknown-key and gone-dir errors to a MissResult, duplicating exactly what ResolveAliasPin already does (the same drift class ResolveBareAll avoids). Two independent implementations of the same alias-domain rule can silently diverge if the single-pin validation ever changes.
    (3) The path and zoxide domains have no *All variant at all; their error→collected-miss translation lives in cmd/open_surfaces.go (resolveOpenSurfaces manually appends t.Value to misses for -p/-z). So the "collected miss" contract is implemented in the resolver for glob-capable domains but in the cmd layer for the two non-glob mint domains — the same responsibility owned by two different layers depending on which domain is being resolved.
  This is correct today (well test-guarded) but it is an API-surface smell: the resolver exposes a lopsided set of methods, callers must know which domains return collected misses vs which they must collect themselves, and the alias logic exists twice.
  RECOMMENDATION: Make the multi-target/collected-miss translation a uniform, resolver-owned concern that composes over the single-result resolvers. Have ResolveSessionPinAll / ResolveAliasPinAll wrap their single-pin siblings and convert any error to a *MissResult (mirroring ResolveBareAll ⇄ Resolve), so each domain rule has exactly one implementation. Then give path/zoxide the same collected-miss entry point (a trivial single-element wrapper) so resolveOpenSurfaces makes one uniform per-domain call and the "collect the miss" logic no longer straddles the resolver↔cmd boundary. Net effect: one authoritative implementation per domain, and the collected-miss contract lives entirely in the resolver.

- FINDING: Multi-target routing depends on a hand-rolled argv rescan whose subcommand-locating heuristic is invocation-shape-fragile
  SEVERITY: low
  FILES: cmd/open_burst_run.go:43, cmd/open_targets.go:41, cmd/open.go:228
  DESCRIPTION: The burst gate recovers ordered targets from a raw os.Args rescan layered on top of cobra (spec-mandated, since cobra loses interleaved order). Two seams here rest on caller-shape assumptions rather than being self-contained: (a) openOwnArgs finds "the first token equal to 'open' after argv[0]" to locate the subcommand's args — self-flagged in-code as a "documented, pragmatic — flagged for review" assumption that no value-taking global/persistent flag precedes the open token; (b) openTargetPins is a hand-maintained mirror of cobra's value-taking flag set (both short and long forms, plus the excluded flags) so the rescan knows each flag's arity. If a value-taking persistent flag is ever added before the subcommand, openOwnArgs would misattribute the subcommand boundary; if a new value-taking flag is added to openCmd without a matching openTargetPins entry, its value is misrouted as a bare positional target. Both are currently safe: no persistent value-flags exist, openOwnArgs degrades to nil (single-target path) rather than a wrong open, and open_targets_guard_test.go pins the flag mirror at test time. The risk is latent, not active — flagged because the correctness of a headline path (single-vs-multi routing) leans on invocation shape + a parallel flag registry rather than on cobra alone.
  RECOMMENDATION: No structural change required now given the guard test and safe degradation; keep it on the radar. If Portal ever gains a value-taking persistent/global flag, harden openOwnArgs to skip known global flags (or key off cobra's already-matched command rather than a literal "open" scan) so the boundary is derived, not assumed.
SUMMARY: Architecture composes well overall — the spawn service is cleanly single-sourced across the picker and open-burst callers, seams are richly typed and test-guarded, and attach/spawn/clean/status retirement is clean. The one substantive structural issue is the resolver's lopsided single-vs-All method surface, where the collected-miss contract is duplicated (alias) and split across the resolver↔cmd boundary (path/zoxide) instead of composing each All-path over its single-result sibling. A secondary low note flags the argv-rescan's invocation-shape assumptions, which are currently safe and guarded.
