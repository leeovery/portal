AGENT: architecture
FINDINGS:
- FINDING: `doctor`'s read-only stale-checks re-derive the stores' staleness classification instead of composing a shared read-only primitive
  SEVERITY: medium
  FILES: cmd/doctor.go:399-444 (checkStaleHooks + countStaleHookKeys), cmd/doctor.go:451-477 (checkStaleProjects), internal/hooks/store.go:249-298 (Store.CleanStale), internal/project/store.go:191-237 (Store.CleanStale), cmd/run_hook_stale_cleanup.go:90-141 (hazard guard)
  DESCRIPTION: `portal doctor` is a new surface whose whole value proposition is that its read-only diagnosis accurately predicts what `--fix` (and the daemon's automation) will prune — the exit code is a scriptable gate on exactly that. But the "how many are stale?" query and the "delete the stale ones" mutation each author the staleness definition independently:
    * Hooks: `countStaleHookKeys` reimplements the persisted-key ∉ live-set difference that `hooks.Store.CleanStale` computes internally, and `checkStaleHooks` also reimplements the empty-live-set hazard guard that `runHookStaleCleanup` already owns — so the hooks staleness rule + guard now live in three places (Store.CleanStale, runHookStaleCleanup, doctor.checkStaleHooks).
    * Projects: `checkStaleProjects` copies `project.Store.CleanStale`'s exact three-way os.Stat switch, including the subtle "permission-denied → retained (NOT stale)" default branch.
  This is the "compose, don't duplicate / logical-inverse" case from the architecture focus: the read-only count is the companion of the mutating prune and should be derived from one authority. As written, a change to either store's classification (e.g. a key-normalisation step, or treating an errored stat differently) silently diverges the diagnosis from the repair — `doctor` reports "no stale" while `--fix`/daemon actually prune, or vice versa. The duplication is acknowledged in the comments ("strictly READ-ONLY mirror", "mirrors ... CleanStale's os.Stat classification"), which is precisely the drift smell. It compounds a pre-existing footgun: `hooks.Store.CleanStale` deletes every persisted entry when handed an empty live set, so its safety depends on each caller re-implementing the guard — and this feature added a second caller (doctor) that had to do exactly that.
  RECOMMENDATION: Give each store a single read-only stale-set primitive that both CleanStale and the doctor check derive from — e.g. `project.Store.StaleProjects() ([]Project, error)` and `hooks.Store.StaleKeys(live []string) ([]string, error)` (the latter baking in the empty-live hazard guard so it is self-contained rather than caller-discipline). Then `CleanStale` becomes "compute the stale set, Save(kept)", the doctor check becomes "len(stale-set)", and `runHookStaleCleanup`'s guard collapses into the primitive. One authority for "what is stale", consumed read-only by diagnosis and destructively by repair, so the two can never disagree.
SUMMARY: The redesigned command surface composes cleanly overall — shared spawn Burster/classify/message primitives, a single emitResolveDecision source, and well-scoped DI seams. The one genuine architectural concern is the new `doctor` command independently re-deriving the stores' staleness classification (hooks and projects) rather than deriving its read-only diagnosis from a shared primitive the mutating `--fix`/daemon repairs also use, leaving the health verdict able to drift from the repair it is meant to predict.
