- FINDING: Domain routing is a stringly-typed enum spanning three overlapping vocabularies across the cmd↔resolver boundary
  SEVERITY: medium
  FILES: cmd/open_targets.go:9-41, cmd/open_burst.go:76-83, cmd/open_surfaces.go:56-97, internal/resolver/query.go:45-71
  DESCRIPTION: The open resolution/routing pipeline discriminates its target domains with bare strings, and there are three distinct-but-overlapping closed vocabularies that must stay hand-aligned: (1) cobra flag names ("session"/"path"/"alias"/"zoxide", the openDomainPinFlags slice + pinResolvers keys), (2) Target.Domain ("bare"/"session"/"path"/"zoxide"/"alias", produced by the openTargetPins map values), and (3) the resolver result Domain fields ("session"/"path"/"alias"/"zoxide"/"glob"/"miss"). The concrete set is fully known at design time, yet it is routed by string-literal switches — globExpandableDomain(domain string) switches on "bare"/"session"/"alias"; resolveOpenSurfaces switches on t.Domain literals. This is exactly the "untyped parameters when concrete types are known" / "runtime string checks where the signature should be a specific type" anti-pattern in code-quality.md. Alignment is currently enforced only by convention plus guard tests (open_targets_guard_test, open_pin_source_guard_test) — i.e. at test time, not compile time. A typo in one switch arm, or a new domain added to one vocabulary but not another, is a silent misroute unless a guard test happens to cover it.
  RECOMMENDATION: Introduce a single typed Domain constant set (e.g. `type Domain string` or an int enum with a String()) shared by cmd and resolver, so Target.Domain, the resolver result Domain, and the switches all key off the same typed values and the switches become exhaustive-checkable. The resolver's spec-coupled log-attr value (session/path/alias/zoxide/miss) can be produced by the enum's String(), preserving the log line unchanged. This collapses vocabularies (2) and (3) and removes the drift class the guard tests currently paper over.

- FINDING: The resolver's *All family is a partial, independently-authored parallel to the single-pin methods rather than a composition of them
  SEVERITY: low
  FILES: internal/resolver/query.go:132-278, cmd/open_surfaces.go:56-97
  DESCRIPTION: The resolver exposes single-result pins (ResolveSessionPin/ResolveAliasPin/ResolveZoxidePin/ResolvePathPin) AND "*All" burst variants (ResolveBareAll/ResolveSessionPinAll/ResolveAliasPinAll) whose only real difference is miss policy (hard-error vs collected *MissResult). The *All variants re-author the domain match rather than deriving from the single pins — the exact-session match (ListSessionNames + slices.Contains) is written three separate times (Resolve, ResolveSessionPin, ResolveSessionPinAll), and with subtly inconsistent error handling: Resolve gates on `err == nil && slices.Contains(...)` while both pins ignore the lister error entirely (`names, _`). The family is also incomplete: session/alias/bare have resolver-side *All variants, but path/zoxide do not, so resolveOpenSurfaces must interleave two different miss-collection idioms in one switch — some domains return misses in-band, path/zoxide translate a single-pin error into a miss at the cmd layer. The result is a large, asymmetric resolver surface (8 public resolve methods, 3 with *All siblings) that a reader must hold two mental models for.
  RECOMMENDATION: Extract a shared exact-session-match helper so the three match sites (and their error handling) cannot diverge, and derive the single pins from the collected-miss form where practical (single = All + "if the sole result is a miss, promote to the hard error"). Alternatively, make the miss-collection uniform: give every domain the same "*All returns results, only ErrZoxideNotInstalled propagates" contract, or push the error→miss translation entirely into one generic adapter at the burst layer so resolveOpenSurfaces reads one idiom for all five domains.

- FINDING: The "attach-or-mint" outcome is modeled by two isomorphic sum types with bidirectional converters at the resolver↔spawn seam
  SEVERITY: low
  FILES: internal/resolver/query.go:45-71, internal/spawn/surface.go:1-42, cmd/open_surfaces.go:43-53, cmd/open_burst.go:154-163
  DESCRIPTION: The same domain concept — "a resolved target is either an attach to a named session or a mint at a literal directory" — is represented twice: resolver.QueryResult (SessionResult|PathResult|MissResult) and spawn.Surface (SurfaceAttach|SurfaceMint). resolveOpenSurfaces.collect() maps QueryResult→Surface, and dispatchOpenBurst.surfaceToResult() maps Surface→QueryResult so the single-surviving-surface degenerate case can reuse openResolved's QueryResult switch. In that path a target is resolved to a QueryResult, converted to a Surface, then converted back to a QueryResult — a round trip through two parallel representations of the same two-way choice. A cross-package boundary justifies some mapping, but a bidirectional round trip signals the seam owns two overlapping models where one shared shape (or a one-directional adapter) would do.
  RECOMMENDATION: Pick one representation as the canonical outcome across the resolve→dispatch path. Either have openResolved accept a spawn.Surface directly (so the degenerate case needs no back-conversion), or have the burst consume QueryResult and drop spawn.Surface's attach/mint duplication — keeping a single sum type for "attach-or-mint" and at most one converter at the true package boundary.

- FINDING: doctor reconstructs the spawn detector/resolve seams by hand, opting out of the shared anti-drift bundle it was built to prevent
  SEVERITY: low
  FILES: cmd/doctor.go:131-158, cmd/spawn_seams.go:27-61
  DESCRIPTION: productionSpawnSeams (buildProductionSpawnSeams) exists specifically so the open-burst and picker paths cannot silently diverge on how the host-terminal Detector + Resolve seams are constructed — the comment states the compiler cannot catch a seam added/swapped on only one side, so the bundle is the single source both read. resolveDoctorDeps deliberately does NOT use the bundle; it rebuilds spawn.NewDetector(client) and a buildResolver().Resolve closure by hand, becoming a third construction site the comments themselves say "must be kept in sync by hand." The stated justification is a lazy terminals.json read (buildResolver only reads the file when a non-null identity computes the line), but doctor invokes Resolve at most once per run and already reads hooks.json/projects.json/sessions.json — so the saving is at most one file read on a NULL-identity run, traded against reintroducing the exact by-hand sync obligation the bundle abolished. doctor's host-terminal line is informational-only, so the blast radius is small, but the opt-out weakens the bundle's "single source" guarantee.
  RECOMMENDATION: Either route doctor's Detector/Resolve through productionSpawnSeams (accepting one eager terminals.json read on a bootstrap-exempt, already-file-heavy command), or make the bundle itself lazy so all three consumers share one construction site without the eager read — rather than maintaining a hand-synced third copy.
SUMMARY: The open/burst/spawn/doctor composition is well-factored overall — shared bundles, single-sourced log/message renderers, and careful burst error paths — but the target-domain routing leans on three hand-aligned string vocabularies where a typed enum belongs, the resolver's single-pin vs *All split is an asymmetric parallel implementation rather than a composition, the attach-or-mint outcome is modeled twice with round-trip converters, and doctor hand-rebuilds the spawn seams the anti-drift bundle exists to centralize.
