Cycle 4 already assessed the exact cycle-5 recommendation ("have each store own a read-only staleness query, define both the doctor check and the prune in terms of it") and rejected it: the projects half is cleanly extractable, but the hooks half's empty-live-set hazard-guard is genuinely caller-specific — `runHookStaleCleanup`/daemon must defer-and-warn (never delete) while `doctor.checkStaleHooks` must report `checkNotEvaluable` (never "all stale"). Medium, not high, so the never-discard-high rule does not apply. Discarded, unchanged.
