- FINDING: doctor's daemon check drags the whole over-scoped CollectStatus, discarding most of it and double-reading the index
  SEVERITY: medium
  FILES: cmd/doctor.go:492-509, cmd/doctor.go:500, internal/state/status.go:81-90, internal/state/status.go:142, internal/state/status.go:185, cmd/doctor.go:613-643
  DESCRIPTION: After `state status` was deleted, `state.CollectStatus` was retained wholesale as doctor's daemon-liveness probe, but `checkDaemonAlive` reads only three of its ~ten fields (DaemonRunning / DaemonPID / DaemonVersion). CollectStatus still eagerly performs a full state-dir tree walk (computeStateSize), a full portal.log line-by-line scan (scanRecentWarnings), and a sessions.json parse (collectIndexState) on every `portal doctor` invocation — all discarded. StateSize / RecentWarnings / LastWarning / LastSaveAt / HasLastSave / SessionsCount / PanesCount and their helpers are now dead weight in production (no non-test consumer). Worse, the sessions/panes counts are computed twice per run: once inside CollectStatus (thrown away) and again by doctor's own `checkSessionsJSON` → `state.ReadIndex` + `doctorPaneCount`, the latter re-implementing the identical triple-nested pane loop that `state.countPanes` already owns (unexported, so it couldn't be reused). This is a heavy, redundant I/O cost baked into a command explicitly designed as a fast scriptable health gate (`portal doctor && …`), plus a mirrored-computation maintenance hazard.
  RECOMMENDATION: Give doctor a narrow daemon-liveness probe (pid file read + IsProcessAlive + version), and either trim CollectStatus down to that or delete it if doctor is its only remaining caller. Have doctor source the sessions/panes counts from one index read (export/share `countPanes` instead of the parallel `doctorPaneCount`). Drop the now-unconsumed StateSize/RecentWarnings/LastWarning machinery so a routine `doctor` stops walking the state tree and scanning the whole log.

- FINDING: single-vs-multi routing runs a raw os.Args re-parse parallel to cobra, leaving a production-dead-but-divergent glob branch in the resolver
  SEVERITY: medium
  FILES: cmd/open_burst_run.go:43-70, cmd/open.go:207-210, cmd/open.go:262, internal/resolver/query.go:134-141, internal/resolver/query.go:184-224
  DESCRIPTION: Whether an `open` invocation is single-target or a burst is decided by `isMultiTarget(orderedOpenTargets(openOwnArgs()))`, a raw `os.Args` re-scan that runs in parallel to cobra's own parse and hinges on a documented assumption (the "open" subcommand token is present and no value-taking global flag precedes it). Because a single glob-bearing bare/session/alias target is classified multi and diverted to the burst, `QueryResolver.Resolve`'s glob branch (query.go:134-141) is unreachable in production — its two callers (open.go:262 and ResolveBareAll at query.go:219) only ever pass non-glob input. But that branch is not a harmless no-op: it independently re-implements session-glob expansion (duplicating `expandSessionGlobAll`/`ResolveBareAll`) and returns only `matches[0]` — the FIRST match — whereas the live burst path opens ALL matches. So the moment the os.Args assumption breaks (a future persistent value-flag, an invocation shape that omits the "open" token, or the fallback returning nil), a single glob silently degrades from "burst every matching session" to "attach only the first" — a semantic fork with no error, no test signal, masked entirely by the parallel parse. Correctness of the glob fan-out rests on caller-side argv discipline rather than being self-contained.
  RECOMMENDATION: Make the routing self-contained — derive the ordered target set from cobra's parsed state (or make the multi/single decision the single authority the resolver also obeys) so there is one parse, not two that can disagree. Then either remove the dead glob branch from `Resolve` (letting a single glob route through the same `expandSessionGlobAll` the burst uses) or make it hard-fail rather than silently returning a first-match, so a routing miss can never quietly change glob semantics.

- FINDING: open-target domains are a bare-string vocabulary spread across four sites, and the burst classify switch silently drops an unknown domain
  SEVERITY: low
  FILES: cmd/open_surfaces.go:56-98, cmd/open_targets.go:27-35, cmd/open.go:236-244, cmd/open_burst.go:76-83, cmd/open.go:326, cmd/open_burst_run.go:151
  DESCRIPTION: The open target domain ("bare"/"session"/"path"/"alias"/"zoxide") is a plain string threaded through four independent sites: the `openTargetPins` map, the `pinDispatch` table, `globExpandableDomain`, and the `resolveOpenSurfaces` classification switch. Only the map↔cobra pairing is drift-guarded (TestOpenTargetPinsCoverValueTakingFlags); nothing guards that every domain the map can emit is handled by the `resolveOpenSurfaces` switch — and that switch has no `default` arm, so a Target carrying an unmodelled domain is silently dropped from the surface set (not a miss, not an error, just absent from the burst). A future value-taking pin added to the map but missed in the switch would therefore vanish targets rather than fail loudly. Separately, the user-facing command-on-attach guard string is authored verbatim in two files (open.go:326 and open_burst_run.go:151) rather than single-sourced, so the single-target and multi-target arities of the "same guard" can drift.
  RECOMMENDATION: Promote the domain to a small typed enum shared by the map, the pin table, and both switches, and add a `default` (panic/error) arm to `resolveOpenSurfaces` so an unhandled domain fails loudly instead of dropping a target. Hoist the command-on-attach message to one shared const referenced by both guard sites.
SUMMARY: The resolve→surfaces→burst→connect flow composes cleanly and its seams (shared spawn.Burster, productionSpawnSeams bundle, classify.go chokepoint, byte-tested composeOpenArgv) are well-factored, but three structural weaknesses remain: doctor drags the over-scoped CollectStatus for a 3-field daemon probe while double-reading the index; the single/multi routing runs a raw os.Args parse parallel to cobra that leaves a production-dead-but-divergent glob branch in the resolver; and the bare-string domain vocabulary lacks an exhaustiveness guard so a future domain could silently drop targets.
