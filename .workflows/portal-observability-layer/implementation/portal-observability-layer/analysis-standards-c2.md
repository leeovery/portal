FINDINGS:

- FINDING: `signal-hydrate` command's FIFO-signal lines attributed to `hydrate`, not `signal`
  - SEVERITY: low
  - FILES: cmd/state_signal_hydrate.go:44, cmd/state_signal_hydrate.go:50, cmd/state_signal_hydrate.go:61, cmd/state_signal_hydrate.go:98
  - DESCRIPTION: The Subsystem prefix taxonomy assigns the `signal` component to "the FIFO signaling MECHANISM — EagerSignalHydrate and the lower-level FIFO signal send/receive plumbing." The two structurally-parallel FIFO signalers that enumerate `@portal-skeleton-*` markers and write the signal byte are split across components: `cmd/bootstrap/eager_signal_hydrate.go` (EagerSignalHydrate) and `internal/state/signal_hydrate.go` both bind `signalLogger = log.For("signal")`, but the `portal state signal-hydrate` command body `runSignalHydrate` wires `cfg.Logger = hydrateLogger` (component=hydrate) and emits its marker-enumeration WARN ("list skeleton markers failed"), per-session WARN ("list panes for session failed"), and per-FIFO-write WARN ("write fifo failed") under `hydrate:`. `runSignalHydrate` is the per-session peer of `EagerSignalHydrate` — same enumerate-markers-then-write-FIFO shape, fired from the client-attached hook — yet its lines are not greppable under `signal:`. The spec's grep idiom ("grep signal: reconstructs the FIFO-signaling behaviour") therefore misses the hook-driven signaling path. The ambiguity is real (the spec scopes `signal`'s plumbing to "in internal/state" and `runSignalHydrate` lives in cmd/, while `state signal-hydrate`'s process_role is `hydrate`), but the resulting attribution is inconsistent with its own EagerSignalHydrate sibling.
  - RECOMMENDATION: Decide whether `runSignalHydrate`'s diagnostics belong under `signal` (matching EagerSignalHydrate and the internal/state plumbing, so `grep signal:` is complete) or `hydrate` (matching the command's process_role), and apply it consistently. If `hydrate` is the intended home, add a one-line note to the taxonomy that the `signal-hydrate` command's own diagnostics render under `hydrate` (only the lower-level send plumbing renders under `signal`) so the boundary is documented rather than incidental.

- FINDING: EagerSignalHydrate loop cycle emits no INFO cycle summary
  - SEVERITY: low
  - FILES: cmd/bootstrap/eager_signal_hydrate.go:93-118
  - DESCRIPTION: The Cycle-level summary mechanical rule states every "Loop cycle — a for loop iterating distinct items" MUST emit exactly one INFO summary at completion (start time captured, items + anomalies counted, one INFO with took). EagerSignalHydrate iterates the skeleton-marker map writing one FIFO signal per pane (a loop cycle by that definition) but emits only per-FIFO WARN/DEBUG and no terminal INFO summary — there is no "signal: ... complete <unit>=N took=T" line. This is a deliberate-looking omission: the Concrete cycle catalog (the authoritative enumeration of "sites this rule mandates a summary at") lists ten cycles and does NOT include EagerSignalHydrate, and the closed cycle-summary unit keys (sessions/panes/entries/steps + sweep outcomes reaped/killed/skipped/unset) have no natural unit for "FIFOs signalled" — adding one would need a vocabulary amendment. So the mechanical rule and the catalog are in mild tension here. Flagging it as the one cycle-shaped function with no summary so the divergence is a recorded decision rather than an oversight; the bootstrap step-complete INFO (step=EagerSignalHydrate) already gives operators a coarse per-step marker, which softens the impact.
  - RECOMMENDATION: Confirm EagerSignalHydrate's no-summary state is intended (catalog is authoritative, no closed unit key fits) and leave a one-line note to that effect at the function, OR if a summary is wanted, add the unit key to the closed cycle-summary vocabulary via amendment first. Do not invent a unit key ad hoc at the call site.

- FINDING: Bare `os.Exit(1)` in the hydrate exec-failure fallback (outside the sanctioned exception)
  - SEVERITY: low
  - FILES: cmd/state_hydrate.go:434-437
  - DESCRIPTION: The Defensive-invariants rule states "Bare `os.Exit` is prohibited outside `main`" (PR-review reject), with exactly one sanctioned exception (the daemon self-eject, which pairs `daemon: self-eject` + `log.Close(0)` before exiting). `defaultExecShell` calls `syscall.Exec` and, only if exec returns an error (exec failed), falls through to a bare `os.Exit(1)` with no preceding terminal marker and no `log.Close`. On that exec-failure path the just-emitted `hydrate: exec` INFO marker becomes misleading (it announces a handoff that did not happen) and the process exits unmarked by a `process: exit`/`process: panic` line — the exact "process vanished without a terminal marker" shape the four-way classification calls "genuinely alarming." The window is narrow (exec almost never fails after argv validation), and on the success path syscall.Exec never returns, so the line is effectively dead on the happy path — but it is a literal exception to the prohibition that the spec did not sanction.
  - RECOMMENDATION: On the exec-failure fall-through, emit a terminal marker before exiting (e.g. `log.Close(1)` so a `process: exit code=1` is paired, mirroring the daemon self-eject's Close-before-exit discipline), or have `defaultExecShell` return the exec error to the caller and let the normal return/Close path own termination. Either keeps the exec-failure path inside the "every termination is marked" contract.
