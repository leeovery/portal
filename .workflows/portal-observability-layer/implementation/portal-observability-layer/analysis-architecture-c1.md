- FINDING: Two-logger split inside a single cycle function obscures the seam contract
  - SEVERITY: medium
  - FILES: internal/state/fifo_sweep.go:36-72, internal/state/fifo_sweep.go:13-20
  - DESCRIPTION: `SweepOrphanFIFOs(dir, liveMarkerKeys, logger *slog.Logger)` takes an injected `*slog.Logger` but uses it for only the per-item WARN lines (lstat/remove failures). The cycle-summary INFO (`"orphan-fifo sweep complete"`) and the per-reaped DEBUG breadcrumb go to a *different*, package-level logger (`cleanLogger = log.For("clean")`). So a single invocation of one function emits records under two components driven by two distinct logger objects, with the injected parameter no longer being the function's logger. This is the only function in the state package that does this — its peers (commit.go, capture.go, scrollback.go) use the injected logger uniformly, and signal_hydrate.go's WriteFIFOSignal uses only a package-level logger and takes no logger param. The split is documented as intentional (WARNs carry the caller's bootstrap component for correlation; the summary groups under `clean`), but the resulting signature is misleading: a caller passing `logger` reasonably expects it to be the function's logging sink, yet the headline summary silently bypasses it. A future "consolidation" onto the injected logger would silently re-attribute the summary away from `clean`; dropping the now-near-vestigial parameter would silently re-attribute the WARNs.
  - RECOMMENDATION: Make the component-attribution intent explicit at the boundary rather than relying on a body-internal dual binding. Either (a) drop the injected `*slog.Logger` parameter entirely and emit every line under `cleanLogger` (simplest; per-item WARNs lose the bootstrap-step component but gain consistency), or (b) keep the split but rename/document the parameter to signal it is a *caller-component WARN sink* distinct from the sweep's own summary logger. Apply the same decision to any other cycle function that grows a caller-vs-self component distinction so the pattern is uniform.

- FINDING: process_role taxonomy decoupled from the Cobra command tree it mirrors
  - SEVERITY: low
  - FILES: internal/log/process_role.go:41-72, main.go:32
  - DESCRIPTION: `ResolveProcessRole` re-derives the subcommand-to-role mapping from a hand-maintained longest-prefix match over `os.Args` (`state daemon` → daemon, `open`/`x`/`attach` → tui, etc.) because `Init` must run before Cobra parses argv. This is a reasonable necessity, but it places a second, independent copy of the command-routing knowledge in `internal/log`, structurally divorced from `cmd/`'s actual Cobra command registration. If a subcommand is renamed or added, the role table will not fail to compile and will silently fall through to `roleBootstrap` — every affected line then mis-attributes `process_role`, which the spec calls out as "critical for multi-writer disambiguation on reboot-recovery days." There is no compile-time link or test fixture tying the role table to the real command set.
  - RECOMMENDATION: Add a single guard test that enumerates the production command verbs (or the canonical argv shapes for each role) and asserts `ResolveProcessRole` returns the expected role for each, co-located so a renamed/added subcommand forces a visible failure. No production restructuring needed — the resolution-before-parse constraint is sound; the gap is the absence of a drift tripwire across the cmd/-to-internal/log boundary.

SUMMARY: The observability layer composes cleanly overall — the swappable-handler indirection, the single-owner internal/log package with a strict import-cycle guard, the shared retention/sweep entry point (no duplicated walk logic), and the consistent store-seam audit chokepoints are well-structured. Two seam-clarity issues remain: a single cycle function (SweepOrphanFIFOs) splits its output across an injected logger and a package-level logger in a way its signature does not convey, and the process_role taxonomy is an unguarded second copy of command-routing knowledge that can silently drift from the Cobra tree.
