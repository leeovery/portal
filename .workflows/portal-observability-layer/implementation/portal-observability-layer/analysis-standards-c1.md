- FINDING: Redundant daemon startup INFO line ("daemon: starting") retained against explicit spec drop
  - SEVERITY: medium
  - FILES: cmd/state_daemon.go:596
  - DESCRIPTION: The spec's "Saver and daemon lifecycle event taxonomy" → "Process/subsystem boundary" section explicitly resolves that the daemon's startup is marked by `process: start process_role=daemon` and that a redundant `daemon: spawn` event "(it would fire at the same instant as `process: start process_role=daemon`, carrying the same data) is therefore dropped." The cataloged daemon lifecycle events are exactly three: `lock acquired`, `self-eject`, `shutdown`. The implementation still emits `logger.Info("starting")` under component=daemon at the head of the daemon RunE — an uncataloged INFO firing at the same instant as `process: start process_role=daemon`, carrying no unique data (version/pid are now baseline attrs). The in-source comment admits it is a migration bridge: "Kept here so no existing log line silently disappears mid-migration." This is the redundant spawn marker the spec said to drop; leaving it produces a duplicate daemon-start INFO at the production-default level and an event outside the closed daemon lifecycle catalog.
  - RECOMMENDATION: Remove the `logger.Info("starting")` call in cmd/state_daemon.go's daemon RunE. The OS-process-boundary `process: start process_role=daemon` (emitted by log.Init in main) plus the cataloged `daemon: lock acquired` cover the daemon's startup; no additional daemon-component start line is sanctioned by the taxonomy.

- FINDING: State-mutation `op` rendered as the slog message, not as the required `op=` attr
  - SEVERITY: medium
  - FILES: internal/hooks/store.go:95, internal/hooks/store.go:140, internal/hooks/store.go:190, internal/alias/store.go:187, internal/alias/store.go:215, internal/project/store.go:141, internal/project/store.go:255, internal/project/store.go:291, cmd/config.go:59, cmd/config.go:66, cmd/config.go:72
  - DESCRIPTION: The "State-mutation audit trail" mechanical rule lists `op` under "Required attrs: op — drawn from the closed value space below," and `op` is an explicit entry in the closed attr-key vocabulary (49 keys). Across every store mutation site and the config-migration site, the implementation passes the op verb as the slog *message string* (e.g. `logger.Info(op, "hook_key", key, ...)` and `logger.Info("migrate", "via", "migrate", ...)`) rather than carrying it as an `op=` attr. The rendered text line is therefore `hooks: set hook_key=... via=cli ...` with no `op=set` token, and the JSON handler emits no `"op"` field. `grep "hooks: set"` works, but programmatic filtering by the `op` attr (the spec's stated rationale) and `grep op=` do not. This is a systematic, consistent divergence from the literal "Required attrs: op" rule for the entire state-mutation audit-trail family.
  - RECOMMENDATION: Decide whether op-as-message satisfies the audit-trail intent and either (a) carry `op` as an explicit attr — e.g. `logger.Info("mutate", "op", op, "hook_key", key, ...)` — so the closed `op` attr-key is actually emitted and JSON filtering works, or (b) if op-as-message is the accepted interpretation, record it as a deliberate spec deviation. The closed attr vocabulary still counts `op` as a key, so the current code leaves that key unused in the rendered output.

- FINDING: `project` attr carries the filesystem path, not the project name as specified
  - SEVERITY: low
  - FILES: internal/project/store.go:136, internal/project/store.go:141, internal/project/store.go:251, internal/project/store.go:255, internal/project/store.go:286, internal/project/store.go:291
  - DESCRIPTION: The closed attr-key vocabulary defines `project` as "project name from projects.json" and `value` as the "verbatim new value for set/modify." The project store inverts this: the `project` attr carries the PATH (the addressable match key) and `value` carries the NAME. The deviation is documented in an in-source comment, but it still contradicts the spec's closed-vocabulary definition of what `project` holds. A reader grepping `project=<name>` per the spec will instead find `project=<path>`.
  - RECOMMENDATION: Either align with the spec (emit the project name under `project`, path under the existing `path` key), or amend the spec's attr-key definition to state that `project` carries the path for the projects store. The current state is a silent code-vs-spec contradiction.

- FINDING: Production code constructs *slog.Logger directly outside internal/log (discard fallbacks)
  - SEVERITY: low
  - FILES: cmd/bootstrap/bootstrap.go:48, internal/state/logger_nil.go:14, internal/tmux/portal_saver.go:27, internal/restore/logger_nil.go:14
  - DESCRIPTION: The "Call-site logging pattern" → Prohibited list names "Direct construction of *slog.Logger outside the internal/log package" as a PR-review-reject item, and the package overview states "No *slog.Logger is constructed anywhere outside it." Four production files construct `slog.New(slog.NewTextHandler(io.Discard, nil))` as nil-tolerant fallback sinks. The migration-sweep section sanctions this exact idiom but scopes the sanction to tests. These instances are inert io.Discard sinks used only so a nil Logger field can be dispatched unconditionally — they never emit real output and so do not bypass the configured rotating handler — which makes the divergence benign in practice, but it is a literal violation of the no-direct-construction rule in production code. (Cross-references the duplication agent's discardLogger finding — same sites, consolidation into a single internal/log helper would resolve both.)
  - RECOMMENDATION: Expose a single `log.Discard()` helper in internal/log for nil-tolerant fallback sinks so production never constructs a *slog.Logger directly; alternatively extend the spec's sanction to cover these production nil-fallback sinks explicitly.

SUMMARY: The internal/log foundation, hydrate-helper catalog, lifecycle/cycle-summary/boundary-preservation instrumentation, and component/attr taxonomy conform closely to the specification. The notable drift is a retained redundant `daemon: starting` INFO line the spec explicitly dropped, and a systematic decision to render the audit-trail `op` as the slog message rather than the spec-required `op=` attr; two lower-severity items (project attr carrying path-not-name, and production direct slog.New discard sinks) round out the findings.
