  SEVERITY: low

  DESCRIPTION: Spec §4 states the correctness invariant "`m` appears in `?` help iff `m` is functional." The implementation realises that invariant with two independently-authored predicates for the same underlying condition ("m is blocked at entry"). The actual entry gate in handleMultiSelectToggle is `if m.DetectUnsupported()` inside the surrounding `if !m.multiSelectMode` branch (effective condition: `!multiSelectMode && DetectUnsupported()`). The help-suppression filter in sessionsHelpKeymap recomputes the identical condition as `mBlocked := m.DetectUnsupported() && !m.multiSelectMode`. Nothing structurally ties them together: a future change that narrows or nuances the entry gate (e.g. making it identity-aware, or altering the in-flight handling) would silently desynchronise the help filter, causing the help to hide a working `m` or list a blocked one — the exact invariant the spec elevates. This is the "compose, don't duplicate / derived-from-source" case from code-quality.md: the help predicate is logically "the entry block would fire," and should be derived from one source rather than re-authored. The spec's own §7 tests would catch a drift only if the author remembered to also touch the help path. Both predicates are correct today; this is a latent seam-drift risk, not a present bug.
  RECOMMENDATION: Extract a single predicate method — e.g. `func (m Model) multiSelectEntryBlocked() bool { return m.DetectUnsupported() && !m.multiSelectMode }` — and have both consumers read it: sessionsHelpKeymap's `mBlocked` and the entry-block gate in handleMultiSelectToggle (the redundant `!multiSelectMode` conjunct is harmless there since it is already inside `if !m.multiSelectMode`, and it makes the "help hides m iff entry blocks m" relationship self-evident). This makes the spec invariant structurally true rather than dependent on two authors staying in lockstep. Low priority given the fix's deliberately minimal scope — a small consolidation, not a restructure.

The only observation is a low-severity composition opportunity — the "m blocked at entry" condition is authored independently in the entry gate and the help filter, so they can drift against the spec's "help lists m iff m works" invariant.
