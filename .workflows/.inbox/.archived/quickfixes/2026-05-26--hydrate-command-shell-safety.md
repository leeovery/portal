# Harden the hydrate-command pipeline against whitespace and shell-meta characters in session names

Two related changes that together make the `portal state hydrate` invocation structurally shell-safe regardless of what bytes appear in a tmux session name. The bug surfaces today as a complete resume failure for any session whose name contains whitespace or shell-special characters — the shell word-splits the unquoted argv inside `respawn-pane -k`, the helper exits with bad argv, no FIFO is opened, no hook fires, no scrollback is hydrated. The session `evvi webhooks and watchers` is the live example that triggered this.

The first atom restores the `shellQuoteSingle` helper that was deleted in commit `98086bd9` ("T8-1 — delete shellQuoteSingle and emit bare hydrate invocation") and applies it to all three interpolated values in `buildHydrateCommand` (`internal/restore/session.go` ~line 434): `fifo`, `file`, and `hookKey`. Single-quote wrap with the standard `'\''` close-escape-reopen idiom. The docstring on `buildHydrateCommand` needs its incorrect claim removed — it asserts that sanitization filters dangerous bytes so the bare form is safe in practice, which is false (sanitization currently only strips `/`, `\`, `\0`) and was the root cause of this defect.

The second atom broadens `sanitizeSessionName` in `internal/state/panekey.go` (~line 41) from a blocklist to an allowlist: any byte not in `[A-Za-z0-9._-]` becomes `_`. The existing collision-suffix mechanism in `SanitizePaneKey` (lines 30-33) already handles disambiguation of distinct names that collapse to the same stem, so no further change is required there. The `sanitizeSessionName` docstring needs to reflect the allowlist contract.

These compose: atom 1 fixes the bug; atom 2 closes the predicate gap that caused atom 1's docstring to be wrong. After both, sanitization matches what downstream consumers were implicitly relying on, and quoting remains as defense-in-depth for `hookKey` — which is preserved-raw by design because it's the round-tripping key for `hooks.json` and cannot be sanitized without breaking lookup. The allowlist is the smallest invariant that holds across every consumer (filesystem paths, tmux option names, shell command interpolation) and mirrors how other path-naming systems sanitize identifiers; the collision suffix preserves separability of distinct names that map to the same stem.

Scope: `internal/restore/session.go` and `internal/state/panekey.go` for production. Test updates: `session_build_hydrate_test.go` has three existing sub-tests pinning the bare-form output — two need updating to expect the quoted form, plus new sub-tests for whitespace and embedded-single-quote inputs. `panekey_test.go` needs new sub-tests for whitespace and shell-meta-character inputs; the existing five sub-tests for `work`, `foo/bar`, `.hidden`, `foo\x00bar`, and the hash-collision case all pass unchanged because the broaden is additive over the current cases.

Non-scope: no change to `hooks.json` schema, `sessions.json` schema, marker name format, or FIFO path format. The eleven production callers of `SanitizePaneKey` across `cmd/`, `internal/state/`, `internal/restore/`, `internal/bootstrap/`, `internal/tui/`, and `internal/restoretest/` all use it transparently as the single source of truth, so the output change propagates without per-site updates. `PaneKeyFromFIFOPath` in `internal/state/paths.go` is unaffected — it extracts whatever appears between `hydrate-` and `.fifo` in a basename and doesn't depend on the sanitization shape.

Migration: a one-time event per affected session. Scrollback `.bin` files for sessions whose name newly sanitizes to a different stem become orphans on first daemon tick after the change (for example `evvi webhooks and watchers__1.1.bin` becomes orphan when the broaden produces a different paneKey). Daemon's scrollback GC sweeps orphans within a tick or two. Sessions named normally — alphanumeric, dash, underscore, dot only — are unaffected because their sanitized stem doesn't change. Cost is at most one tick's worth of pre-change scrollback on the first reboot after the fix ships, for the affected sessions only; from then on full operation resumes.

Context: surfaced during the verification of the `slow-open-empty-previews-and-zombie-sessions` bugfix release while reviewing whether the 22 currently-registered hooks would all resume cleanly on next reboot. The `evvi webhooks and watchers` session — created externally via `tmux new-session -s` with that name, bypassing Portal's own session-name generator — was identified as the one session that would fail to resume. This quickfix closes that single-session gap and prevents the entire bug class for any future externally-created session name carrying spaces, quotes, `$`, backticks, or other shell-meta bytes.
