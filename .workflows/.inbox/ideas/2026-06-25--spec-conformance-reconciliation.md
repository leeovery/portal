# Reconcile three spec-vs-implementation divergences (fix code or amend spec)

The spectrum-tui-design review found three places where the implementation diverges from the literal specification without a recorded decision. Each clears its functional bar (legibility / floors / behaviour), so none is a defect — but each needs a deliberate "fix the code OR amend the spec" call by the spec-owner, and at least one has a test that currently *encodes* the divergence (so it pins the implementation, not the spec).

1. **No-tags signpost on-band text token.** `notice_band.go:372-377` (`noticeBandOnBandText`) renders the §11.3 signpost message in `text.on-selection`, but §11.3 AND the plan acceptance criterion (`tick-c2c98c`) specify `text.strong`. The justification is a §2.9 co-tuning argument (the band sits on the `bg.selection` tint, the same pairing the selected-row name uses), but it isn't recorded, and `bytag_signpost_reskin_test.go:74-77` asserts `text.on-selection` — so the test encodes the deviation. Decide: reconcile to `text.strong` (verifying it clears the 4.5:1 text floor on the `bg.selection` tint), or amend §11.3 to record `text.on-selection` as the co-tuned on-band token (the §2.9 "text token moves too" rule); then update the test to match the agreed token.

2. **Phase-1 light token values vs the §2.9 table.** `theme.go` pins `text.detail` Light `#586093` and `text.dim` Light `#767DA2`, where the §2.9 table lists `#5A6296` / `#7C84AA`; `accent.violet` light `#8A3FD1` measures 4.37 against `#e1e2e7`, not the published 5.7. These are documented in-source as `#FFFFFF`-vs-`#e1e2e7` erratum corrections (and all clear their floors), but the §2.9 golden table still shows the original values. Decide which is canonical and reconcile theme.go or the spec table so they agree. (The review already added an inline erratum note to `accent.violet` for consistency with the other six corrected tokens; this item is the spec-table side.)

3. **No-matches glyph.** `filtering_no_matches.go:26` ships `∅` (U+2205 EMPTY SET) where §7.3 byte-specifies `⌀` (U+2300 DIAMETER SIGN). Visually near-identical. *Reviewer lean: keep `∅` and amend the spec — U+2205 "empty set" is arguably more apt for a no-results state than "diameter sign", and it is more widely supported across terminal fonts (U+2300 renders inconsistently).* Either way, record the decision; changing the glyph is a one-line edit but a VISUAL change requiring a `filtering-no-matches` re-capture.

Source: review of spectrum-tui-design/spectrum-tui-design (reports 4-3, 2-7, 1-4, 2-9).
