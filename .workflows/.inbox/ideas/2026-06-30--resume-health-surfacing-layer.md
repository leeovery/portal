# A health validation + surfacing layer for the resume system

**Status: undiscussed, zero decisions.** This note exists only to seed a future discussion workflow. Everything below the motivation is starting material to react to, not a proposal or a plan. Nothing here is agreed, and "should we build it at all" is itself open.

The idea: Portal should proactively detect and surface resume-system health problems, so that when you open it, it tells you something's wrong rather than failing silently. The motivating framing was: "If I open Portal now and there's some issue going on — the daemon has stopped running, or the saver session has stopped running, or the hooks are empty but we have a load of sessions — I'd like some type of validation layer that runs relatively frequently that surfaces issues." It came out of a day spent chasing silent resume failures (renamed sessions losing their hooks, sessions not resuming) where everything *looked* fine but the resume system was quietly broken.

Some grounding established while this came up (facts, not decisions). A partial check engine already exists: `portal state status` (`cmd/state_status.go`) collects a `state.StatusReport` with the daemon's pid/version and last-save freshness, and exits "unhealthy" when the daemon is down or the last save is stale. Bootstrap also already self-heals the big two on every open — a dead daemon or `_portal-saver` gets recreated by `EnsureSaver`, and a persistent failure surfaces as `SaverDownWarning`. So "detect daemon down" is largely handled already. Two gaps were noticed: (1) nothing surfaces health in the TUI picker proactively — `state status` is a manual command you have to think to run; (2) the specific invariant named — "hooks empty but a load of live sessions," i.e. resume silently broken while everything looks healthy — isn't checked anywhere.

Seed framing to react to (un-decided): one possible shape is two halves — extend the check engine (`StatusReport`) with invariants bootstrap doesn't cover, chiefly hooks↔sessions coherence and explicit saver-session liveness; and add a proactive surface that runs at TUI launch and shows a notice only when unhealthy, with `state status` (or a friendlier `portal doctor`) for detail. One lean was floated — start tight and quiet, problem-only notice, no auto-repair beyond bootstrap — but that is a lean to argue with, not a choice.

Open questions for the discussion: scope (quiet notice + extended status vs an always-on indicator / auto-repair / full doctor), cadence (on-open vs continuous), which invariants are worth checking, how loud the surface should be, and whether to build it at all.
