# Remote-triggered multi-window burst opens windows on the wrong machine

Triggering a multi-select spawn burst from a **remote** tmux client — e.g. Blink on an iPhone/iPad connected to the Mac over SSH or mosh — while a **host-local** terminal client (e.g. Ghostty on the Mac itself) is also attached to the same tmux server, opens the N−1 spawned windows on the Mac's local terminal. The person who pressed Enter is on the iPad; the windows appear on a screen they are not looking at. Their own (trigger) window does self-attach to the Nth session, so from the iPad it looks like a partial or confusing success while a pile of new Ghostty windows silently accumulates on the desk at home.

The condition is specific: it needs a remote triggering client *plus* at least one host-local client attached to the same tmux server at detection time. With no local client attached, detection resolves NULL and the burst correctly takes the honest unsupported no-op (`⚠ no host-local terminal — nothing opened`), so the pure-remote case is fine. The mixed case is the problem.

Mechanism as observed in conversation (not yet validated): inside-tmux detection (`internal/spawn/detect_inside.go`) enumerates `list-clients`, filters to host-local clients (remote/mosh clients walk to a NULL identity and are excluded), and uses `client_activity` only as a tiebreak *among locals*. Detection therefore answers "does any host-local terminal exist on this server?" rather than "is the client that triggered this burst local?". The remote trigger's own locality never gates the decision, so the burst resolves to whatever local terminal happens to be attached and drives it.

Expected behaviour: when the client that triggered the burst is remote, the burst should resolve unsupported and take the same atomic no-op as the pure-remote case — even when local clients are attached. Windows should never open on a machine/display the triggering user isn't at.

Impact: real-world for this user — they routinely leave a local terminal attached on the Mac and connect from Blink on iPad. Severity is moderate: nothing is destroyed, but host windows are spawned invisibly (Portal deliberately never tears down host windows it opened, so they linger until manually closed), and the triggering user gets no indication anything went to the wrong place.

Related: the broader question of ever supporting mobile terminals as spawn targets was discussed and judged infeasible (no host→device control channel); this bug is independent of that — it is about the detection gate, not about adding a Blink adapter.

---

**Note (2026-07-18, cli-verb-surface-redesign planning):** The CLI-verb redesign adds a **second trigger surface** for this same bug. Today only the TUI multi-select and `portal spawn` reach the burst; the redesign re-homes the burst under a CLI **multi-target `portal open <a> <b> …`** (N≥2), which shares the identical `internal/spawn` detection gate (`detect_inside.go`). So `open a b c` typed from a remote client (Blink/mosh) with a local Ghostty attached will hit the exact same wrong-machine spawn. When this bug is scoped, the trigger-locality fix should cover the CLI `open` burst too, not just the picker. (This bug stays out of scope for the redesign — `internal/spawn` detection is explicitly out of scope there; the redesign only makes the CLI burst take the honest unsupported block on whatever `Detect()` reports.) Also note `spawn --detect` retires into `portal doctor` (host-terminal line) under the redesign, so the diagnostic surface for this is now `doctor`.
