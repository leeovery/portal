# Daemon self-deadlocks at midnight log-rotation day-roll — captures silently freeze

The `portal state daemon` deadlocks the moment its log write crosses a calendar-day boundary with a retention-deletion candidate present, and stays alive-but-frozen indefinitely: no capture ticks, no `sessions.json` commits, no scrollback dumps. Symptom that surfaced it: a session created 2026-07-11 had no scrollback at all in the TUI preview, and every other session's preview was days out of date.

Diagnosed 2026-07-12 against the live install. `rotatingSink.Write` holds `s.mu`; on a date change `reopen` fires `s.dayRoll()` while still holding the mutex (`internal/log/sink.go:338-342`). The retention sweep inside that callback emits `rotateLogger.Info("deleted", ...)` per deleted file (`internal/log/retention.go:190`), which re-enters the same sink's `Write` and blocks on `s.mu.Lock()` — a self-deadlock on a non-reentrant mutex. The capture loop logs every tick, so the entire daemon freezes on that one call. The `sealPastDayFiles` chmod WARN and the sentinel-prune WARN are the same hazard.

The trigger only armed recently: logging shipped ~2026-06-04 and retention is 30 days, so the first midnight with a deletion candidate was ~Jul 5-6. From now on a candidate exists every midnight, so any long-lived process crossing midnight with the configured handler deadlocks nightly. Incident of record: daemon pid 49945 froze at 2026-07-06 00:00 (last log line 2026-07-05T23:59:53; its open fd was `portal.log.2026-07-06` containing zero daemon records). It sat wedged for six days and passed every guard — flock, saver-pane pid probe, version guard, orphan sweep, and self-supervision (which runs on the deadlocked goroutine) — because all of them check aliveness/identity, never liveness. Killed and respawned 2026-07-12; the replacement will re-deadlock tonight and every night until fixed, needing a manual kill each morning.

Secondary bug in the same area: fresh processes run the identical day-roll inside `Init`'s `sink.probe()` before `setHandler` swaps in the configured handler (`internal/log/init.go:54-55`), so their sweep breadcrumbs route to the pre-Init stderr handler and vanish — the deletion audit trail never lands in `portal.log` (verified: zero `log-rotate:` lines in any July log despite daily deletions). This accidental ordering is also the only reason short-lived processes don't deadlock.

Fix direction discussed: emit day-roll sweep events outside the sink mutex (collect under lock, emit after unlock, or defer `dayRoll` past the critical section), and fix the `Init` probe/`setHandler` ordering so sweep breadcrumbs reach the real handler. Tertiary and possibly separate scope: no liveness detection exists for a wedged-but-alive daemon.
