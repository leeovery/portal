# Renaming a session silently orphans its resume hook

When a tmux session is renamed with `tmux rename-session`, its Portal resume hook stops working, and the session comes back as a bare shell after the next reboot instead of resuming what was running in it. The session keeps running normally in the meantime, so nothing looks wrong until a restart — at which point the resume silently fails.

The resume hooks in `hooks.json` are keyed by the structural key `session_name:window.pane` (the format lives in `internal/tmux/tmux.go`, `StructuralKeyFormat`). A rename changes the session name but does not restart whatever process is running inside the pane. So after a rename three things line up against the session: the stored hook key no longer matches the live pane's structural key; Portal's stale-hook cleanup then deletes the now-unmatched key because no live pane corresponds to it; and because the inner process never restarted, nothing re-registers a hook under the new name. The hook is simply gone, and the user has no signal that it happened.

Observed live on 2026-06-30: of 24 live Claude sessions, exactly two had no resume hook, and both were sessions that had been renamed. `finder-v2` was created the day before as `finder-wlRUOm` with a hook under `finder-wlRUOm:1.1`; after the rename, the same Claude process is still running but carries no hook. `agentic-workflows-refactor-wt` shows the identical pattern — originally hooked under `agentic-workflows-vAKe79:1.1`, renamed, hook gone. The correlation is clean: every session whose hook survived still carries its original generated nanoid name, and the only two that lost their hooks are the two that were renamed away from that name. The underlying Claude history files still exist, so the existing prune-on-missing-history path is not involved.

Impact: any time a session is renamed to something more memorable — a common, encouraged workflow — its reboot resume is silently dropped. This is one of the "didn't resume" cases behind recent partial-restore reports (e.g. 28 of 32 sessions resuming).

Root cause and a fix direction were worked through in the originating conversation (the keying is on the mutable session name; a stable rename-immune session identity is the candidate fix) — left for the discovery/scoping phase to validate and shape. Distinct from the separate, parked SessionEnd-cleanup question (bare shells resurrecting Claude).
