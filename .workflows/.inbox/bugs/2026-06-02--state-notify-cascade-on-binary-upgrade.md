# `portal state notify` fires continuously at multi-Hz rate while the daemon is alive; killed by saver-recycle

Between 20:35:29 and 20:42:24 BST on 2026-06-02 — a ~7-minute observation window — `portal state notify` was invoked 2315 times. Each invocation is a full short-lived process: it boots, emits its three lifecycle marker lines (`process: start`, `process: log-level resolved`, `process: exit`), and exits in approximately 500µs. The rate was not uniform: an initial 53-second burst at ~23/sec (the first 1224 invocations), then a slower-but-still-significant ~2–14/sec sustained for another ~6 minutes, until it ceased completely at 20:42:24. Total impact for the window: ~6945 log lines and ~933 KB written to `portal.log.2026-06-02`.

The cascade stop is the most diagnostically important observation. At 20:42:21 the user invoked `portal open` for the first time since the brew upgrade from 0.5.12 → 0.6.0. The new 0.6.0 binary's bootstrap step 5 detected the saver-pane daemon was still running 0.5.12 (PID 36689), executed Component A's kill-barrier, and Component F respawned the daemon as PID 36776 on 0.6.0. The final `state notify` invocation timestamp is 20:42:24.218 — eight milliseconds before `daemon: lock acquired tmux_pane=%144 pid=36776`. From that point onward, zero `state notify` invocations have appeared in the log, despite the daemon continuing to run normal capture ticks every ~32 seconds with `sessions=21 panes=23 natural_churn=0`.

This redirects the diagnosis. The cascade is not tied to the upgrade moment or the saver-recycle itself; the saver-recycle is what STOPS it. The cascade was being driven by something the old 0.5.12 daemon was doing — or by stacked tmux hooks the new bootstrap reset — and went silent the instant the 0.5.12 daemon was killed. The old daemon's actual lifetime is not established from current evidence; the brew upgrade preceded the portal open by approximately 5–10 minutes, so all that can be claimed is that the cascade was visible for ~7 minutes between 20:35:29 and 20:42:24. Whether it was running before brew install is unknown — pre-0.6.0 binaries had no `process:` lifecycle markers, so the spawned `state notify` processes left no per-invocation trace in `portal.log`, and only the daemon-side capture-tick churn would have been visible (capture ticks proceed at their normal 1s cadence regardless of how many `state notify` calls fire). The "cascade was running invisibly for hours" inference is plausible but not evidenced by what's been captured today.

The investigation needs to discriminate between two structurally different root causes that the post-recycle silence is compatible with: (1) the 0.5.12 daemon itself was generating the notify invocations via some internal loop or feedback, and the new daemon does not exhibit this behaviour; or (2) tmux global hooks were stacked from prior binary lifetimes such that one tmux event fired many `state notify` invocations, and the new bootstrap's `RegisterPortalHooks` step reset the stack to a single entry per hook. Both are consistent with the observation; the evidence on hand cannot tell them apart. Reproduction strategy depends on which: hypothesis 1 requires running the 0.5.12 binary again in isolation; hypothesis 2 requires examining the tmux global hook list shape across a forced saver recycle (`tmux show-options -g` before and after).

The `0.6.0` binary itself appears to be cascade-free in the post-recycle window: ~3 minutes of normal idle activity has produced only daemon capture-tick INFO lines at ~32s intervals, no `state notify` invocations whatsoever.

Severity is high. A sustained multi-Hz invocation rate (averaging ~5.5/sec across the observed window, peaking at ~23/sec) is significant fork+exec overhead, tmux server load (each invocation presumably touches tmux state), and write amplification on `save.requested` (the marker `state notify` exists to write). Whether this cascade runs continuously while a 0.5.12 daemon is alive — or fires only during certain windows — cannot be determined from a single 7-minute observation. The user-visible symptom that may correlate is the recurring `hooks-and-saver-vanish-after-recent-fixes` defect — if a high-rate notify cascade was driving `save.requested` writes, the daemon's capture loop would be under sustained pressure during the windows when the previously-observed wipes occurred.

Relevant files: `cmd/state_notify.go` (or wherever the notify subcommand is declared), `cmd/bootstrap/bootstrap.go` (step 2 `RegisterPortalHooks`), `internal/bootstrapadapter/hook_registrar.go` (production hook registrar — for cross-binary-lifetime idempotency), `internal/tmux/portal_saver.go` (Component A kill-barrier + Component F respawn — the path that stops the cascade), `internal/state/capture.go` (the daemon's `save.requested` polling consumer), and the `portal.log.2026-06-02` evidence file pinning both the 2315-invocation window and the post-recycle silence.
