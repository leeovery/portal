# Bootstrap step 11 (CleanStale) silently wipes all entries from hooks.json when tmux is in a transient state

On 2026-05-26 we observed `~/.config/portal/hooks.json` go from 22 valid entries (a complete set of per-pane on-resume hooks registered by the user's `portal-resume-backfill` script at 13:59 BST, covering every live Claude session in every active tmux pane) to `{}` empty at approximately 17:04 BST, with no user action that should have produced that outcome. The user did not run `portal hooks rm` for any pane. The user did not run `portal clean`. No Claude `SessionEnd` events fired against the user-level Portal hook script during that window — `/tmp/portal-resume-hook.log` only shows CodexBar probe invocations that exited early at the `TMUX_PANE unset` guard before reaching the SessionEnd branch, and the script's SessionEnd branch had already been softened earlier the same day so it no longer calls `portal hooks rm` even when it does fire.

The wipe correlated tightly in time with a Portal bootstrap that left a single WARN entry in `~/.config/portal/state/portal.log` at `17:11:25Z`: `sweep: list-panes _portal-saver failed, legitimate set empty: list-panes -t _portal-saver: exit status 1: can't find window: _portal-saver`. That WARN is Component B's orphan-sweep noting a transient tmux state during which `_portal-saver` was momentarily absent. Component B logged the warning and continued (its documented best-effort posture). But the same bootstrap cycle also ran step 11 (`CleanStale`), which hit the same kind of transient tmux failure when it tried to enumerate live panes — and unlike Component B, CleanStale did not log and did not skip. It interpreted the transient as "zero live panes exist" and proceeded to remove every hook entry whose key was not in the empty live set, i.e. all 22 of them.

User-visible impact: after the wipe, no Claude session auto-resumes on next reboot because every per-pane on-resume hook entry is gone. The user has to manually identify and resume each Claude session via Claude's own session picker, exactly the scenario the Portal resume system was built to prevent. From the user's perspective, this is indistinguishable from the original `slow-open-empty-previews-and-zombie-sessions` bug ("none of my Claude sessions resumed") even though the upstream cause is entirely different — that bug was about FIFO timing during restore, this one is about the cleanup-after-restore step destroying state during a tmux transient.

Conditions under which this is observed (or expected) to trigger: any Portal-bootstrap-triggering command (`portal open`, `x`, `portal hooks set`/`rm`, etc.) invoked while the tmux server is in a brief transient state where `tmux list-panes -a` returns a non-zero exit code. Empirically, the moments of highest risk appear to be: (a) the saver-respawn window after the recent zombie-sessions bugfix triggers a kill-and-recreate of `_portal-saver`, (b) the version-upgrade kill cycle in `EnsurePortalSaverVersion`, (c) any time the tmux server is under heavy load, and (d) the same race that produced the `sweep: list-panes _portal-saver failed` WARN in the observed instance. The bash-side mitigations we put in place earlier the same day (softening SessionEnd to a no-op; rebuilding the backfill script) are completely undermined by this defect — every bootstrap is a fresh opportunity for hooks.json to be silently emptied.

Not a regression from any recent change. The destructive pattern has existed since `CleanStale` was wired to the live-pane enumeration; the recent zombie-sessions and hydrate-command-shell-safety releases increase exposure because they involve more bootstrap activity during which tmux transients become more frequent.

Files most directly involved: `cmd/bootstrap_production.go` (`cleanStaleAdapter.CleanStale` at line 76), `cmd/clean.go` (the `portal clean` subcommand at line 82 — same destructive pattern via the same `ListAllPanes` path), `internal/tmux/tmux.go` (`ListAllPanes` at line 687), `internal/hooks/store.go` (`Store.CleanStale` at line 130). The relevant `portal.log` entry sits at line 1 of the current log file; previous log content was rotated during one of today's bootstrap cycles. The 22 hook entries that were wiped are recoverable via `~/.claude/hooks/portal-resume-backfill.sh --apply`, which walks every live Claude session inside tmux and re-registers a hook for each — but that recovery is one-shot and not durable until this bug is fixed.
