# Killed session resurrects within daemon-tick window

After killing a tmux session (via `Option-Q` / tmux's `M-q kill-session`, or via Portal's TUI `K` binding + confirm), the session continues to appear in the Portal Sessions list for several seconds before settling. On the user's machine running Portal 0.5.0 the threshold is reproducibly between ~2 and ~5 seconds wall-clock between the kill and the next `portal` invocation: at a 2-count the dead session is still listed and is actively reconstructed in tmux as a skeleton pane (`pane_start_command` is `portal state hydrate ...`, the signature of bootstrap step 5 restore in `internal/restore/session.go` / `internal/restore/restore.go`); at a 5-count the session is gone from both the list and from tmux and stays gone.

To the user this presents as "I killed this session, but Portal brought it back" — a trust regression on the same user-visible surface as the two recently-shipped resurrection-class bugfixes (`daemon-merge-reintroduces-dead-sessions` and `killed-sessions-resurrect-on-restart`). This is a sibling symptom, not a regression of either: those fixes target the stale-`@portal-skeleton-*`-marker class. On this machine the marker class is observably resolved (`tmux show-options -s | grep @portal-skeleton` is empty after a cleanup earlier today, and the four previously-affected sessions now have fresh scrollback `.bin` files), yet the symptom is still reproducible from a clean marker state.

The kill plumbing involved is the tmux `session-closed` global hook that fires `portal state notify` (`cmd/state_notify.go`), and the daemon's tick loop (`cmd/state_daemon.go`) that owns rewriting `sessions.json`. The relevant read path on the consumer side is bootstrap step 5 `Restore` in `internal/restore/restore.go`, which reads `sessions.json` on every Portal invocation and reconstructs any session present in the file that is not currently live in tmux.

The user has expressed a clear directional preference for any fix work: **synchronous, not eventually consistent**. Specifically, they do not want this handled with timeouts, retry tuning, daemon tick-rate adjustments, or other narrowing of the race window — they want the race eliminated at its source by making the kill-side path commit the persistence change before returning, so a subsequent Portal invocation always reads a state consistent with the kill that just happened. They have explicitly flagged that hacky / probabilistic mitigations are not acceptable.

Severity: trust-tier user-visible symptom. Cardinality: any user who kills a session and reopens Portal within the race window. Single-user-per-machine product so no concurrency-with-others concern. Surfaced 2026-05-14 during post-upgrade verification of Portal 0.5.0 on the user's primary development machine.
