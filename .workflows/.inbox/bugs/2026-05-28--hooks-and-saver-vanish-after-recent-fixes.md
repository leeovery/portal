# `hooks.json` continues to be silently wiped and `_portal-saver` + daemon keep vanishing despite v0.5.11 and v0.5.12 fixes

`~/.config/portal/hooks.json` was wiped from 21 entries to `{}` between 08:08:02 BST and 08:18 BST on 2026-05-28, with no user action and no reboot. The most recent bash-hook prune-summary log entry at 08:08:02 BST showed `pruned=0 kept=21 skipped=0`; ten minutes later the file was empty. System uptime traces continuously to the previous reboot at 2026-05-26 18:55, so no boot event explains the gap. This is the same user-visible symptom as the earlier `bootstrap-cleanstale-wipes-hooks-on-tmux-transient` defect, but on a build (`portal version 0.5.12`) that supposedly fixed every known wipe vector.

A compounding symptom appeared shortly afterwards: `_portal-saver` was absent, no `portal state daemon` process was running, and `daemon.pid` / `daemon.version` / `daemon.lock` were all missing from the state directory. Earlier in the same hour (08:35 BST) a bootstrap had spawned a fresh daemon at PID 48213 with `daemon.version=0.5.12` and the saver pane pid matching `daemon.pid`. By the time the user noticed mid-morning, PID 48213 was dead, the saver session was gone, and the state-coordination files had been removed. No reboot in that interval.

`portal.log` was truncated to 0 bytes at 08:38 BST, destroying every log entry between 08:18 and 08:35 — i.e. exactly the window covering both the wipe and the saver disappearance. The single surviving entry is a Component B WARN at 07:35:43Z UTC (`sweep: list-panes _portal-saver failed, legitimate set empty: list-panes -t _portal-saver: exit status 1: can't find window: _portal-saver`). This same WARN pattern has recurred on essentially every bootstrap observed this week, indicating the saver is repeatedly absent at bootstrap moments under steady-state operation.

The bash hook script at `~/.dotfiles/home/.claude/hooks/portal-resume-hook.sh` has its own prune logging at `/tmp/portal-resume-hook.log`. Every prune summary visible in that log shows `pruned=0` — the bash-side prune has never removed an entry across any observed invocation. The Portal-side hazard-guard log line (`stale-hook cleanup: zero live panes parsed with N hook(s) present; skipping to avoid mass-deletion hazard`) added by v0.5.12 has not appeared in any surviving `portal.log` entry for the wipe events.

Severity is high: silent data destruction with no user-visible warning, recurring on a multi-hour cadence, eliminating the auto-resume behaviour that is the resume system's primary value proposition. The only recovery path is the user-side `portal-resume-backfill.sh --apply` script. Reproduction is intermittent and the symptoms only become visible after the destroying event, so live capture is hard.

Relevant files: `~/.config/portal/hooks.json` (wiped state), `~/.config/portal/state/portal.log` (truncated evidence), `~/.config/portal/state/daemon.{pid,version,lock}` (missing), `/tmp/portal-resume-hook.log` (intact, exonerates bash-side prune for observed wipes), `cmd/bootstrap_production.go` (`cleanStaleAdapter`), `cmd/clean.go` (RunE hook-cleanup tail), `internal/tmux/portal_saver.go` (saver lifecycle, Component F flow), `cmd/state_daemon.go` (Component D self-supervision).
