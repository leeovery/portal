## spawned-window-dead-ends-on-session-exit-1-1 | approved

### Task 1.1: Wrap Ghostty window command in shell fallback and drop `wait after command`

**Problem**: A burst-spawned (N−1 external) native-Ghostty window dead-ends when its session exits or detaches: the window's only process is the one-shot exec chain `bash -c` → `env` → `portal open` → `syscall.Exec` into `tmux attach-session`, so when tmux returns there is no parent interactive shell to fall back to, and the Ghostty adapter's `wait after command:true` converts "command finished" into the "Process exited. Press any key to close the terminal." keypress dead-end. Only the spawned windows suffer this; the trigger window self-attaches in-process from a Portal that is a child of the user's login shell, so it lands cleanly at a prompt.

**Solution**: Scoped entirely to the native Ghostty adapter (`internal/spawn/ghostty.go`), wrap the composed open argv as `bash -lc '<composed open argv>; exec "$SHELL" -il'` so that when the session command finishes the wrapper execs the user's interactive login shell into the window (keeping it visible AND usable), and drop `wait after command` from the osascript since the exec'd shell — not the flag — now keeps the window alive. Build the wrapper as a real 3-element argv `["bash", "-lc", <payload>]` and render it through the existing shared shell-quote helper (`renderCommandString`) so the inner per-element single-quotes nest correctly via the POSIX `'\''` close-escape-reopen idiom rather than naive string concatenation. Both burst entry points (picker multi-select and `portal open` multi-target) benefit automatically because they share this adapter.

**Outcome**: `ghosttyOpenScript(command)` emits a `new window with configuration {command:"…"}` whose `command` is the AppleScript-escaped rendering of `bash -lc '<composed open argv>; exec "$SHELL" -il'`, carries no `wait after command`, preserves the `/usr/bin/env … PATH=<…> -u TMUX -u TMUX_PANE` prefix inside the wrapper, and round-trips a quote-sensitive argv element (single quote, `;`, `$`, `"`) uncorrupted through the added `bash -lc` layer's `'\''` double-escaping. `composeOpenArgv`, `renderCommandString`, `shellQuote`, the `syscall.Exec` attach path, ack ordering, and the trigger path are all unchanged, and `go test ./...` is green.

**Do**:
- In `internal/spawn/ghostty.go`, add a pure helper `wrapWithShellFallback(command []string) []string` that returns the 3-element argv `["bash", "-lc", <payload>]`, where `<payload>` is the string `renderCommandString(command)` followed by the literal suffix `; exec "$SHELL" -il`. Do NOT single-quote or otherwise pre-escape the payload here — hand the raw 3-element argv to the shared helper (next step), which owns the nesting. Document that this wraps the composed open argv so the window lands in an interactive login shell when the session's exec chain finishes.
- Change `ghosttyEmbed` so its first line renders the WRAPPED argv instead of the bare command: replace `embedded := renderCommandString(command)` with `embedded := renderCommandString(wrapWithShellFallback(command))`. Leave the two AppleScript-escape lines (`\` → `\\` first, then `"` → `\"`) unchanged — the backslash-before-quote order stays load-bearing, now also over the extra backslashes the wrapper's second `'\''` layer introduces.
- Drop `wait after command:true` from `ghosttyScriptTemplate` so the record literal is `new window with configuration {command:"%s"}` (single `command` field). Update the template's doc comment: the record now carries only `command`, and the exec'd fallback shell — not `wait after command` — keeps the window visible after the session exits.
- Update `ghosttyEmbed`'s doc comment to describe the new wrapper layer: the composed argv is first `renderCommandString`-rendered, appended with `; exec "$SHELL" -il`, wrapped as `["bash","-lc",<payload>]`, then `renderCommandString`-rendered again (so inner per-element single-quotes nest via `'\''`) before the AppleScript escape.

No behavioural change to that test is required.

**Edge Cases**:
- Quote-sensitive argv element (single quote, `;`, `$`, `"`) delivered via a mint `-- <command…>` passthrough: it must round-trip byte-identically through the added `bash -lc` layer. The inner `renderCommandString` single-quotes the element (single quote → `'\''`); the outer `renderCommandString` on the `["bash","-lc",payload]` argv re-single-quotes the payload, so every `'` (including those inside the inner `'\''`) is re-escaped, and the AppleScript escape then doubles the introduced backslashes (`'\''` → `'\\''`, the same doubling the existing `it's` test documents, now one layer deeper). Naive concatenation of the schematic `bash -lc '<argv>; exec …'` byte template would let the first inner `'` terminate the outer quote and corrupt the command — the exact failure class this fix removes.
- Mint surface (`--path <dir>`) is wrapped identically to attach (`--session <name>`): the wrap is argv-agnostic — it lives at the adapter and wraps whatever argv it is handed — so mint windows (and any `-- <command…>` passthrough element) get the same fallback shell.
- Degenerate `exec "$SHELL"` failure: not specified to be guarded — if `exec "$SHELL"` ever failed, the inner bash exits with nothing left to run and the window closes cleanly (an accepted fallback), with no "Press any key" dead-end reappearing. No `$SHELL` fallback is added.

**Context**:
> The fix mechanism was already validated live during investigation (sandboxed `-L` tmux socket): the explicit `bash -lc '<composed open argv>; exec "$SHELL" -il'` wrapper landed at the user's normal interactive login shell (`$SHELL`, login+interactive) after the session was killed — not the dead-end. So no manual-validation deliverable is gated by this task; only the code change and its unit coverage remain.
>
> Why the EXPLICIT wrapper (not implicit-append): Ghostty runs a window's `command` by prepending `exec -l` (effectively `/usr/bin/login -flp <user> /bin/bash --noprofile --norc -c "exec -l <command>"`). `exec -l` replaces the outer bash with the command's FIRST TOKEN, so an implicit-append form `<argv>; exec "$SHELL" -il` would make the `; exec` fallback unreachable (and fatal if the first token weren't found). The explicit `bash -lc '…'` form works because Ghostty's `exec -l bash -lc '…'` replaces the outer bash with OUR inner bash, which runs the session command as a child then execs `$SHELL`.
>
> Why the `bash -lc` wrapper is NOT placed in shared composition (`composeOpenArgv`/`renderCommandString`): doing so would inject shell metacharacters (`;`, `exec`) into the `{command}` string handed to config-`terminals.json` adapters, which only work if the terminal runs `{command}` through a shell — a guarantee `terminals.json` does not make. A direct-exec custom terminal would silently break. Scoping the wrap to the Ghostty adapter avoids this; custom terminals stay unchanged (a known, accepted residual: they retain full control of their own recipe's post-command behaviour).
>
> PATH must still be carried: the composed argv keeps its `/usr/bin/env … PATH=<picker PATH> -u TMUX -u TMUX_PANE …` prefix. Ghostty's inner bash runs `--noprofile --norc` with a login default PATH, so a PATH-less command cannot find `tmux`. The wrap must not strip it. `$SHELL` is populated by `/usr/bin/login`, so `exec "$SHELL" -il` is reliable and no `$SHELL` fallback is specified.
>
> Accepted residual (not a defect): closing the window from the idle fallback prompt shows Ghostty's one-click "Close Window?" confirm — intrinsic to landing in a shell Ghostty did not launch itself (missing shell integration), fired even with `wait after command` omitted. Do NOT ship any mitigation (no shell-integration re-injection, no `confirm-close-surface = false`).
>
> Osascript boundary has no automatable lane — coverage stays at the command-composition seam (this task). The `//go:build ghosttycompile` template guard (`ghostty_compile_ghosttycompile_test.go`) still passes after this change: dropping the optional `wait after command` field does not alter the `tell application "Ghostty"` terminology it resolves against Ghostty's dictionary; no change to that test is required.
