## FINDING: Lazy stamp-on-render resolution pass runs in Flat mode, where its output is discarded
- SEVERITY: medium
- FILES: internal/tui/model.go (rebuildSessionList — `filtered := m.resolveSessionDirs(m.filteredSessions())` above the mode switch), internal/session/dirresolve.go:49-76
- DESCRIPTION: `rebuildSessionList` unconditionally calls `resolveSessionDirs(m.filteredSessions())` BEFORE the mode switch. Only ModeByProject / ModeByTag consume Session.Dir; the Flat and byTagSignpost arms route through `ToListItems(filtered)`, which ignores Dir entirely. For any un-stamped session (every live session on first ship, all restored sessions post-reboot) `ResolveAndStampDir` falls into its lazy path and performs an `ActivePaneCurrentPath` read + a `ResolveGitRoot` `git rev-parse` subprocess + a best-effort `SetSessionOption` write — per session, per render — and the resulting Dir is then thrown away because Flat does not group. The spec frames the lazy fallback strictly as a grouped-render mechanism ("when the grouped render encounters a session with no @portal-dir"); applying it in Flat mode is a chokepoint over-application. The cost is the spec's "bounded one-time amortisation" of N git-root derivations + N stamp writes, but incurred for a mode that derives no value from it, and re-incurred on every Flat render until the stamps happen to land. Flat is BOTH the no-regression default AND the first-ever-launch default, so this is the common path. The seam is otherwise clean (value-copy, nil-guarded, best-effort) — the fix is purely a placement guard.
- RECOMMENDATION: Gate the resolution pass on a grouped mode: only call `resolveSessionDirs` when `sessionListMode` is ModeByProject or ModeByTag and byTagSignpost is not active. Reuse the same `grouped := ... ModeByProject || ... ModeByTag` predicate already used at the ProjectsLoadedMsg re-group site, or move the `resolveSessionDirs` call inside the two grouping arms of the switch. Flat / signpost arms consume `m.filteredSessions()` directly, matching the spec's grouped-only contract and keeping Flat byte-for-byte today's behaviour with zero pane reads.

SUMMARY: Architecture is sound overall — single rebuildSessionList chokepoint, setProjects-paired index (no stale-index path), clean small-interface seams (PaneStamper, ModePersister, ProjectEditor, WithDirResolver), tolerant prefs leaf, and the startup-ordering re-group / WithInsideTmux routing all compose well. One remaining seam issue: the lazy stamp/resolve pass is hoisted above the mode switch and fires its pane-read + git rev-parse + stamp-write side effects in Flat mode where the resolved Dir is discarded.

NOT flagged (no action): project.MatchProjectByDir is now production-dead (all callers use Index.Match) but is intentionally retained as the test oracle / canonical-contract reference for Index.Match. SessionItem.Description()/Title() satisfy the list.Item contract. loadPrefsStore/prefsFilePath live in cmd/clean.go (odd placement, harmless).
