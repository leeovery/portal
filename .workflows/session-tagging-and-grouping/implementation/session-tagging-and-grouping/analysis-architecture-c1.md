## FINDING: Lazy stamp-on-render fallback is fully orphaned — never wired into the render path
- SEVERITY: high
- FILES: internal/session/dirstamp.go:48, internal/session/dirresolve.go:49, internal/tui/grouping.go:43, internal/tui/grouping.go:106, internal/tui/model.go (rebuildSessionList), cmd/open.go (tuiConfig)
- DESCRIPTION: Phase 1 produced ResolveAndStampDir (per-session derive-use-then-stamp helper) and ResolveSessionDir (active-pane → git-root resolver), plus PaneStamper / PaneCurrentPathReader seams and a new ActivePaneCurrentPath tmux method — all unit-tested. But no production code calls them. Builders buildByProject/buildByTag consume s.Dir directly; empty Dir routes straight to Unknown/Untagged. rebuildSessionList never resolves un-stamped sessions, and cmd/open.go's tuiConfig never constructs a stamper/resolver dependency. The two halves were each built locally-correct but never joined: tasks 1-7/1-8 assumed "the render-layer entry point calls the lazy resolver", while tasks 2-2/2-3 assumed Session.Dir was "already canonicalised upstream by Phase 1's lazy fallback/restamp" (comment at grouping.go:28). Neither holds — no integrating caller. Consequence: acceptance criterion #9 unmet; every session lacking a stamp (post-reboot restored sessions; any session live when the feature ships) is permanently mis-bucketed into Unknown/Untagged until killed and recreated. Spec's "no restart-to-appear gap" broken.
- RECOMMENDATION: Introduce a render-layer resolution pass invoking ResolveAndStampDir for each session whose Dir is empty, before/inside the grouping builders. Thread a session.PaneStamper + resolver.CommandRunner into the Model (new Option wired in cmd/open.go to *tmux.Client), and in rebuildSessionList (the chokepoint) replace each empty Dir with ResolveAndStampDir(s.Name, s.Dir, stamper, runner) so the derived value is used for this render and best-effort stamped for the next. Keep builders pure (receive already-resolved Session.Dir).

## FINDING: ProjectsLoadedMsg updates the grouping data source but never re-groups — startup-ordering-dependent first paint
- SEVERITY: medium
- FILES: internal/tui/model.go (ProjectsLoadedMsg handler caches m.projects; transitionFromLoading; rebuildSessionList)
- DESCRIPTION: rebuildSessionList groups against cached m.projects, but the ProjectsLoadedMsg handler caches m.projects WITHOUT calling rebuildSessionList, and transitionFromLoading/evaluateDefaultPage don't re-group either. At startup fetchSessions and loadProjects are batched concurrently. If SessionsMsg arrives before ProjectsLoadedMsg (common — session listing is one fast tmux call), applySessions → rebuildSessionList groups against an empty m.projects, so in a persisted By-Project/By-Tag mode every session lands in Unknown/Untagged on first paint and is never corrected until the next session refresh (only preview-dismiss or projects-page round-trip). The render path is not the single chokepoint it claims: it reacts to session-slice changes but is blind to project-slice changes. TestProjectsTransitionRegroupsWithUpdatedTags only passes because it pre-seeds m.projects directly rather than exercising real load→navigate ordering — gap untested.
- RECOMMENDATION: Make the chokepoint reactive to its full input set: have the ProjectsLoadedMsg handler call rebuildSessionList() (batched with the existing setItemsCmd) when sessions are already loaded and a grouped mode is active, so a late project load re-groups the visible list regardless of message arrival order.

SUMMARY: One high-severity cross-phase seam failure — the Phase-1 lazy stamp-on-render fallback is fully orphaned, never invoked by the render path, leaving un-stamped (post-reboot/first-ship) sessions permanently mis-bucketed and acceptance criterion #9 unmet. A second medium ordering gap: ProjectsLoadedMsg updates the grouping data source without re-grouping, so the grouped first paint can render against stale/empty projects depending on startup message order. Both stem from rebuildSessionList not being the true chokepoint over all its inputs (sessions, projects, directory resolution).
