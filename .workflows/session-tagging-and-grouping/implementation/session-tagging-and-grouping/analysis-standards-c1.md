## FINDING: Lazy stamp-on-render fallback is implemented but never wired into the render path
- SEVERITY: high
- FILES: internal/session/dirstamp.go:48, internal/session/dirresolve.go:49, internal/tui/grouping.go:43, internal/tui/grouping.go:106, internal/tui/model.go (rebuildSessionList)
- DESCRIPTION: Spec mandates a lazy stamp-on-render fallback as a core mechanism (§ Session → Directory Resolution → "The lazy stamp-on-render fallback"; Acceptance Criterion 9). When the grouped render encounters a live session with no @portal-dir, it MUST resolve that session's directory from active pane → git-root, group it THIS render, and stamp it for subsequent renders. ResolveAndStampDir / ResolveSessionDir implement exactly this and are unit-tested — but they are dead code: nothing in the TUI render path (rebuildSessionList → buildByProject/buildByTag) ever calls them (grep confirms no production caller outside their own files). Builders consume tmux.Session.Dir verbatim; empty Dir routes straight to Unknown (By Project) / Untagged (By Tag) at grouping.go:48-51 and grouping.go:146-149. Consequences: (1) post-reboot restored sessions return without @portal-dir (it is in-memory tmux state, not persisted in sessions.json) and sit permanently in Unknown/Untagged (breaks AC 9 and "no restart-to-appear gap"); (2) pre-existing live sessions on first ship never grouped; (3) QuickStart-created sessions deliberately skip create-time stamp (quickstart.go) on the explicit promise the lazy fallback re-stamps on first grouped render — but it never runs, so every x / portal open <path> session is mis-bucketed. Only SessionCreator.CreateFromDir-stamped sessions group correctly.
- RECOMMENDATION: Wire ResolveAndStampDir into the render pipeline before the builders consume Session.Dir. Construct the Model with a PaneStamper + resolver.CommandRunner seam (new functional option, mirroring existing seam-injection), and in rebuildSessionList map each session whose Dir is empty through ResolveAndStampDir(s.Name, s.Dir, stamper, runner) before buildByProject/buildByTag, using the derived directory for this render. Add an integration test for an empty-Dir session that resolves via active-pane fallback and appears under its project (not Unknown). Update grouping_test "empty Dir → Unknown" cases so Unknown is only reached when the fallback ALSO fails (no readable current_path / unresolvable), per dirstamp.go's contract.

## FINDING: Remove-then-re-add of the same tag within one edit-modal session loses the tag
- SEVERITY: medium
- FILES: internal/tui/model.go (handleEditProjectConfirm tag persist; editRemovedTags handling; additions diff !slices.Contains(m.editProject.Tags, tag))
- DESCRIPTION: In a single open of the projects edit modal, if the user highlights an existing tag and presses x (appends to editRemovedTags, removes from editTags) then re-adds the same tag via Add input + Enter (editTags gets it back; editRemovedTags still holds it), handleEditProjectConfirm executes removals THEN additions: RemoveTag(tag) runs, but the addition loop skips it because the diff !slices.Contains(m.editProject.Tags, tag) is false (tag was in the originally-loaded set). Net: tag removed from projects.json even though the user re-added it before saving — silent data loss.
- RECOMMENDATION: Reconcile the two buffers before persisting: compute removedTags := editRemovedTags MINUS final editTags (or drop the tag from editRemovedTags on re-add), so the persisted delta reflects (original → final editTags). Add a modal test for remove-then-readd-same-tag asserting the tag survives.
