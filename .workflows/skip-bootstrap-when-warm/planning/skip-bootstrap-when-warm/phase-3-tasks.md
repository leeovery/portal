### Task skip-bootstrap-when-warm-3-3: Place the cleanup gate on the tick idle branch

**Problem**: The throttled gate helper (task 3-2) must actually fire on a mostly-idle warm server — the weeks-long scenario the feature targets. The daemon's `tick` (`cmd/state_daemon.go` ~lines 336-362) has three early returns in order: (1) `@portal-restoring` set → return; (2) `!dirty && !gap` idle fast-path → bare `return` (~lines 348-350); (3) `captureAndCommit`. The gate MUST be placed on the **idle branch** (replacing the bare idle `return`). Placing it after the idle return would gate cleanup behind capture work → it would NEVER run on an idle server (the opposite of the goal), and placing it before the `@portal-restoring` check would let cleanup run during a restore window (it must not). This is a load-bearing placement decision.

**Solution**: In `tick`, replace the bare idle-branch `return` (the `if !dirty && !gap { return }` body ~lines 348-350) with "call `maybeRunHookCleanup(deps)` then `return`". This runs cleanup only on idle ticks, after the `@portal-restoring` early-return and at the `!dirty && !gap` point — so a restoring tick skips the whole tick (no cleanup), a capture-pending tick (`dirty || gap`) runs capture and never reaches the idle branch (cleanup skipped, scrollback wins), and an idle tick with the throttle elapsed runs cleanup then returns. The daemon is single-threaded so cleanup never competes with a pending capture. Also assert here that the daemon is the ONLY remaining automatic hooks-`CleanStale` caller (bootstrap no longer runs it after Phase 1 task 1-3).

**Outcome**: On an idle tick (`!dirty && !gap`, and `@portal-restoring` not set), `tick` calls `maybeRunHookCleanup(deps)` before returning; on a restoring tick the whole tick is skipped (no cleanup); on a capture-pending tick capture runs and cleanup is skipped this tick; the daemon is the sole automatic caller of hooks-`CleanStale` (bootstrap's former step 11 is gone).
