## skip-bootstrap-when-warm-2-1 | approved

### Task skip-bootstrap-when-warm-2-1: Liveness-only EnsureSaver helper in package cmd

**Problem**: The abridged bootstrap path (Class 2 — protective liveness) must keep the `_portal-saver` daemon alive across a weeks-long warm server lifetime without running the full orchestrator. On an already-bootstrapped server the daemon can still die mid-lifetime (a self-eject `os.Exit(0)` tears down its pane and kills the `_portal-saver` session), so every warm command must re-probe and revive it. The existing `saverAdapter.EnsureSaver` (`cmd/bootstrap_production.go`) delegates to `tmux.EnsurePortalSaverVersion` — which runs the version-gate kill-barrier — but a *satisfied* latch already proves the running daemon is the current binary, making the version re-check redundant and its kill-barrier a needless concurrency hazard under the reopen burst. There is no liveness-only saver helper in package `cmd` today.

**Solution**: Add a new liveness-only helper function in package `cmd` (a new file `cmd/abridged_saver.go`) that composes the existing primitives directly: probe `_portal-saver` presence via `tmux.SaverPanePIDOrAbsent`, and if the saver is absent (or the probe returns a transient error, treated as absent) re-ensure it via `tmux.BootstrapPortalSaver`. On a revive failure it constructs a `bootstrap.SaverDownWarning()` and feeds it into the package-level `bootstrapWarnings` sink (identical to the sync path's warning plumbing), then returns without error so the command proceeds. It **never** calls `tmux.EnsurePortalSaverVersion` — the version-gate lives solely in the full-bootstrap orchestrator step.

**Outcome**: A function `ensureSaverLiveness(client *tmux.Client, stateDir string)` exists in package `cmd` that: (a) does nothing when the saver pane is present and alive; (b) calls `BootstrapPortalSaver` to revive an absent saver; (c) on a revive error adds a `SaverDownWarning` to `bootstrapWarnings` and returns non-fatally; (d) treats a `SaverPanePIDOrAbsent` transient error as absent and attempts a revive; and (e) never invokes `EnsurePortalSaverVersion` or any kill-barrier.

Document it as the abridged path's liveness-only EnsureSaver (Class 2), contrasting it against `saverAdapter.EnsureSaver` (which keeps the version-gate) — call out explicitly that this helper MUST NOT call `EnsurePortalSaverVersion`.
- Body: call `_, present, err := tmux.SaverPanePIDOrAbsent(client, tmux.PortalSaverName)`. Treat `present == true && err == nil` as alive → return immediately (no revive, no warning). Treat every other shape — `!present` (absent) OR a non-nil `err` (transient probe error) — as "needs revive" (per spec §Abridged EnsureSaver: a transient error folds into "attempt revive"; mirror Component D's "treat any error as absent" collapse for the liveness case).

Do not return the error; do not append anything else. `BootstrapPortalSaver` is the correct primitive: it is idempotent, probes `HasSession` + `BootstrapAliveCheck` internally, kills a lingering-dead saver via its own synchronous barrier, and (re)creates + respawns the daemon — but it never touches version-marker logic (see its godoc: "Does not touch @portal-restoring or version-marker logic").
