- FINDING: Latch value-format contract split across two layers (read helper in state, inline write in bootstrap)
  SEVERITY: low
  FILES: internal/state/markers.go:195, cmd/bootstrap/bootstrap.go:497-501
  DESCRIPTION: The `@portal-bootstrapped` latch has an asymmetric produce/consume shape. The READ side is a named domain helper — `state.BootstrappedLatchSatisfied(c, runningVersion)` — that owns the satisfaction semantics (present AND `stored == runningVersion`) and lives beside the `BootstrappedMarkerName` constant. The WRITE side is a raw call inlined in the orchestrator: `o.Latch.SetServerOption(state.BootstrappedMarkerName, o.Version)`. So the value-format contract ("the stored value is exactly cmd.version") is encoded in two packages: the equality lives in `internal/state`, the "store the bare version" lives in `cmd/bootstrap`. The spec explicitly notes a future format change (forensic extras — timestamp/pid) would require a delimiter + parse-and-compare rule so equality "could no longer be naive"; that change would have to touch both the state-side equality and the bootstrap-side write, in different packages, to stay consistent. This also breaks the codebase's own convention in the very same file: skeleton markers colocate produce+consume (`SetSkeletonMarker` / `ListSkeletonMarkers` both live in `state`, both key off the shared prefix). The latch alone puts its writer outside `state`. No current bug (the shared constant prevents name drift and v1 is trivially parse-free) — this is a latent-maintenance / mirror-layer concern.
  RECOMMENDATION: Add a `state.StampBootstrappedLatch(w state.ServerOptionWriter, version string) error` beside `BootstrappedLatchSatisfied` (mirroring `SetSkeletonMarker` beside `ListSkeletonMarkers`), and have the orchestrator call it via the existing seam. This colocates the latch's value produce+consume in the domain package so a future format change is a single-site edit, and lets `bootstrap.LatchWriter` collapse onto (or delegate to) the state-owned writer.

- FINDING: Daemon's critical capture path coupled to the best-effort hooks-cleanup dependency (fatal at wiring, soft at runtime)
  SEVERITY: low
  FILES: cmd/state_daemon.go:698-701, cmd/state_daemon.go:413-421
  DESCRIPTION: The daemon gained hooks stale-cleanup as a new, explicitly best-effort responsibility whose runtime posture is "log WARN and swallow, never crash the daemon" (`maybeRunHookCleanup` swallows every cleanup error). But its dependency wiring is fatal: `loadHookStore()` failing in the daemon `RunE` returns an error that aborts daemon startup before the tick loop begins — so an unresolvable hooks-store path takes down the daemon's PRIMARY responsibility (scrollback capture + resurrection state), not just the inert cleanup feature. This is a posture inconsistency (fatal wiring vs soft runtime for the same non-critical subsystem) and couples capture availability to a secondary feature's dependency. The comment's stated intent — "surface rather than silently disable cleanup" — is satisfiable without killing capture. The practical trigger is low-probability (the config path shares XDG resolution with `state.EnsureDir()`, which runs first at the top of `RunE` and would already have aborted), which is why this is low severity rather than higher — but the structural coupling is real.
  RECOMMENDATION: Degrade cleanup rather than the daemon: on `loadHookStore()` failure, log one WARN (so the disabled-cleanup state is observable per the comment's intent) and proceed with a nil/absent store, having `maybeRunHookCleanup` no-op when the store is unavailable — keeping capture (the daemon's reason for existing) independent of the cleanup feature's wiring, and making the wiring posture consistent with the runtime "never crash the daemon" posture.

SUMMARY: The feature composes cleanly — the 11→10 orchestrator surgery is complete with no CleanStale residue, the abridged path reuses the sync-plumbing seams correctly, the single-read latch invariant is tight, and the daemon-owned hooks cleanup is a genuine single-home reuse of `runHookStaleCleanup` with strong test coverage (branch selection, set-point gating, self-heal, cadence, idle-branch placement). Two low-severity structural observations: the latch's read/write value contract is split across `state` and `bootstrap` (breaking the same file's colocation convention for markers), and the daemon's critical capture path is now fatally coupled to the best-effort hooks-cleanup store resolution.
