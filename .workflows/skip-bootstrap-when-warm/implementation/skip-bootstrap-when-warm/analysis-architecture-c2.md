- FINDING: runHookStaleCleanup over-parameterised after step-11 removal — dead policy axis + stale contract
  SEVERITY: medium
  FILES: cmd/run_hook_stale_cleanup.go:1-133, cmd/state_daemon.go:426, cmd/clean.go:155-163
  DESCRIPTION: This feature removed bootstrap step 11 (the CleanStale hooks step + its cleanStaleAdapter) and re-homed hooks cleanup on the daemon. That removal deleted the ONLY caller of runHookStaleCleanup that passed swallowListError=false. The two surviving production callers — the daemon's maybeRunHookCleanup (state_daemon.go:426) and portal-clean (clean.go:159) — both pass true. The boolean policy axis the helper is built around therefore no longer varies in production: its `false` short-circuit (`return err`) is unreachable from any production path and is exercised only by run_hook_stale_cleanup_test.go passing itself `false`. The helper was justified as "one source of truth collapsing three sites with two policy variants"; the feature collapsed it to a single variant, leaving a boolean parameter (a flagged anti-pattern) and a dead error-return branch as vestigial complexity. Compounding this, the helper's own contract documentation still describes its callers as "bootstrap step 11 (cleanStaleAdapter.CleanStale)" and "portal-clean RunE" and frames the swallowListError axis as "the bootstrap adapter passes false ... escalates the err up through the StaleCleaner interface" (lines 6-22, 61-77) — a caller and interface that no longer exist. The daemon, now one of the two live consumers of this load-bearing shared seam, is not mentioned anywhere in the helper's contract. A maintainer reading the seam is pointed at removed code and left unaware of the daemon dependency.
  RECOMMENDATION: Simplify the seam to match its remaining usage: drop the swallowListError parameter (both live callers want swallow=true — always log-Warn-and-return-nil on a ListAllPanes error), delete the now-dead `return err` branch, and update the contract doc to name the two current callers (daemon maybeRunHookCleanup + portal clean) instead of the removed step-11 adapter. Retune/retire the run_hook_stale_cleanup_test.go cases that pin the false branch. If retaining the parameter is preferred for the manual-audit path, at minimum correct the contract doc so it no longer describes removed code and records the daemon as a consumer.

- FINDING: RestoringChecker seam now serves the bootstrap latch but its name/doc still describe only the restoring marker
  SEVERITY: low
  FILES: internal/state/markers.go:42-53, internal/state/markers.go:195-204
  DESCRIPTION: BootstrappedLatchSatisfied cleanly reuses the existing TryGetServerOption seam for its read (good composition — the latch read and the restoring read are the same shape). But the seam it depends on, RestoringChecker, is named for and documented as "the seam used by IsRestoringSet"; it is now equally the seam for the version-latch read. The interface is really a generic single-method server-option reader, so a name and doc tied to one of its two use-sites under-describes its role and can mislead a reader into thinking the latch read shares restoring-marker semantics. This is a seam-clarity smell, not a correctness issue.
  RECOMMENDATION: Either rename the interface to a use-site-agnostic form (e.g. ServerOptionReader) and update both IsRestoringSet and BootstrappedLatchSatisfied to reference it, or leave the name and extend its doc comment to state it now backs both the @portal-restoring read and the @portal-bootstrapped latch read.

SUMMARY: The latch/abridged/daemon-cleanup design composes soundly overall — the latch reuses the marker read seam, the abridged path is a thin well-scoped helper, and the daemon reuses the shared cleanup body with no new cycle. The one substantive residue is that removing bootstrap step 11 left the shared runHookStaleCleanup helper over-parameterised (a now-single-valued boolean policy axis with a production-dead branch) and its contract doc pointing at removed code while omitting its new daemon caller; a minor seam-naming inaccuracy on RestoringChecker is the only other note.
