- FINDING: Bootstrap-latch seam is asymmetric — the read is a self-contained domain helper but the write is a raw server-option call, splitting the value-format contract across two packages
  SEVERITY: low
  FILES: internal/state/markers.go:166-204, cmd/bootstrap/bootstrap.go:497-501, internal/state/markers.go:119-132
  DESCRIPTION: The @portal-bootstrapped latch has an encapsulated READ side (state.BootstrappedLatchSatisfied owns the marker name AND the "satisfied == naive string-equality against the running version" contract, documented at length in its own docstring), but no encapsulated WRITE side. The write is an inline o.Latch.SetServerOption(state.BootstrappedMarkerName, o.Version) in the orchestrator. So the load-bearing value-format invariant ("the stored value is EXACTLY the version; satisfaction is parse-free equality") is authored in two packages that must silently agree: internal/state owns the compare, cmd/bootstrap owns the value it writes. This is exactly the "layers that mirror each other when the domain type could own the transformation / correctness depends on caller discipline rather than being self-contained" shape. Its own sibling markers in the same file are symmetric and do NOT have this split: @portal-skeleton-* has both SetSkeletonMarker (write) and ListSkeletonMarkers (read) colocated with the prefix constant, and @portal-restoring's write is behind the RestoringMarker.Set/Clear adapter. The latch is the only marker whose read is a domain helper while its write is raw. The v1 format is pinned to a bare version (so no drift exists today), but the spec explicitly *defers* rather than rejects forensic extras (set-timestamp/pid) — the moment a future value gains a delimiter, the writer and the equality check must be changed in lockstep across two packages with nothing structurally binding them.
  RECOMMENDATION: Add a symmetric state.StampBootstrappedLatch(w state.ServerOptionWriter, version string) error colocated with BootstrappedLatchSatisfied in markers.go, so both halves of the value-format contract live in one file and cannot drift. The orchestrator's Latch seam already satisfies state.ServerOptionWriter (it is *tmux.Client), so the Run site becomes state.StampBootstrappedLatch(o.Latch, o.Version) with the WARN guard unchanged. This mirrors the existing SetSkeletonMarker/ListSkeletonMarkers pairing and keeps the "value is exactly the version" rule self-contained.

- FINDING: RestoringChecker interface name is use-site-specific and now misleads at the latch read site (persists from cycle 2, still unaddressed)
  SEVERITY: low
  FILES: internal/state/markers.go:40-53, internal/state/markers.go:195-196
  DESCRIPTION: BootstrappedLatchSatisfied reads the @portal-bootstrapped latch through the RestoringChecker interface — an interface still named for, and doc-commented as, "the seam used by IsRestoringSet." Its actual shape is a generic single-method server-option reader (TryGetServerOption), and it now has two consumers reading two different markers. Its own siblings in the same file are already role-agnostic (ServerOptionLister, ServerOptionWriter), so RestoringChecker is the odd one out — a use-site name on a role that has outgrown that use-site. This was raised in analysis-architecture-c2 (low) and confirmed unresolved here; it composes directly with the write-symmetry finding above (both are the latch seam under-owning its role in internal/state). Aligns with the project's own "name by intrinsic role, not use-site" convention.
  RECOMMENDATION: Rename RestoringChecker to a role-agnostic ServerOptionReader (matching ServerOptionLister / ServerOptionWriter) and point both IsRestoringSet and BootstrappedLatchSatisfied at it; or, at minimum, extend the doc comment to state it now backs both the @portal-restoring read and the @portal-bootstrapped latch read.
SUMMARY: The latch / abridged-path / daemon-cleanup design composes soundly: the abridged path is a thin well-scoped cmd helper that reuses the sync warning-sink + context-injection plumbing, the daemon reuses the shared runHookStaleCleanup body (now cleanly 4-arg, no boolean — the c2 over-parameterisation is resolved) with correct idle-branch tick placement, and branch selection / set-point gating / self-heal / cadence are all well covered by unit and real-tmux integration tests. The only residue is that the @portal-bootstrapped latch under-owns its role in internal/state — its read is an encapsulated domain helper but its write is a raw cross-package server-option call, and the reader interface it depends on is still named for the older restoring use-site. Both are low-severity seam-symmetry/naming issues, not correctness defects.
