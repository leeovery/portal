AGENT: architecture
FINDINGS: none
SUMMARY: Implementation architecture is sound. The version-stamped latch, abridged path, and daemon-owned hooks cleanup compose cleanly over existing primitives with narrow, well-scoped seams. Reviewed for the concerns below and judged not worth actioning on a fully-implemented, reviewed, green feature.

NOTES (reviewed, not flagged):
- Seam quality: latch read (state.BootstrappedLatchSatisfied via the RestoringChecker seam) and latch write (Orchestrator.Latch, a single-method LatchWriter satisfied by *tmux.Client) are both narrow and role-named, consistent with the codebase's existing generic-method/role-name convention (ServerOptionLister/Writer). The RestoringChecker reuse and the read/write ownership asymmetry were surfaced and accepted in prior cycles; no materially stronger case exists — in v1 both sides are a parse-free bare-version string, and forensic extras are explicitly out of scope.
- Composition: the abridged EnsureSaver (ensureSaverLiveness) composes SaverPanePIDOrAbsent + BootstrapPortalSaver directly rather than reinventing, and deliberately never touches the version-gate kill-barrier (source-guarded by test). The daemon-owned cleanup reuses the shared runHookStaleCleanup verbatim (mass-deletion guard + EmitCleanStaleSummary audit breadcrumb), inventing no new audit event.
- CleanStale (hooks) removal is complete: the step, its seam, and its adapter are gone; only CleanStaleMarkers (a distinct marker concern, step 9) remains. totalSteps=10, totalBootstrapSteps=10, and the stepLabelTable (contiguous 1..10) all move in lockstep with drift-guard tests.
- 4-arg runHookStaleCleanup (dropped swallowListError vs the spec's pinned 5-arg) is correct composition, not drift: both remaining callers (daemon, portal clean) want swallow=true, so the dead policy parameter was correctly removed rather than carried.
- The context-injection + warning-drain plumbing recurs across the three PersistentPreRunE branches (abridged / concurrent-deferred / synchronous-full), but each block injects a meaningfully different key/value set (serverStartedKey vs deferredBootstrapKey; conditional vs unconditional tmuxClientKey). This is a 2-3 instance, semantically-divergent pattern; extracting a shared helper would require a flag-laden signature that reads worse than the explicit branches. Proportional to leave.
- shouldRunConcurrentBootstrap threading a provably-constant latchSatisfied argument (the abridged branch returns upstream on satisfied) is a deliberate, documented choice to preserve the single-read invariant and keep the decider self-describing. Harmless.
- Integration coverage is thorough: all four outcome-matrix rows (open/CLI x satisfied/not-satisfied) plus abridged self-heal (revive killed saver, skip restore), version-mismatch full re-bootstrap + re-stamp, real-daemon throttled idle-cadence cleanup, and tick-placement skip cases (@portal-restoring set, dirty/gap capture-pending) are all covered under IsolateStateForTest.
