- FINDING: Register-set and teardown-set are two independent enumerations of "Portal-managed events" with no shared source of truth
  SEVERITY: medium
  FILES: internal/tmux/hooks_register.go:45-55, internal/tmux/hooks_register.go:60-82, internal/tmux/hooks_unregister.go:52
  DESCRIPTION: The convergence engine iterates managedEvents (a hand-written 9-entry table). Teardown iterates portalEvents, derived as slices.Concat(saveTriggerEvents, HydrationTriggerEvents). These are two parallel, independently-authored representations of the same domain fact — which tmux events Portal manages — with the 8 non-session-closed event names duplicated as raw string literals across both. Nothing in production derives one set from the other, and no test asserts the two event-sets are equal (no parity/superset assertion exists between managedEvents and portalEvents). This is exactly the "managed-event / portalEvents set mismatch that would leave an event unmanaged" seam the task flagged. The failure mode is latent but real: adding an event to managedEvents (registration installs a hook) without adding it to the save/hydration slices means teardown silently never reaps it — portal hooks reset leaves it stacked, re-creating the "registration and teardown disagree about the managed set" bug class this spec exists to eliminate. The inverse leaves an event torn down but never converged. The engine unified the seam and the matching primitives across register/teardown; the event-set is the one piece left un-unified, and it is the piece most likely to drift on a future extension.
  RECOMMENDATION: Derive portalEvents by projecting the event field out of managedEvents rather than re-concatenating saveTriggerEvents + HydrationTriggerEvents, making managedEvents the single source of truth for the event-set. The intentional, documented fingerprint-set divergence (teardown's portalCommandSubstrings includes migrate-rename; registration's does not) is orthogonal to the event-set and must be preserved. If full unification is out of this bugfix's scope, at minimum add a test asserting the set of me.event values in managedEvents equals portalEvents, so drift fails a test rather than a user's hook table.
