- FINDING: Convergence reads a whole-table parser then indexes by a single event — the read seam and the parse seam disagree on cardinality
  SEVERITY: low
  FILES: internal/tmux/hooks_register.go:206, internal/tmux/hooks_unregister.go:100, internal/tmux/hooks_parse.go:42
  DESCRIPTION: After the per-event migration, every production caller of ParseShowHooks now reads exactly one event (ShowGlobalHooksForEvent(event)) and immediately collapses the result with ParseShowHooks(raw)[event]. The parser still returns map[string][]HookEntry — a multi-event shape — even though the input is now guaranteed single-event. This works because real tmux scopes show-hooks -g <event> to one event and the no-arg blind form is deleted, but it leaves a latent foot-gun: the [event] index silently returns nil if the raw output ever keys under a different name than the argument passed (a tmux alias, a future event-name canonicalization), turning a real entry into "absent" with no error — the exact silent-absence failure-class this whole fix exists to eliminate. The cardinality contract is no longer expressed in the type, and the ParseShowHooks(raw)[event] + fingerprint-filter dance is re-implemented in three places (convergeEvent, UnregisterPortalHooks via portalEntriesFor, and the test helpers).
  RECOMMENDATION: Consider a thin unexported parseEventEntries(raw, event) []HookEntry helper in the tmux package wrapping ParseShowHooks(raw)[event], giving the single-event collapse one home and one place to add a future "parsed key != requested event" guard. Low priority — current code is correct against tmux 3.6b; the spec explicitly mandates zero changes to ParseShowHooks itself, and this leaves it untouched.
- FINDING: WARN-sink asymmetry between the two halves of the per-event seam (injected logger vs package-level bootstrapLogger)
  SEVERITY: low
  FILES: internal/tmux/hooks_register.go:200, internal/tmux/hooks_unregister.go:17, internal/tmux/hooks_unregister.go:95
  DESCRIPTION: The fix moved registration AND teardown onto the identical per-event seam with a deliberately identical canonical WARN shape ("show-hooks failed", error_class=unexpected). But the two paths source that WARN from different loggers: convergeEvent uses the caller-injected *slog.Logger (testable via the recording logger, component bound by the caller), while UnregisterPortalHooks emits through a package-level bootstrapLogger = log.For("bootstrap") because its exported signature func(*Client) error carries no logger param (comment at hooks_unregister.go:13 frames this as the function-value constraint from cmd/state_cleanup.go). Consequence: the teardown WARN is structurally untestable through the same seam the registration WARN is tested through — the real-tmux teardown test (TestUnregisterPortalHooks_ReapsAtDepth...) explicitly notes it cannot assert the WARN and falls back to tmux end-state only. Two paths sharing one canonical WARN contract but only one being injection-testable is a seam inconsistency that lets the teardown WARN shape drift from the registration WARN shape undetected.
  RECOMMENDATION: Either (a) accept the asymmetry but add a one-line comment in convergeEvent cross-referencing bootstrapLogger as the deliberate teardown counterpart so future edits keep both WARN shapes in lockstep, or (b) give UnregisterPortalHooks an injected-logger variant (retaining the no-logger wrapper as the function-value adapter) so both WARN emissions are exercised by the same recording-logger seam. Low priority — the shapes match today and the register side is unit-pinned.
