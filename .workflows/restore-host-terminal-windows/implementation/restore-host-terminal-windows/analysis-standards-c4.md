- FINDING: Spawn-failure/permission flash co-renders two rows, deviating from the spec's "single slot, highest wins" notice-band precedence
  SEVERITY: low
  FILES: internal/tui/notice_band.go:347-363, internal/tui/model.go:4690-4742, internal/tui/burst_partial_failure.go:34-79
  DESCRIPTION: The spec (§ Mode affordance → "Notice-band precedence (single slot, highest wins)") repeatedly frames the notice band as a single arbitrated slot where the highest-priority claimant wins and every lower one steps aside. The implementation realises this precedence across TWO physical rows (the ▌-barred §11 band under the title separator + the section-header row), and deliberately lets the spawn-failure / permission flash CO-RENDER with the `N selected` multi-select banner (activeNoticeBand takes the §11 slot regardless of m.multiSelectMode). Its sibling in the SAME spec flash tier — the pre-flight abort flash — instead follows single-slot semantics (applySectionHeader's abortBannerText branch REPLACES the banner). So within one spec flash tier ("transient error/guidance flash: pre-flight abort / spawn-failure / permission") two of the three flashes co-render while one replaces — an intra-tier inconsistency and a divergence from the explicit, repeated "single slot" language. This is documented as a deliberate decision recorded during implementation analysis (2026-07-14), and is arguably better UX (the user sees the failure AND that they remain in multi-select with the retry set marked), but it was decided at implementation time rather than in the spec, and the spec text says single-slot. Surfacing so the divergence is a conscious, visible acceptance rather than a silent drift.
  RECOMMENDATION: Confirm the two-row co-render is the intended reading of "single slot, highest wins" and, if kept, align the pre-flight abort flash to the same treatment (or vice-versa) so all three flashes in the tier behave consistently; otherwise make the spawn-failure/permission flash replace the multi-select banner to match the literal spec. No code change required if the recorded decision is ratified — the in-code note already flags it.
- FINDING: Picker permission path omits the per-window DEBUG the CLI emits, a small observability-parity drift in the "one service, two callers" model
  SEVERITY: low
  FILES: internal/tui/burst_partial_failure.go:43-44, internal/tui/burst_observability.go:46-54, cmd/spawn.go:187-194
  DESCRIPTION: The spec's Observability section is built on "one service, two callers" with a single closed event catalog, and the implementation centralises the emission shapes in internal/spawn/logemit.go precisely so "the two paths cannot drift." On the permission-required outcome the CLI (cmd/spawn.go) emits spawn.LogWindowResults (per-window DEBUG for every external window) before the permission INFO, but the picker (the dominant production path) routes only to emitPermission and skips the per-window DEBUG entirely. The spec's closed event catalog lists "per-window spawn + ack outcome" and "permission-required" as distinct events; on a permission wall the CLI captures which windows confirmed before the wall while the picker does not, so the dominant path loses that forensic detail. The asymmetry is documented in-code as intentional ("the picker's permission path deliberately does NOT call it"), but it is a genuine (if minor) divergence from the observability-parity intent the shared logemit.go design exists to protect.
  RECOMMENDATION: Have the picker's permission arm also emit spawn.LogWindowResults(results) before emitPermission so the per-window DEBUG detail is present on both callers' permission paths (matching the CLI), preserving the "two paths cannot drift" contract; or, if the omission is intended, record the rationale in the spec rather than only in code.
