- FINDING: Vestigial burst-outcome state on an already-large Model
  SEVERITY: medium
  FILES: internal/tui/model.go:502-505, internal/tui/burst_progress.go:264-265, internal/tui/burst_progress.go:477-478
  DESCRIPTION: Four of the ~13 burst-lifecycle fields added to Model are dead. `burstBatch` and `burstResults` are only ever assigned their zero values in resetBurstState (`m.burstBatch = ""`, `m.burstResults = nil`) — they are NEVER assigned a real value anywhere. `burstIdentity`/`burstResolution` are written once in dispatchBurst but never read: the terminal-outcome path (the spawnCompleteMsg arm, burstAllConfirmed, emitBurstSummary) reads the outcome from the tea.Msg (`msg.Batch/Results/Identity/Resolution`) and from `len(m.burstExternal)`, never from these model fields. So the burst outcome has two candidate homes (the message and the model fields) and only the message is live. The model doc comment even describes them as active ("burstIdentity/burstResolution snapshot the resolved terminal", "burstDone/burstBatch/burstResults ... (task 6-4/6-6)"), which makes the trap worse — a future reader wiring logic to `m.burstResolution` would read stale/empty state that is only ever set at dispatch and never maintained. This is latent state on a god-object Model that already gained ~20 fields for this feature.
  RECOMMENDATION: Delete the four unused fields (burstBatch, burstResults, burstIdentity, burstResolution) and their reset lines; the outcome already travels correctly on spawnCompleteMsg. While there, consider extracting the remaining ~9 cohesive burst fields (burstPending/burstPipe/burstCancel/burstTrigger/burstExternal/burstTotal/burstDone/burstCancelled/pendingBurstEnter) into a single `burstState` struct on Model so this sub-state-machine is one addressable unit and the "which fields are live" question is answered by the struct, not by grep.

- FINDING: The spawn-burst orchestration SEQUENCE is reimplemented in the picker and kept in lockstep with the CLI only by discipline and cross-referencing comments
  SEVERITY: medium
  FILES: cmd/spawn.go:115-214, internal/tui/burst_progress.go:404-438, internal/tui/burst_progress.go:188-206, internal/tui/model.go:2486-2517
  DESCRIPTION: The leaf primitives are shared cleanly (PreflightMissing, PartitionResults, FirstPermission, the message renderers, the log emitters, the Burster itself) — that composition is genuinely good. But the control-flow SEQUENCE that stitches them — preflight-before-unsupported-gate, permission-before-generic-partial-failure, clean-markers-on-every-post-burst-path, count total=N-not-len(results) — is authored twice: once linearly in cmd/spawn.go's runSpawn, and once split across the TUI's decideBurst + burstRunner.run + the spawnCompleteMsg arm. The invariants live in prose that cross-references the other path ("mirroring cmd/spawn.go's runSpawn ordering (pre-flight FIRST, then the N>=2 unsupported gate)", "matching cmd/spawn.go's logSpawnPermission branch"). The §8-3 work that had to ADD a preflight pass to decideBurst so the unsupported path would "mirror cmd/spawn.go's preflight-then-unsupported sequence" is direct evidence that these two orchestrations already drifted once and were re-synced by hand. Correctness of the two paths staying identical rests on a maintainer noticing every ordering comment — exactly the "correctness depends on caller discipline rather than being self-contained" hazard.
  RECOMMENDATION: The sync/async split makes a single shared orchestrator impractical, so don't force one. Instead capture the ordering invariants as shared, named, testable units the two callers both consume: e.g. a small `spawn.ClassifyBurst(results) -> {permission | partial | allConfirmed}` decision helper (so the permission-before-partial precedence is defined once, not re-branched in both runSpawn and handleBurstPartialFailure), and a documented single ordering contract for preflight-vs-unsupported that both call sites reference by symbol rather than by "mirrors the other file" prose. At minimum, add a cross-path parity test that asserts the two orchestrations reach the same terminal classification for a shared table of result fixtures.

- FINDING: Sessions-page banner precedence is a single logical stack scattered across two functions with repeated line-0 surgery and load-bearing "do not touch" comments
  SEVERITY: medium
  FILES: internal/tui/model.go:4688-4790, internal/tui/notice_band.go:347-377, internal/tui/model.go:4665-4667
  DESCRIPTION: The spec defines ONE ordered precedence stack for what occupies the notice slot (filter line -> Opening n/N -> transient error/guidance flash -> multi-select banner -> unsupported banner -> no-tags signpost). The implementation realizes it across two physical rows and two functions: applySectionHeader (a 6-claimant if-chain — Opening band, preflight-abort banner, multi-select banner, unsupported banner, locked-query header, standard header — each repeating the identical `idx := strings.IndexByte(listView, '\n'); return header + listView[idx:]` splice), and activeNoticeBand (flash vs signpost). The two are coupled by a shared predicate (unsupportedBannerActive) and, critically, by prose invariants: one flash tier is deliberately split — the preflight-abort flash REPLACES at the section-header row while the spawn-failure/permission flash CO-RENDERS at the notice-band row — guarded only by a comment ("A later reader must NOT add `&& !m.multiSelectMode` here to force a single row"). Reasoning about "what shows when N sessions are marked on an unsupported terminal with a pending flash" requires mentally reconciling both functions and the cross-reference comments. That is fragile for a core UI-correctness surface with this many interacting claimants.
  RECOMMENDATION: Centralize the section-header-row claimant selection into a single ordered resolver that returns the winning header string (or "none"), so the precedence is one readable list and the line-0 splice happens exactly once at the call site. Keep the notice-band arbiter, but make the split-flash-tier decision explicit in a shared enum/table rather than prose, so the "replace vs co-render" choice for each flash kind is data, not a comment a future reader is warned not to break.

- FINDING: `Outcome` enum's zero value is `OutcomeSuccess`, so a zero-value Result silently reads as a successful window open
  SEVERITY: low
  FILES: internal/spawn/adapter.go:35-46, internal/spawn/adapter.go:79-81
  DESCRIPTION: `OutcomeSuccess Outcome = iota` makes 0 == success, so `Result{}.OK()` returns true and would gate a self-attach as if a window opened. This is inconsistent with the SAME package's RecipeKind (`RecipeArgv RecipeKind = iota + 1`, zero = explicit invalid sentinel) and with the project's stated enum convention (start at 1 / zero = unset). Today every Result is built through the Success/SpawnFailed/PermissionRequired constructors and the "opened" partition keys off Ack (not Result.OK), so the practical blast radius is small — but Burster.Run does branch self-attach-await on `result.OK()`, so any future path that yields a zero Result (a map miss, a not-yet-populated field, a partially-built struct) would be classified as a clean success rather than a failure. Making a "success" the accidental default of an outcome that gates an irreversible exec is the wrong-way-round default.
  RECOMMENDATION: Introduce a zero sentinel (`OutcomeUnknown Outcome = iota` then Success/SpawnFailed/PermissionRequired) so a zero-value Result fails `OK()` and is never mistaken for success — matching RecipeKind's treatment in the same file. If the current numbering must stay, add a compile-time/comment invariant documenting why success-at-zero is safe here.

- FINDING: `burstAllConfirmed` re-derives all-confirmed with a parallel loop instead of the documented count-semantics chokepoint
  SEVERITY: low
  FILES: internal/tui/burst_progress.go:240-250, internal/spawn/classify.go:25-34
  DESCRIPTION: classify.go is explicitly framed as "the single count-semantics chokepoint" and documents that "A batch is all-confirmed precisely when the returned failed slice is empty." burstAllConfirmed instead loops over msg.Results applying `!r.Confirmed()` itself. It does reuse the shared Confirmed() predicate (so it is not fully independent) and adds a legitimate extra guard (`len(msg.Results) != len(m.burstExternal)`), but it does not derive all-confirmed from PartitionResults' failed==empty relationship the way the CLI path does. Given the package's stated obsession with a single drift-proof chokepoint, this is a small inconsistency: a future change to what "all confirmed" means would update PartitionResults' contract but silently leave this parallel loop behind.
  RECOMMENDATION: Derive it: `_, failed := spawn.PartitionResults(msg.Results); return msg.Err == nil && len(msg.Results) == len(m.burstExternal) && len(failed) == 0`, so the picker's success gate rests on the same chokepoint the CLI's does.
SUMMARY: The feature's shared LEAF primitives (detection, resolution, ack, classify, message, logemit, the Burster) compose cleanly across the CLI and picker — the "one service, two callers" goal is well met at that layer. The architectural weak points are one layer up: the burst ORCHESTRATION sequence and the Sessions-page banner PRECEDENCE are each reimplemented per-caller/per-row and kept consistent only by cross-referencing comments and manual re-syncs, and the Model carries four dead burst-outcome fields plus an outcome enum whose zero value defaults to success.
