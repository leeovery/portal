- FINDING: Spawn-failure / permission transient flash co-renders with the multi-select banner instead of winning the single notice slot
  SEVERITY: low
  FILES: internal/tui/notice_band.go:347-364, internal/tui/model.go:4676-4739, internal/tui/burst_partial_failure.go:36-78
  DESCRIPTION: The spec (§ Mode affordance → "Notice-band precedence (single slot, highest wins)") enumerates one ordered slot where a transient error/guidance flash — explicitly grouped as "pre-flight abort / spawn-failure / permission" — outranks and therefore REPLACES the multi-select banner. The implementation realizes this precedence across two physical rows and treats the three members of that flash tier inconsistently. The pre-flight abort is a section-header claimant (applySectionHeader) checked ABOVE the multi-select `N selected` banner, so it replaces it — matching its delivered Paper frame. But the spawn-failure and permission outcomes route through setFlash → the §11 ▌-barred notice band (activeNoticeBand returns the flash regardless of multiSelectMode), a SEPARATE row that does not suppress the section-header multi-select banner. Since handleBurstPartialFailure keeps multiSelectMode true, after a partial/permission failure the picker renders BOTH the `N selected` banner (section header) AND the `⚠ … failed to open` flash (notice band) at once — the flash never "wins the slot" over the multi-select banner as the spec's precedence states. Impact is visual-only and arguably informative (you see the error and that you remain in multi-select with the retry set marked); there is no delivered design frame for the spawn-failure/permission flash (the spec records it as a design residual), so the exact placement was underspecified. Flagging because it is a literal divergence from an explicitly-decided precedence and is inconsistent with how the sibling pre-flight-abort flash was realized.
  RECOMMENDATION: Confirm the two-row split is the intended reading of the "single slot, highest wins" precedence. If it is (it matches the delivered frames and is arguably better), no code change is needed — but the divergence is worth an explicit note so a later reader does not "fix" it back to strict single-slot. If strict single-slot was intended, suppress the multi-select section-header banner while a warning/permission flash owns the notice band (mirroring how unsupportedBannerActive already steps aside for multi-select), so the flash presents alone.
