- FINDING: Model caches the Resolution but discards the Adapter, forcing dispatchBurst to re-resolve — a non-deterministic second resolve that can yield a nil adapter and panic the burst goroutine
  SEVERITY: medium
  FILES: internal/tui/model.go:2456, internal/tui/burst_progress.go:464, internal/spawn/resolver.go:104, internal/spawn/configadapter.go:116
  DESCRIPTION: The identity→adapter resolution is produced as a single (Adapter, Resolution) pair by `resolve`, but the terminalDetectedMsg arm keeps only the Resolution (`_, m.detectResolution = m.resolve(msg.identity)`) and throws the Adapter away. dispatchBurst then calls `m.resolve(m.detectIdentity)` a SECOND time to recover an adapter, and trusts it non-nil ("guaranteed non-nil on a supported resolution") without a guard. That guarantee only holds if `resolve` is deterministic — but the config *script* adapter path is not: `newScriptRecipeAdapter` does a live `os.Stat` of the script on every call. So (a) every supported config-script burst stats the script twice, and (b) if the script is deleted or loses its exec bit between page-entry detection and Enter, the cached resolution is still `config` (so decideBurst's DetectUnsupported() gate passes) while the second resolve now returns `(nil, ResolutionUnsupported)`. dispatchBurst proceeds with the nil adapter, builds `spawn.NewBurster(nil, …)`, and the burst goroutine calls `nil.OpenWindow(argv)` → nil-interface panic, which (in a bare `go func()`, not a recovered tea.Cmd) crashes the whole picker. This is the "correctness depends on two mirrored computations agreeing" anti-pattern: the gate decision and the adapter derive from two separate resolve calls that are assumed — but not guaranteed — to agree. The scenario is narrow (config script recipe + mid-session deletion), but it degrades to a crash rather than the honest unsupported no-op the rest of the design is careful to reach.
  RECOMMENDATION: Cache the Adapter alongside the Resolution from the single detection-time resolve (add an `m.detectAdapter spawn.Adapter` field set in the terminalDetectedMsg arm), and have dispatchBurst use the cached adapter instead of re-resolving. This makes the "guaranteed non-nil on a supported resolution" comment actually true (gate and adapter come from ONE resolve), removes the redundant `os.Stat`, and eliminates the TOCTOU nil-adapter panic. If caching the adapter is undesirable, at minimum nil-guard the re-resolved adapter in dispatchBurst and fall back to the unsupported no-op path.

- FINDING: The net-N split invariant is re-authored in both callers instead of centralized alongside its sibling shared spawn primitives
  SEVERITY: low
  FILES: cmd/spawn.go:120, internal/tui/burst_progress.go:461, internal/spawn/classify.go, internal/spawn/preflight.go
  DESCRIPTION: internal/spawn deliberately centralizes every leaf decision the two callers share — PreflightMissing, PartitionResults, FirstPermission, GoneMessage/PartialFailureMessage/UnsupportedNoopMessage, and the Log*/count-semantics helpers — with the stated design goal that "the two paths cannot drift." The one load-bearing invariant NOT centralized is the net-N split itself: cmd/spawn.go computes `external := sessions[:n-1]; trigger := sessions[n-1]` and tui/burst_progress.go independently computes `trigger := ordered[len-1]; external := ordered[:len-1]`. The spec elevates "net N windows, never N+1" to a hard anti-requirement, yet its mechanical realization is the sole shared invariant left duplicated. It is trivially correct today (two slice expressions), so the risk is low — but it is conspicuously the one seam the shared-primitive philosophy skipped, and it sits next to a related soft duplication: the preflight→unsupported→permission-first→partial→success control-flow *ordering* is kept in sync across runSpawn and decideBurst/handleBurstPartialFailure only by "mirroring cmd/spawn.go" code comments (e.g. §8-3 preflight-before-unsupported is coded in both places).
  RECOMMENDATION: Add a small pure helper in internal/spawn — e.g. `SplitNetN(ordered []string) (external []string, trigger string)` — and have both runSpawn and dispatchBurst/beginBurst derive the split through it, so the net-N invariant lives with its siblings and cannot drift. (The control-flow-ordering parallelism is acceptable given the sync/async divergence between the two callers; a shared helper for the split is the proportionate step.)
SUMMARY: The spawn architecture is largely sound — a clean single-capability Adapter contract, well-segregated ack seams (Collector/Cleaner/Writer split by need), a strong quarantine of OS specifics behind the generic Result taxonomy, and genuinely shared leaf primitives so the CLI and picker cannot drift on messages/counts/logging. Two structural issues stand out: the model caches Resolution but not the Adapter, forcing a non-deterministic re-resolve that can nil-panic the burst goroutine on a mid-session config-script deletion (medium), and the net-N split is the one load-bearing invariant left un-centralized across the two callers (low).
