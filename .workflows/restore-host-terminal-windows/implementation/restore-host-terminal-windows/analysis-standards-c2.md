- FINDING: `n` (new-in-cwd) stays live in multi-select mode, contradicting the mode's key-coexistence rule
  SEVERITY: medium
  FILES: internal/tui/model.go:3400-3401 (multi-select rune switch), internal/tui/model.go:3704-3713 (handleNewInCWD)
  DESCRIPTION: The spec (Multi-Select Mode → Key coexistence within the mode) fixes a closed live-set for the mode — "Live in mode: Space (preview), / (filter), s (regroup)" — and suppresses everything else: "Suppressed in mode: k (kill), x (page-toggle), r (rename), and other row actions." The implementation gates k, r, and x behind `if m.multiSelectMode { return m, nil }` (model.go:3383-3425) but leaves `n` (new session in cwd) fully active: in multi-select mode, pressing `n` dispatches handleNewInCWD → createSessionInCWD, which creates a session and quits the picker, silently discarding the entire marked set. `n` is not in the explicit live-set, and `x` — which is likewise not strictly a row action (it is a page-toggle) — is nonetheless explicitly suppressed, establishing the intent that non-selection/mutating actions are locked out while marking. Leaving `n` live is therefore a divergence from the mode's decided key coexistence, and it destroys in-progress selection state without warning. (Ambiguity noted: the spec's catch-all is "other row actions" and `n` does not act on a row, so a strict-literal reading could argue it was not named; but parity with the suppressed `x` and the closed live-set makes suppression the clear intent.)
  RECOMMENDATION: Gate the `n` case in the multi-select rune switch the same way k/r/x are — `if m.multiSelectMode { return m, nil }` — so create-new-session cannot fire (and silently drop the selection) while in the mode.

- FINDING: Spawn-failure / permission flash co-renders with the `N selected` banner rather than winning the single notice-band slot
  SEVERITY: low
  FILES: internal/tui/model.go:4680-4747 (applySectionHeader precedence), internal/tui/notice_band.go:347-412 (activeNoticeBand), internal/tui/burst_partial_failure.go:36-117
  DESCRIPTION: The spec (Multi-Select Mode → Mode affordance) specifies a single-slot arbiter — "One claimant at a time (single-slot arbiter)" — with precedence "filter line → in-burst Opening n/N… → transient error/guidance flash (pre-flight abort / spawn-failure / permission) → multi-select banner → unsupported-terminal banner → no-tags signpost," i.e. the flash tier outranks and displaces the multi-select banner in the one slot. The implementation realises this correctly for the pre-flight abort (renderPreflightAbortHeader replaces the `N selected` banner in the section-header row, model.go:4717-4729, matching design frame 2), but routes the spawn-failure and permission-required flashes through setFlash → the separate §11 notice band (activeNoticeBand), which deliberately CO-RENDERS with the still-shown `N selected` multi-select banner (two rows) instead of displacing it. Per a strict reading of the spec's "single slot, highest wins," the flash should win the slot and hide the banner. The implementer documented this split at length (model.go:4712-4716) and there is no delivered Paper frame for the spawn-failure case (a stated design residual), so this may be intended; flagging so the intended single-slot behaviour for the spawn-failure/permission tier is confirmed against the design rather than silently diverging.
  RECOMMENDATION: Confirm with the delivered/updated design whether the spawn-failure and permission flashes should displace the `N selected` banner (single-slot, per spec) or co-render as two rows (current behaviour). If single-slot is intended, route these flashes to the section-header claimant chain (as the pre-flight abort banner is) rather than the co-rendering §11 notice band.
SUMMARY: The spawn/multi-select implementation is strongly conformant to the specification and project conventions — closed spawn log component + attr set, count semantics (total=N, opened=surfaced), net-N/N-1 split, os.Executable + minimal-PATH env-self-sufficient argv with TMUX/TMUX_PANE stripped, sequential spawn, per-window token-ack timeout, pre-flight all-or-nothing + leave-what-opened, config precedence and recipe validation, permission-quarantine typed Result, and integration/manual test build tags are all faithfully implemented. Two conformance gaps remain: `n` is not suppressed in multi-select mode (contradicting the mode's closed live-set), and the spawn-failure/permission flash co-renders with the multi-select banner rather than winning the spec's single notice-band slot.
