- FINDING: Result classification (the "count-semantics chokepoint") is authored independently per caller instead of composed from one spawn helper
  SEVERITY: medium
  FILES: cmd/spawn.go:213, cmd/spawn.go:231, internal/tui/burst_observability.go:43, internal/tui/burst_partial_failure.go:55, internal/tui/burst_progress.go:241, internal/spawn/burst.go:57
  DESCRIPTION: The spec designates a single "count-semantics chokepoint": a WindowResult is "opened" exactly when Ack==AckConfirmed, the failed/confirmed partition drives leave-what-opened retry, and "first permission wins" drives the burst-stop message. That rule is a pure function of []spawn.WindowResult, but it is re-implemented in at least five sites across two callers: cmd/spawn.go tallyWindowResults (opened/failed partition) and firstPermission (permission scan); internal/tui emitBurstSummary (opened count), handleBurstPartialFailure (confirmed/failed partition), burstAllConfirmed (all-confirmed check), and burstPartialFailureFlash (permission scan). The code comments themselves concede the two paths must be manually kept "byte-identical" (burst_observability.go header, burst_partial_failure.go). This is the classic missed-composition seam: the spawn package already hosts the sibling pure helpers PreflightMissing, QuoteJoin and GoneVerb, yet the confirmed/failed/permission classification — the one the spec calls a chokepoint — is conspicuously not hoisted. Each independent copy is a drift point at exactly the seam the spec wanted unified; a future change to what "confirmed" means (e.g. counting AckTimeout differently) must be found and edited in five places or the CLI and picker silently diverge.
  RECOMMENDATION: Add the classification to internal/spawn as the single source both callers derive from — e.g. a WindowResult.Confirmed() predicate plus PartitionResults(results) (confirmed, failed []string) and FirstPermission(results) (WindowResult, bool). Have cmd/spawn.go's tally/firstPermission and the three TUI sites call these rather than re-loop over Ack, so opened/failed/permission semantics live in one place alongside the existing shared spawn helpers.

- FINDING: CLI and picker diverge on the permission-required observability event, breaking one-service lockstep
  SEVERITY: medium
  FILES: cmd/spawn.go:180, cmd/spawn.go:276, internal/tui/burst_partial_failure.go:36, internal/tui/burst_observability.go:70
  DESCRIPTION: The spec's closed `spawn` event catalog lists permission-required as a distinct outcome, and the whole architecture is "one service, two callers ... so their behaviour cannot drift." On a permission-required burst the CLI emits a dedicated INFO event `spawn: permission required — nothing self-attached` (logSpawnPermission) and deliberately skips the opened/total cycle summary. The picker has no equivalent: handleBurstPartialFailure unconditionally calls emitBurstSummary, so a permission burst logs the generic `spawn: opened 0/N` summary and never emits the permission event at all (the permission detail survives only in the DEBUG per-window line). burst_observability.go has emitUnsupportedNoop and emitPreflightAbort but no emitPermission, and its header claims to mirror "logSpawnSummary / tallyWindowResults / logSpawnUnsupported / logSpawnGone" — logSpawnPermission is silently absent from that list. The result is that the two callers of the same service emit different instrumentation for the same outcome, and a spec-catalogued event is unreachable on the picker path (the dominant real path).
  RECOMMENDATION: Give the picker an emitPermission mirror (resolution/terminal/bundle_id/detail, no counts) and route the permission arm of handleBurstPartialFailure through it instead of emitBurstSummary, matching the CLI's branch. Folds naturally into Finding 1's shared FirstPermission helper so both callers detect and log the permission outcome identically.

- FINDING: Exported spawn.AttachCommand has no production caller — dead public API
  SEVERITY: low
  FILES: internal/spawn/command.go:24
  DESCRIPTION: AttachCommand is a public function that resolves the executable and composes the attach argv, but neither caller uses it: Burster.Run resolves os.Executable once up front and calls the pure composeAttachArgv per window (documented at burst.go:113 as "not AttachCommand"), and both the CLI runSpawn and the picker dispatchBurst go through the burster. Grep shows the only remaining references are a doc-comment and a spawntest comment. It is public surface that is exercised by nothing, so it can drift from the real composition path (composeAttachArgv) without any test noticing a behavioural mismatch.
  RECOMMENDATION: Either remove/unexport AttachCommand, or make the burster compose through it (accepting the redundant per-window executable read, or refactoring it to take a pre-resolved path) so the one composition path is the tested path. Keep the spawn package's public surface to what is actually reached.

- FINDING: Deferred N>=2 Enter opens a stale selection snapshot when detection is still in flight
  SEVERITY: low
  FILES: internal/tui/burst_progress.go:370, internal/tui/model.go:2466, internal/tui/model.go:3280
  DESCRIPTION: When an N>=2 Enter lands before async terminal detection resolves, beginBurst stashes m.orderedMarkedSessions() into pendingBurstOrdered and returns without setting burstPending. The input-lock guard (updateSessionList: `if m.burstPending`) is therefore NOT engaged during the defer window, so a subsequent `m` toggle is processed normally and mutates selectedSessions — but not the stashed snapshot. When terminalDetectedMsg fires it calls decideBurst(m.pendingBurstOrdered), spawning the stale set (a just-unmarked session still opens; a just-marked one is skipped). Detection begins on Sessions-page entry and is near-instant, so the window is tiny and requires a mark change between Enter and the detection reply, but the correctness rests on caller/detection timing rather than being self-contained — the same "correctness depends on caller discipline" smell the spec warns against.
  RECOMMENDATION: Re-derive orderedMarkedSessions() inside decideBurst (from the live selectedSessions) rather than replaying the pendingBurstOrdered snapshot, or extend the burst input-lock to cover the pre-detection defer so the marked set cannot change under it. The former is cheaper and keeps the burst self-contained.
SUMMARY: Seams are clean (1–3-method interfaces, thorough DI, nil-tolerant options) and the burst goroutine/channel/context lifecycle is correct — the naked terminal send plus ctx-guarded progress is sound, and burstChannelClosedMsg correctly absorbs the extra receiver cancelBurst adds, with no goroutine leak or send-on-closed. The substantive issues are compositional: the spec's count-semantics/result-classification is re-authored per caller instead of derived from one spawn helper, and the CLI/picker paths diverge on the permission-required log event, weakening the "one service, two callers cannot drift" guarantee at its instrumentation seam.
