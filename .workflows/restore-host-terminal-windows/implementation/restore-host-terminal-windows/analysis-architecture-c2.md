- FINDING: Picker gates the N≥2 unsupported no-op ahead of pre-flight, inverting the CLI's deliberate order and skipping the spec-mandated gone-session prune
  SEVERITY: medium
  FILES: internal/tui/burst_progress.go:399-418, internal/tui/burst_progress.go:188-193, cmd/spawn.go:132-155
  DESCRIPTION: The spec frames spawn as "one service, two callers" running "the identical pre-flight → sequential spawn → per-window ack → self-attach-last flow," and the CLI's runSpawn runs the pre-flight has-session gate FIRST (cmd/spawn.go:132) — ahead of the N≥2 unsupported gate (cmd/spawn.go:152) — with an explicit in-code rationale ("so a gone session aborts with the more-actionable gone-session message even on an unsupported terminal"). The picker inverts this: decideBurst short-circuits to the unsupported atomic no-op on DetectUnsupported() (burst_progress.go:410) BEFORE dispatchBurst launches the goroutine, and pre-flight only runs inside burstRunner.run (burst_progress.go:190), which is never reached on an unsupported terminal. Consequence on the exact same scenario (N≥2 marked, unsupported/undriven terminal, one marked session externally killed): the CLI reports "'s2' is gone — nothing opened" and prunes it; the picker re-asserts "unsupported terminal — … — nothing opened" and never runs pre-flight, so the gone session is neither surfaced nor pruned from the selection — diverging from both the CLI and the picker's own pre-flight-abort prune-keeping-survivors contract (handlePreflightAbort). Pre-flight is the spec's primary Enter gate ("Before opening a single window, verify every selected session still exists"; "All-or-nothing applies at the pre-flight gate"), so the picker's ordering is the outlier. Impact is bounded (nothing opens on an unsupported terminal either way), but it is a genuine cross-caller flow-order divergence plus a skipped selection-prune the spec mandates.
  RECOMMENDATION: Run pre-flight before the unsupported gate on the picker path so both callers share the ordering: have decideBurst (or a step just ahead of it) evaluate spawn.PreflightMissing over the marked set first and route a non-empty result to the pre-flight-abort arm (gone message + prune survivors) before the DetectUnsupported() no-op — mirroring cmd/spawn.go's preflight-then-unsupported sequence. This keeps the "more-actionable gone message wins on an unsupported terminal" behaviour and honours the gone-prune on every path.

- FINDING: Partial-failure "leave-what-opened" message is authored independently in each caller, so the CLI and picker render non-identical text despite the spec requiring "the same one-line message"
  SEVERITY: medium
  FILES: cmd/spawn.go:195, internal/tui/burst_partial_failure.go:116, internal/spawn/message.go:40-58
  DESCRIPTION: Cycle 1 built internal/spawn/message.go as the shared cross-caller renderer layer precisely so the CLI and picker "name sessions identically" — GoneMessage and UnsupportedNoopMessage are both derived there from QuoteJoin/GoneVerb. The partial-failure ("others left open") line was left out of that layer and is hand-built separately in each caller with different wording: the CLI emits "spawn: failed to open window(s) for 's2' — others left open" (cmd/spawn.go:195) while the picker emits "'s2' failed to open — others left open" (burst_partial_failure.go:116). The spec's `portal spawn` CLI contract explicitly requires the partial-spawn-failure exit-1 line to be "the same one-line message the picker would show," so this is a broken cross-caller contract and an incomplete application of the very message-parity abstraction cycle 1 established — the same missed-composition shape as the gone/unsupported renderers, one message short. A future copy edit to either sentence widens the drift silently.
  RECOMMENDATION: Add a sibling renderer to internal/spawn/message.go (e.g. PartialFailureMessage(failed []string) string returning the bare body with no "spawn:" prefix and no ⚠ glyph, composed from QuoteJoin) and have both cmd/spawn.go and burst_partial_failure.go render through it, so the partial-failure line joins GoneMessage/UnsupportedNoopMessage in the single parity layer and the two callers cannot diverge.

- FINDING: The Result taxonomy's OutcomeUnsupported member (and Unsupported() constructor) is unreachable by any adapter and is a latent mis-classification trap
  SEVERITY: low
  FILES: internal/spawn/adapter.go:29-31, internal/spawn/adapter.go:59-62, internal/spawn/burst.go:164-168, internal/spawn/classify.go:43-50
  DESCRIPTION: "Unsupported" is a resolution-tier concept: the Resolver returns (nil, ResolutionUnsupported) when no driver matches, and OpenWindow is only ever called on a resolved (supported) adapter. No adapter constructs an OutcomeUnsupported Result — the ghostty/argv/script adapters return only Success/SpawnFailed/PermissionRequired — so both OutcomeUnsupported and the Unsupported() constructor are referenced solely at their own declaration (adapter.go). This duplicates the "unsupported" concept across two enums (Resolution and Outcome) where only Resolution's is reachable. Worse than merely dead: it is a latent trap. If a future driver author does return Unsupported() from OpenWindow (a natural reading of the "closed taxonomy"), the burster classifies it as !OK() → AckFailed (burst.go:165) → a failed window handled by leave-what-opened with a "failed to open" flash, and FirstPermission (classify.go) only recognises OutcomePermissionRequired — so it would NOT route to the atomic unsupported no-op the resolution tier owns. The taxonomy invites a value the orchestrator has no handler for at the OpenWindow layer.
  RECOMMENDATION: Remove OutcomeUnsupported and the Unsupported() constructor from the Adapter Result taxonomy (leaving "unsupported" solely on Resolution, where it is decided and handled), or, if retained for spec-completeness, document at the declaration that OpenWindow must never return it and that "unsupported" is a resolution-tier outcome — so a future adapter author does not reach for it and silently mis-classify.
SUMMARY: The spawn feature's "one service, two callers" composition is largely sound (cycle 1's shared classify/message/exec-boundary helpers hold), but two cross-caller seams still drift from the CLI-vs-picker parity the spec mandates: the picker gates the unsupported no-op ahead of pre-flight (inverting the CLI's order and skipping the gone-prune), and the partial-failure message is authored separately in each caller so the rendered text differs despite a "same one-line message" requirement. A third, low item: the Result taxonomy carries an OutcomeUnsupported member no adapter can produce and that would mis-classify if one did.
