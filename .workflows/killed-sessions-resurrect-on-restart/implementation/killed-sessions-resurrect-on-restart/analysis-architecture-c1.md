# Architecture Findings — killed-sessions-resurrect-on-restart (cycle 1)

```
AGENT: architecture
FINDINGS:
- FINDING: Two-layer EagerHydrateSignaler wrapping when one would suffice
  SEVERITY: medium
  FILES: cmd/bootstrap/eager_signal_hydrate.go:30-91, internal/bootstrapadapter/adapters.go:164-186, cmd/bootstrap_production.go:123-133
  DESCRIPTION: The eager-signal step has two parallel public types both implementing `bootstrap.EagerHydrateSignaler`: `bootstrap.EagerSignalCore` (the algorithm) in cmd/bootstrap/, and `bootstrapadapter.EagerHydrateSignaler` (the production wiring) in internal/bootstrapadapter/. The adapter exists *only* to bind one closure: `func(path string) error { return state.WriteFIFOSignal(path, state.OpenFIFOForSignal, time.Sleep) }`. Compare with the sibling `MarkerCleanupCore`, which is constructed inline at the wiring site in cmd/bootstrap_production.go (`*tmux.Client` satisfies every seam directly, no glue type). The asymmetry is a clear "different-shape-for-same-role" outcome: `EagerSignalCore` accepts a `WriteFIFOSignal func(path) error` field while `MarkerCleanupCore` accepts typed interface seams. The function-valued field forced the adapter into existence. Side effects: (a) every Run allocates a fresh `EagerSignalCore` inside the adapter's `EagerSignalHydrate()` method (lines 177-185) instead of constructing the core once at wiring time, (b) two doc-comment surfaces describing the same step must stay in sync, (c) two parallel `var _ EagerHydrateSignaler = ...` assertions exist for what is conceptually one production type.
  RECOMMENDATION: Either (a) replace the `WriteFIFOSignal func(path) error` field on `EagerSignalCore` with a typed seam — e.g. `FIFOSignaler interface { SendSignal(path string) error }` plus a default `state.DefaultFIFOSignaler{}` value — so production wiring constructs the core inline at cmd/bootstrap_production.go, mirroring `MarkerCleanupCore`; or (b) collapse `bootstrapadapter.EagerHydrateSignaler` into a single helper constructor (`bootstrapadapter.NewEagerHydrateSignaler(client, stateDir, logger) *bootstrap.EagerSignalCore`) that returns the core directly with the production closure pre-bound, eliminating the wrapper type entirely.

- FINDING: Inconsistent test seam shape for the WriteFIFOSignal primitive across cmd vs cmd/bootstrap callers
  SEVERITY: low
  FILES: cmd/state_signal_hydrate.go:16-23, cmd/bootstrap/eager_signal_hydrate.go:30-35, internal/state/signal_hydrate.go:45-68
  DESCRIPTION: Both `runSignalHydrate` (in cmd) and `EagerSignalCore.EagerSignalHydrate` (in cmd/bootstrap) drive the same `state.WriteFIFOSignal` retry-ladder primitive in production, but expose different test seams. `signalHydrateConfig` injects `OpenFIFO func(path) (*os.File, error)` + `Sleep func(time.Duration)` — the retry-ladder primitives — and lets the real `state.WriteFIFOSignal` body run end-to-end in tests. `EagerSignalCore` injects a single `WriteFIFOSignal func(path) error` and replaces the entire ladder body in tests. Result: `cmd/bootstrap/eager_signal_hydrate_test.go` cannot exercise the ENXIO/EAGAIN retry path or the cumulative-budget contract — those only live in `internal/state/signal_hydrate_test.go`. Two parallel test fixtures (`recordingFIFOWriter` vs the open-pipe `fakeSleep` harness) verify overlapping but non-identical contracts.
  RECOMMENDATION: Tied to Finding 1 — once the writer becomes a typed `FIFOSignaler` interface, both call sites can share the same fake (`recordingFIFOSignaler`) and the production-shape end-to-end retry-ladder verification stays in `internal/state` where the primitive lives. Delete `recordingFIFOWriter` in cmd/bootstrap/eager_signal_hydrate_test.go in favour of the unified shape.

- FINDING: Dual integration-orchestrator builders forced into manual lock-step
  SEVERITY: low
  FILES: cmd/bootstrap/orchestrator_builder_test.go:33-87, cmd/reattach_integration_test.go:163-187
  DESCRIPTION: Two sibling builders (`buildIntegrationOrchestrator` in package `bootstrap_test`, `buildReattachOrchestrator` in package `cmd`) construct the same `*bootstrap.Orchestrator` struct with the same NoOp defaults, separated only because Go test-file symbols are package-private. Both builders explicitly enumerate every step seam (Server, Hooks, Restoring, Saver, Restore, EagerSignaler, StaleMarkers, Sweeper, Clean). The orchestrator-builder file's own comment acknowledges the cost: "Adding a new step interface therefore requires editing two files." With this work adding a tenth step (EagerSignaler), both builders had to be edited. Future step additions will continue to incur this tax.
  RECOMMENDATION: Promote a NoOp-defaulted builder to a small non-test helper in cmd/bootstrap (e.g. `bootstrap.NewWithDefaults(server ServerBootstrapper, restoring RestoringMarker, opts ...Option) *Orchestrator`) — production needs no NoOps, but the helper's defaulting policy can be reused by both test packages via standard import. Tests then call `bootstrap.NewWithDefaults(client, &bootstrapadapter.RestoringMarker{Client: client}, bootstrap.WithRestore(...), bootstrap.WithEagerSignaler(...))`. A new step requires only an `Option` constructor and one default-NoOp wire-up at the helper site.

- FINDING: Existing reboot-roundtrip integration tests silently default EagerSignaler to NoOp, masking step-level integration drift
  SEVERITY: low
  FILES: cmd/bootstrap/orchestrator_builder_test.go:63-65, cmd/bootstrap/reboot_roundtrip_test.go:332-339, cmd/reattach_integration_test.go:182
  DESCRIPTION: `buildIntegrationOrchestrator` defaults `EagerSignaler` to `bootstrap.NoOpEagerHydrateSignaler{}` unless the caller explicitly opts in. Existing reboot/reattach integration tests (`TestPhase5RebootRoundTripEndToEnd`, `TestPhase5RebootRoundTripBaseIndexDrift`, `TestPhase5RebootRoundTripBothSessionsHydrateViaSignalHydrateBinary`, every test in `reattach_integration_test.go`) consequently exercise a non-production orchestrator: in production, EagerSignalHydrate runs every bootstrap and drives helpers' marker-clear before any client attaches; in these tests, that step is no-op'd and signal-hydrate is driven manually post-Run. The eager-signal step is only exercised inside the dedicated `eager_signal_hydrate_integration_test.go` and `phase2_hook_fire_integration_test.go`. The risk: a regression that makes the eager step interfere with the existing manual-signal-hydrate harness (e.g. a future change that has the eager step double-write or race with the daemon) would not surface in the existing reboot tests, only in the dedicated AC1/AC2 tests. The test pipeline diverges from the production pipeline on a load-bearing dimension.
  RECOMMENDATION: Flip the default in `buildIntegrationOrchestrator` for the EagerSignaler field: when caller has provided a real RestoreAdapter, default EagerSignaler to a real adapter wired against the same client/stateDir/logger rather than NoOp. Tests that need the manual-drive harness can still opt out by explicitly passing `EagerSignaler: bootstrap.NoOpEagerHydrateSignaler{}`. This keeps the test pipeline in lock-step with production and surfaces eager-signal regressions across the reboot suite, not just dedicated tests.

- FINDING: `WriteFIFOSignal` exposes seam parameters production callers never vary
  SEVERITY: low
  FILES: internal/state/signal_hydrate.go:45-68, internal/bootstrapadapter/adapters.go:180-182, cmd/state_signal_hydrate.go:97-99
  DESCRIPTION: `state.WriteFIFOSignal(path, openFIFO, sleep)` accepts two function-value seams that production callers always wire the same way: `state.OpenFIFOForSignal` and `time.Sleep`. The seam exists to support tests, but it leaks the test-only generalisation into the production API surface. Every call site has to remember to pass the production defaults — and one of them (`bootstrapadapter.EagerHydrateSignaler`) wraps the call in a closure just to fix the parameters, which then forces the adapter type to exist (Finding 1). A cleaner factoring exposes a no-seam production entry point (`state.SendHydrateSignal(path) error`) plus a test-internal variant (`state.WriteFIFOSignal(path, openFIFO, sleep)`) for unit tests of the retry ladder. Per code-quality.md "Concrete Over Abstract": the production callers know exactly what they want; the abstract seam is design-time generality without runtime variance.
  RECOMMENDATION: Add `func state.SendHydrateSignal(path string) error { return WriteFIFOSignal(path, OpenFIFOForSignal, time.Sleep) }`. Make production callers use it directly: `EagerSignalCore.WriteFIFOSignal: state.SendHydrateSignal` (no closure); `runSignalHydrate` calls `state.SendHydrateSignal(fifoPath)`. Retain the seam-bearing variant only for `internal/state/signal_hydrate_test.go`. Removes the closure, removes one reason for the bootstrapadapter wrapper, leaves retry-ladder test coverage intact.

SUMMARY: The eager-signal step lands cleanly behind a single-method interface and respects the orchestrator's degrade-and-continue contract, but the production wiring carries a redundant adapter layer (`bootstrapadapter.EagerHydrateSignaler` wrapping `bootstrap.EagerSignalCore`) that exists solely to bind a fixed closure. The asymmetry with `MarkerCleanupCore` (constructed inline) is a tractable refactor — replacing the function-valued `WriteFIFOSignal` field with a typed seam plus a no-seam production helper would collapse the adapter and unify the divergent test fixtures across cmd and cmd/bootstrap. Two minor seam-fidelity issues with the integration-test orchestrator builders (defaulting EagerSignaler to NoOp in the reboot suite, dual builders maintained in lock-step) are existing taxes this work compounded but did not introduce.
```
