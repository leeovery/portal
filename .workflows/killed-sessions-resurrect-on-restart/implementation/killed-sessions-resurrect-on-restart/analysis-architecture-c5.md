# Architecture Findings — killed-sessions-resurrect-on-restart (cycle 5)

```
AGENT: architecture
FINDINGS:
- FINDING: shellQuoteSingle's escape sequence is a latent bug in the post-wrapper-drop bare-form invocation — the `'\''` close/escape/reopen idiom requires an outer single-quoted envelope that no longer exists at the call site
  SEVERITY: low
  FILES: /Users/leeovery/Code/portal/internal/restore/session.go:426-440, /Users/leeovery/Code/portal/internal/restore/session_build_hydrate_test.go:30-40
  DESCRIPTION: After Fix 3 (wrapper drop, T3-1) buildHydrateCommand emits `portal state hydrate --fifo <fifo> --file <file> --hook-key <hookKey>` with each value-arg passed through shellQuoteSingle, whose body is `strings.ReplaceAll(s, "'", `'\''`)`. The `'\''` four-char sequence (close-quote, escaped-quote, reopen-quote) is well-formed shell ONLY inside an outer `'…'` envelope: the close-quote terminates a running single-quoted region, the `\'` emits a literal apostrophe in the unquoted region, and the reopen-quote restarts the envelope. With the wrapper dropped, no outer envelope exists. A session name containing a literal `'` (e.g. `it's-cool`) flows through SanitizePaneKey unchanged — sanitizeSessionName at internal/state/panekey.go:41-58 only filters `/`, `\`, and `\0` — into the fifo basename and into hookKey (`session:window.pane`). shellQuoteSingle would then emit unbalanced-quote tokens that tmux's internal `/bin/sh -c <command>` parses as malformed input, producing a syntax error or argv-shape drift instead of the intended literal apostrophe.
    The architectural concern is the doc-comment claim at session.go:424-425 ("shellQuoteSingle still escapes embedded single quotes … for defensive parity with the prior call-site contract") and the white-box test at session_build_hydrate_test.go:30-40 ("shellQuoteSingle replaces each ' with '\\'' (close, escaped quote, reopen). Defensive parity with the prior call-site contract so any future re-introduction of an outer single-quoted envelope would not change embedded-quote semantics"). Both presume the escape sequence retains meaning in the new context. It does not — the escape is now a no-op-for-most-inputs that flips to corruption-for-apostrophe-inputs. The "defensive parity" rationale is incoherent: parity with what? The prior call-site context (outer `'…'` envelope around the entire command) was deleted by T3-1.
    Two architectural failure modes follow from leaving the helper in place: (a) a maintainer sees shellQuoteSingle being called in the bare-form path, follows the doc, and concludes the value-args are quote-safe under the bare form — they are not; (b) a future caller adds a fourth value-arg routed through shellQuoteSingle and inherits the same broken contract, propagating the latent bug to a new code path.
    This is a real correctness gap (latent bug) rather than a "different-but-working" call: pathological-but-valid session names exist (tmux session names tolerate `'`), Portal does not reject them at create time, and the resulting shell parse would corrupt the helper's argv shape.
  RECOMMENDATION: Two equivalent options — pick the simpler:
    1. Delete shellQuoteSingle and its three call sites in buildHydrateCommand. Replace the body with `fmt.Sprintf("portal state hydrate --fifo %s --file %s --hook-key %s", fifo, file, hookKey)` and update the white-box test to drop the single-quote round-trip sub-test (the negative-assertion sub-test at lines 42-52 stays). Combined with a one-line note in buildHydrateCommand's docstring stating that `'`-bearing inputs would break shell parsing and Portal does not currently produce them in practice, the contract becomes self-describing.
    2. If defence-in-depth against `'`-bearing inputs is desired, replace shellQuoteSingle with a true POSIX shell-quoter that wraps each value in `'…'` (translating embedded `'` into the `'\''` close-escape-reopen idiom inside the wrap). The output for `it's-cool` would then be `'it'\''s-cool'` — which IS well-formed shell. The helper rename (e.g. shellQuotePOSIX) documents the semantics change. Same blast radius as option 1; one additional call-site change per value-arg.
    Option 1 matches the spec/plan's "defensive parity" intent better — the plan accepted that pathological inputs are out of scope, and explicit acknowledgement is cleaner than a half-broken escape. Option 2 is the right answer if any future input source can produce `'`.

SUMMARY: Cycle 4's two cleanup tasks (T7-1 collapsing pollUntilMarkersCleared into the parameterised restoretest helper, T7-2 correcting NewRestoreAdapter's production-parity rationale) both landed cleanly — the marker-poll helper is now the single canonical surface across all seven call sites, and the constructor docstring matches the source. One residual low-severity item surfaced during the broader sweep: shellQuoteSingle's `'\''` close-escape-reopen idiom requires an outer single-quoted envelope that the Fix 3 wrapper drop removed, leaving a latent shell-parse bug for `'`-bearing session names alongside a doc-comment that justifies the escape with an incoherent "defensive parity" rationale. Behavioural correctness for AC1-AC8 success cases is unaffected; the issue is a structural latent risk plus doc-vs-code drift, fixable by a one-line helper deletion.
```
